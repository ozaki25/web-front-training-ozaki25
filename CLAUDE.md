# Web Front-end Training

## プロジェクト概要

エンジニア初心者が Web フロントエンドの知識の引き出しを増やすための学習コンテンツ。
毎日15分の LT を聞く感覚で、HTML/CSS の基礎から Next.js まで幅広いトピックに触れる。
VitePress で構築されたドキュメントサイトとして提供する。

## ターゲット

- エンジニア初心者。AI 丸投げ開発の経験が多少ある程度
- これから Next.js（App Router）を使うプロジェクトに配属される予定の人
- HTML/CSS/JavaScript の基礎を自分の言葉で説明できないレベル

## 配属先の技術スタック

- Next.js（App Router、最新バージョンに追従）
- TypeScript
- Tailwind CSS

## ゴール

Web フロントエンドに関する知識の引き出しを増やすこと。何かの場面で「この話、どこかで聞いたな」と思い出せる状態を作る。

- AI に指示を出すとき、的確な言葉で伝えられる
- AI が返してきたコードに違和感があったとき、何がおかしいか気づける
- チームの技術的な会話で、話題についていける

すべてを深く理解する必要はない。「聞いたことがある」という引き出しがあるだけで、AI への指示や AI の成果物を判断する際の土台になる。

## コンテンツの使われ方

- チームで毎日30分の枠を確保して取り組む想定
  - 前半15分: その日のレッスンを読む
  - 後半15分: Q&A・派生した議論
- 1日1レッスン、読む部分は約15分で読み切れるボリューム
- 毎日少しずつ進めてもらう（一気にやるものではない）
- 順番に進める前提（前の Day の知識を前提にしてよい）

## ブランチ運用と情報の永続化

- 決定事項や方針は必ずファイルに記録し、コミット・プッシュする。会話の中だけに情報を残さない
- **main ブランチ**: プロジェクト方針（CLAUDE.md）、スキル定義、VitePress 設定など、コンテンツの具体に関わらないもの。`docs/index.md` や `docs/introduction/` は main に置くが、未公開レッスンへのリンクや具体的なレッスン内容は含めない
- **draft ブランチ**: レッスン本文、カリキュラム（curriculum.md）、サイドバー設定など、コンテンツの具体に関わるもの
- コンテンツは draft に蓄積し、公開時に publish-lesson スキルで Day 単位の PR を作成して main にマージする
- **draft の内容は何も確定していない**: draft のレッスンの内容・順番・数はいつでも変わりうる仮の状態であり、確定しているのは publish-lesson で main に公開されたものだけ。main には公開済みの情報のみを含め、未公開の情報（レッスンリンク、Day 数、カリキュラムの構成など）は一切含めないこと
- main に変更があった場合（方針変更、レッスン公開など）は、必ず draft と open 中の全 `publish/dayXX` ブランチに同期すること。古い main の状態を持ったままだと、以降の作業やマージで不整合が起きる
- 同期は手動で行う。main への push 後、`git merge origin/main` で draft に反映し、open 中の `publish/*` PR にはチェーンの根元から順に `gh pr update-branch` を実行する。コンフリクトしたら手動で解決する

### 許可されたブランチ

| ブランチ | 用途 | 変更してよいファイル |
|---------|------|---------------------|
| `main` | 方針・設定 | CLAUDE.md、`.claude/`、`.github/`、`docs/index.md`（公開済みリンクのみ）、`docs/introduction/`、`docs/.vitepress/config.mts`（公開済みのみ）、`package.json` |
| `draft` | コンテンツ | `docs/lessons/`、`docs/drafts/`、`curriculum.md`、`tech-versions.md`、`retrospective.md`、`docs/index.md`、`docs/.vitepress/config.mts` |
| `publish/dayXX` | 公開用（一時的） | publish-lesson スキルが作成・管理する。該当 Day のファイルのみ |

**これ以外のブランチを作成・使用してはならない。** feature ブランチ、個人ブランチ等は禁止。

### コンテンツのディレクトリ構成

| ディレクトリ | 用途 | 命名規則 |
|-------------|------|---------|
| `docs/lessons/dayXX/` | 公開済みレッスン | `dayXX`（XX は公開順の 0 埋め 2 桁） |
| `docs/drafts/<ID>/` | 候補の下書き | 3 桁ゼロパディング連番（001, 002, ...） |

- `docs/drafts/` の連番は単なる ID として扱う。順序に意味はなく、欠番 OK、詰め直し不要
- 新規追加時は `docs/drafts/` 配下の最大番号 +1 を採番する
- 公開時は `docs/drafts/<ID>/` → `docs/lessons/dayXX/` に移動する（publish-lesson スキルが行う）
- 不要になった下書きは削除する

## テーマ選定の基準

判定の最上位は「**AI ネイティブな若手に教える価値がありそうか**」。この感覚は作者のバックグラウンド（蔵書・イベント参加・発信履歴）が形作る。言語化すると「**AI に任せきりでは埋まらない欠落を埋めるか**」。

- AI が肩代わりすること（構文の書き方、ボイラープレート、コマンドの都度説明）→ 教える価値は低い
- AI に任せていると育たないこと（**出力を評価する目・指示の語彙・事故時の対応力・会話への参加力**）→ 価値が高い
- 公開済みレッスンが「仕組みもの」中心なのは、仕組みの理解が評価の目を作る最大の欠落だから。**仕組みジャンル縛りではなく**、欠落を埋めるなら新系統（テスト・ツール・UX・AI 等）も座れる
- 近似指標（テーマとレッスンの評価に使う）: E1 仕組みの核 / E2 体験への接地 / E3 AI への指示・評価の語彙になる / E4 単一の核 / E5 作者の知見との接続

## PR 運用

- **公開前レビューは必須・省略不可**: レッスンを公開（main への PR 作成）する直前に、その時点の本文へ必ず①ビルド②太字の描画チェック（`node .claude/hooks/check-bold.js`）③`review-lesson` の全カテゴリ、を通すこと。「前にレビュー済み」を理由に省略しない。下書きは公開までに何度も修正されるため、過去のレビュー後に崩れている可能性が常にある。記憶ではなく手順で縛る（詳細は publish-lesson スキル）
- **報告は機械チェックと判断レビューを分ける**: ビルド・太字崩れは通過/不通過が確定するので「通過」と言ってよい。自然な日本語・構成は判断であり漏れがありうるので、「完璧」「全部通った」と過信した言い方をしない
- **PR の説明は常に最新を保つ**: PR 作成後にコミットを追加・修正した場合や、レビューで変更が入った場合は、**必ず PR 本文を最新の内容に更新する**こと。タイトル・サマリ・変更点・テストプランすべてが最終状態を反映している必要がある。古い説明のまま放置しない
- PR 本文にはレビューに必要な情報だけを書く。作業メモや会話の経緯、内部的な注意事項は含めない

## コンテンツ作成のループ（スプリント運用）

コンテンツは**小さいタスク単位**で 1 周ずつループを回す。粒度は 2 階層で持つ。

- **ロードマップ（どのレッスンをやるか）はレッスン単位**: `docs/drafts/` の候補一覧・`curriculum.md`
- **実際に回すバックログは小タスク単位**（図 1 枚・1 節・1 つの修正など）: Plan でレッスンを分解して作る

**動き出しのテーマ選定（どのレッスンから始めるか）はユーザーが行う。** それ以降（Plan → Do → Review）は自分で回す。**プランの合意やタスクごとの受け入れを待たない。自分が完璧と思える品質に達したら、次のタスクへ進む。** 1 レッスンは、Plan で小タスクに分解してから、各タスクをこのループで仕上げる。大きくまとめて一気に作らない。

1. **Plan（作る前に、プランを固める）**
   - 構成案を作る: テーマ → 柱（3 つ程度）→ 各節、図の要否、ペルソナ適合、そして**技術で裏取りすべき点**
   - 技術前提を**この段階で一次情報（公式ドキュメント）で確認**する。実装してから前提が覆らないようにする
   - **塊レベルの網羅を先に確認する**: テーマに標準の体系がある場合（例: テストは JSTQB のテスト技法分類、a11y は WCAG）、ロードマップをその体系に当てて穴を洗う。個々のレッスンの質だけを見て「あるものを磨く」に流れると、束ねる柱や一系統が丸ごと抜けたまま気づけない
   - レッスンを**小タスクに分解**してバックログにする（合意は待たず、そのまま実装へ）
2. **Do**: `draft` で 1 小タスクを実装する（新規レッスンは add-lesson スキル）
3. **Review（実物で、あらゆる観点から、何度も）**: 自分が完璧と思える品質になるまで繰り返す。「前にやった」「1 回見た」で止めない
   - ビルド（`npm run docs:build`）／太字（`check-bold.js`）／タイトル（`check-lesson-title.js`、公開レッスンのみ）
   - `review-lesson` の全カテゴリ（A〜I）
   - **技術検証**: 扱う API・仕様を一次情報で裏取りする。記憶で書かない
   - **実物の目視**: 図は実際にレンダリングしてライト・ダーク・グレースケールで確認する（色だけに依存していないか）。表・コードの表示崩れもプレビューで確認する
   - ここで完璧と思える品質に達したら、合意を待たず次のタスクへ進む
4. **Retro（振り返り）**: ユーザーのレビューが入ったら、その指摘を `retrospective.md`（draft）に記録し、再発防止を**仕組みに還元**する（skills / CLAUDE.md / hooks の更新。例: タイトル抜けに対して `check-lesson-title.js` を追加した）。記憶に頼る運用に戻さない

- ユーザーのレビューは合意ゲートではなく、随時のフィードバック。来たら Retro に取り込み、仕組みに変える
- 自分の判断で最高品質まで詰めて進む。ただし報告では「完璧」と断言はしない（最終判定は読者役のユーザー）。機械チェックの通過と判断レビューは分けて報告する

## 執筆方針

### 何を・どの順で教えるか

- **常に最新版を参照する**: コンテンツで扱うライブラリ・フレームワーク・Web 標準は、必ず執筆時点の最新安定版に基づいて書く。古いバージョンの情報を載せない。レッスン作成・修正時には以下を必ず行うこと:
  1. 公式ドキュメント（Context7 MCP または Web 検索）で最新の API・仕様を確認する
  2. `tech-versions.md`（draft ブランチ）と照合し、矛盾があれば `tech-versions.md` を更新する
  3. 同じ技術を扱う他のレッスンとの整合性を確認する
- **仕組みから語ることで引き出しの質を上げる**: 引き出しを増やすだけなら表面的な紹介で足りるが、仕組みや背景から語ることで記憶に残りやすくなり、応用が利く引き出しになる。「CSS Modules というものがある」より「CSS はグローバルスコープだから名前が衝突する。それを解決するのが CSS Modules」と語るほうが、AI に指示を出すときにも的確な言葉が出てくる
- **素の技術はフレームワークの引き出しの前提条件**: 素の HTML/CSS/JavaScript の仕組みを知らなければ、React や Next.js が何を解決しているか伝わらない。CSS のグローバルスコープ問題を知らなければ CSS Modules や Tailwind の価値がわからず、DOM 操作の煩雑さを知らなければ宣言的 UI のありがたみがわからない。基礎は「教養」ではなく、フレームワークの引き出しが意味を持つための必須の土台として位置づける
- **ピラミッド構造でトップダウンに整理する**: レッスンの構成はテーマ → 柱（3つ程度） → 各柱の詳細、の順で設計する。セクションを並べる前に「何を伝えたいか」を上位から整理し、各セクションがピラミッドのどこに位置するか明確にする
- **カタログ構成を避け、問い駆動の因果の鎖で書く（薄さの主因対策）**: レッスンが薄くなる根本原因は、掘り下げ不足という個々の節の問題ではなく、**トピックを横並びに列挙する「カタログ構成」を選んでいること**にある。カタログ構成にした時点で各節は「〜というものがある＋表1枚」に痩せ、15分に収めようとすると掘り下げが削れる。防ぐには構成の型を変える:
  - 書き出す前に、**このレッスンが答える「問い」を1文**で書く（「〜とは何か」の紹介ではなく「なぜ〜なのか」「〜すると何が起きるか」の形）
  - **各節がその問いへの因果の一歩**になる1本の鎖にする（「盗聴されると困る → 暗号化する → でも相手は本物か → 証明書」のように前の節が次を呼ぶ）。横並びで因果に繋がらない節は削るか統合する
  - **比較表・属性表・分類表の前に、具体を1つ最後まで追う節を必ず置く**（リクエスト1往復・操作1回・値の一生など）。表は「追った流れの整理」に限る
  - 「15分・簡潔に」は「掘るな」ではなく「**核を1つに絞れ**」の意味。核が絞れていれば、絞った1本は深く掘れる

### 読者（ペルソナ）に向き合う

- **初心者目線**: 専門用語は初出時に必ず平易な言葉で説明する。「知っていて当然」という態度を取らない
- **ペルソナの共感を軸にする**: 読者（AI で Next.js アプリを作った経験があるが中身は理解していない初心者）が共感できるかどうかを常に意識する。読者が経験していない架空のシナリオや、実務経験がないと共感できない問題設定は避ける。「見たことがある」「使ったことがある」から入る
- **コードを読んだ経験を仮定しない**: ペルソナは AI に丸投げした経験はあっても、生成されたコードを注意深く読んだ経験はない。「AI が書いたコードで〜を見たことがあるはず」「〜をそのままにしていないでしょうか」のように、コードの中身に気づいた経験を読者に仮定する書き方をしない。「AI はこういうコードを書いてくる」という事実の提示として書く。「見たことがある」前提にしてよいのは UI などの利用者としての体験のみ
- **「AI に〜」という枕を共感できる場面に限る**: 説明の導入で「AI に〜と頼むと」「AI はこう書いてくる」を使うのは、読者がその AI 体験を実際にやったと共感できる場面だけにする。仕組みやデフォルト挙動など、AI 体験と関係なく成り立つ話に「AI に任せると」を枕として被せない（読者は共感できず、説得力が下がる）。AI への言及で価値が高いのは「AI にどう指示するか・AI の出力をどう評価するか」という語彙づくりの文脈であり、それは積極的に使ってよい。判定基準は「読者がその AI 体験を自分のこととして思い出せるか」
- **「AI への指示」を独立した節として毎回入れない**: 上の語彙づくりが許可されているからといって、レッスンの終わりに「AI への指示」的な節を定型で置く必要はない。**既定は「節を作らない」で、そのレッスン固有の指示例がよほど有用なときだけ**、本文の該当箇所に自然に混ぜて触れる。他のレッスンと見出し・構成が似た「指示」「確認」のテンプレートを複製しない（同じ型を繰り返すとテンプレートの穴埋めに見え、内容が薄くなる）

### レッスンの分量と独立性

- **簡潔に**: 15分で読み切れるボリュームを意識する。網羅性より「今日はこれだけ覚えれば OK」を重視する
- **読むだけで完結**: 演習セクションは設けない。LT を聞く感覚で、読むだけで完結する内容にする
- **レッスン単体で完結させる**: 各レッスンは他の Day を読んでいなくても違和感がないように書く。「前回学んだ〜」「Day X で〜」のような他レッスンへの参照は避ける。前提知識が必要な場合は、参照ではなくその場で簡潔に説明する
- **繋がりを意識**: 前日のレッスンで学んだ内容を活かす場面があれば積極的に触れる
- **ゴールとまとめは短く、本文の代わりにしない**: 15分で読み切る前提なので、「今日のゴール」と末尾の「まとめ」は本文の要約ではない。ゴールは読む前の方向づけ、まとめは後から要点を思い出すトリガーに徹する。
  - それぞれ**3項目程度**（多くても4）に絞る
  - 各項目は1行・1文以内の短いフレーズにし、本文のように説明を書き込まない。まとめは凝縮したキーワード的な一覧にとどめる
  - 本文の説明を繰り返さない。長く書くくらいなら本文を読めばよい、という基準で削る
  - **項目が増えそうなときは、分割して数を増やすのではなく統合して数を減らす**（過去に「1文1項目」を分割の口実にしてまとめが7項目に膨張した事故がある）

### 自然な日本語で書く

文章品質は「**同僚に口頭で説明するとき、その言い回しを使うか**」を基準にセルフチェックする。使わない言い回しは AI 生成感・直訳調・足場であり、型ごと避ける。

- **根本原因は「英語の発想を日本語に直訳すること」**: 不自然さの多くはここから生まれる。英語の比喩をそのまま訳す（tool→道具、fold→畳む、discard→捨てる、under the hood→裏側）、英語の構文をなぞる（名詞化、不自然な受動態、「the design that …→〜という設計です」）、英語の説明の修辞（問いかけ+即答、ドラマチックな締め）。下の各項目はこの症状を型ごとに並べたもの。書くときは「英語を訳す」のではなく、伝えたい中身を**日本語話者が口頭で言うならどう言うか**で書き起こす

- **長い段落を作らない**: 総量が多いこと自体は問題ないが、長い文章の塊が並ぶと読者は読む気をなくす。エッセンスは削らず、3文以上続く説明や複数の論点を含む段落は、短文・箇条書き・表・図に分解して視覚的に追えるようにする。「削って短くする」のではなく「分解して読みやすくする」
- **ダッシュ（——）を文中に使わない**: 日本語の文章中に em dash（——）を挟む書き方は不自然。文を分けるか、カッコを使う
- **括弧書き・記号（＝・＋）を多用しない**: 補足を「（…）」で何度も挟む、語を「＝」「＋」でつなぐ書き方は AI 生成感が強く出る。型ごと避ける。
  - 装飾的な括弧（「（コストがかかる）」「（毎回生成）」など）は、文に溶かすか別の文にする
  - 括弧を使ってよいのは「後で実際に使う用語・コードや会話で出会う語の説明」「短い例の列挙」に限る。一度きりで平易な言葉で足りるなら出さない（「写し（スナップショット）」のように使い回さない語を括弧で足さない）
  - 「A＝B」は「A は B」、「A ＋ B」は「A に B を足す」のように記号でなく言葉で書く
- **直訳調の日本語を避ける**: 英語の技術文書を下敷きにすると直訳調の日本語が混じりやすい。以下のパターンを意識してセルフチェックすること:
  - 名詞化の硬さ: 「〜の多さ」「〜の欠如」→ 日本語として自然な表現に言い換える
  - 不自然な受動態: 「処理されています」「提供されています」→ 能動態にできないか検討する
  - 英語由来のラベル語: 「落とし穴」(pitfall)、「柱」(pillar)、「層」(layer) を見出しや番号付きラベルに使わない。番号を振るなら「1. 具体名」のように中身を表す言葉だけで十分
  - 回りくどい修飾: 「〜だった瞬間に消えていた」のような多重構造 → 「〜のままでは見えなかった」のように素直に
  - カタカナ専門用語の安易な採用: 「断片化」(fragmentation) など、前文で平易に言い換えた直後に急にカタカナ・漢語に切り替えない
- **AI 生成感のある詩的な文体を避ける**: 以下のパターンはセルフチェックで潰すこと
  - 体言止めの列挙で始める: 「赤いエラー、緑の成功。」→ 文の中に組み込む（「画面の色には意味を持たせることが多いですが（赤はエラー、緑は成功など）」）
  - 短文の連打でリズムを作る: 「虹の色順と同じです。リンゴが赤く見えるのは〜」→ 因果を1文に繋げる
  - 問いかけ + 即答の語り口: 「なぜ〜なのか。答えは〜です。」→ 平叙文にする（「〜の理由は、〜と比べると分かります」）
  - ドラマチックな一文で締める: 「感度曲線の形がその答えです。」→ 具体的に書く（「感度曲線を見ると、〜」）
- **AI が書きがちな「説明の足場」を本文に残さない**: 内容を整理するための骨組み（メタな前置き・数え上げの宣言・ラベル・教科書的な締め）は、書き手が考えるときには役立つが、読み手にはノイズになる。これらは禁止フレーズを覚える話ではなく、**文の型として現れる癖**なので、型ごと避ける。判定基準は「**同僚に口頭で説明するとき、その言い回しを使うか**」。使わないなら足場なので削り、中身だけ残す。代表的な型（これだけではない。同類はすべて対象）:
  - メタな前置き: 「まず全体像を押さえると」「ここでは〜を見ていきます」「結論から言うと」「〜という地図」
  - 数え上げの宣言と展開: 「違いは大きく 2 つです」→「1 つ目は〜」「2 つ目は〜」。予告せず、いきなり中身から書く
  - 番号・カテゴリのラベル見出し: 「違い 1:」「ポイント 2:」→ 中身を表す見出しにする（番号を振るなら「1. 具体名」）
  - 教科書的な締め: 「〜という設計です」「〜が重要です」「〜と言えるでしょう」で段落を結ばない
  - 不自然な比喩でわかった気にさせる: 「〜の地図」「〜の正体」「〜という旅」など、中身を具体語で書けば不要なもの
- **見出しは語り口調にしない**: 見出しは目次にもなるため、名詞句や体言止めで簡潔に書く。「〜しよう」「〜ですよね」「〜してみる」のような話しかけ調は使わない
  - 詩的・文学的な単語を見出しに使わない: 「正体」「裏側」「命を吹き込む」「鍵」「秘密」「魔法」「旅」「世界」「扉」「真実」は技術文書の見出しとして不自然。中身を表す具体的な言葉を使う（「正体」→「仕組み」「実体」「役割」、「裏側」→「仕組み」「注意点」）

### コード・図・アクセシビリティ

- **コード例は正確に**: コード例は断片ではなく、文脈がわかる完全なコードを載せる。ただし手を動かすことは前提としない。読むだけで概念が伝わる構成にする
- **アクセシビリティを意識**: コード例ではセマンティックな HTML を使い、適切な alt 属性、aria 属性、キーボード操作への配慮などを自然に盛り込む。「なぜそう書くのか」をアクセシビリティの観点から説明する。専用レッスンとしてではなく、各レッスンの中で当たり前の習慣として扱う
- **図は Mermaid を使う**: フローやシーケンスなどの図は ASCII アートではなく Mermaid（`\`\`\`mermaid`）で記述する。VitePress の Mermaid プラグインでレンダリングされる

### VitePress デモの技術的注意

- **インタラクティブデモはダークモード対応必須**: レッスン内に HTML デモを埋め込む場合、VitePress のダークモードでテキストが見えなくなる問題を防ぐため、以下を必ず守ること:
  - 明るい背景（`background:#f8fafc` や `background:white`）を持つコンテナには `color:#1e293b` を明示的に指定する（VitePress ダークモードでテキスト色が白に変わり、明るい背景上で見えなくなるため）
  - `<dialog>` や `popover` など独立して表示される要素にも `background:white;color:#1e293b` を指定する
  - `<details>` など子要素にも `background:white` を指定して背景を明示する
  - 補足テキストに `color:#475569` や `color:#64748b` を使う場合は、背景色とのコントラスト比が十分か確認する
- **デモの CSS クラス名・id はファイル固有プレフィックスを必須**: VitePress は全 `.md` の `<style>` ブロックを1つのグローバル CSS にまとめるため、ファイル間でクラス名が衝突するとレンダリングが壊れる（過去に Day 4 と html-candidates/13 の `.demo-base-select` が衝突し、前者で Android Chrome の表示崩れが発生）。`<style>` 内のセレクタと、対応する HTML の `class=` / `id=` / `name=` は以下の名前空間で必ず分けること:
  - レッスン: `day{NN}-` プレフィックス（例: `.day04-base-select`、`id="day04-custom-select"`）
  - HTML 候補: `c{NN}-` プレフィックス（例: `.c13-base-select`）
  - 汎用名（`.modal`、`.card`、`.box`、`.container`、`.hero`、`.tooltip`、`.trigger`、`.btn`、`.form`、`.hint`、`.item`、`.row`、`.demo-*` 等）を裸で使わない
  - `<style scoped>` は VitePress の Markdown では効かないため頼らない
  -  ```css や ```html の**コードブロック内**の例示用クラス名は対象外（実際のスタイルとして出力されないため、教育上わかりやすい名前を優先してよい）
