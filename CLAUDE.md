# Web Front-end Training

## プロジェクト概要

エンジニア初心者が Web フロントエンドの知識の引き出しを増やすための学習コンテンツ。
毎日15分の LT を聞く感覚で、HTML/CSS の基礎から Next.js まで幅広いトピックに触れる。
VitePress で構築されたドキュメントサイトとして提供する。

## ターゲット

- エンジニア初心者。AI 丸投げ開発の経験が多少ある程度
- これから Next.js（App Router）を使うプロジェクトに配属される予定の人
- HTML/CSS/JavaScript の基礎を自分の言葉で説明できないレベル

## 配属先の技術スタック

- Next.js（App Router、最新バージョンに追従）
- TypeScript
- Tailwind CSS

## ゴール

Web フロントエンドに関する知識の引き出しを増やすこと。何かの場面で「この話、どこかで聞いたな」と思い出せる状態を作る。

- Server Components と Client Components という概念があること、なぜ分かれているかを知っている
- データ取得にはどこで fetch するか・キャッシュをどうするかという設計観点があると知っている
- パフォーマンス、アクセシビリティ、テスト、CI/CD といった観点が存在することを知っている
- 「なぜその技術を使うのか」という問いが立てられる

これらの引き出しは、AI への指示や、AI の成果物を判断する際の土台になる。

## コンテンツの使われ方

- 1日1レッスン、約15分で取り組む想定
- 毎日少しずつ進めてもらう（一気にやるものではない）
- 順番に進める前提（前の Day の知識を前提にしてよい）
- 平日のみ、約2.5ヶ月（約51日分）で完了する想定

## ブランチ運用と情報の永続化

- 決定事項や方針は必ずファイルに記録し、コミット・プッシュする。会話の中だけに情報を残さない
- **main ブランチ**: プロジェクト方針（CLAUDE.md）、スキル定義、VitePress 設定など、コンテンツの具体に関わらないもの。`docs/index.md` や `docs/introduction/` は main に置くが、未公開レッスンへのリンクや具体的なレッスン内容は含めない
- **draft ブランチ**: レッスン本文、カリキュラム（curriculum.md）、サイドバー設定など、コンテンツの具体に関わるもの
- コンテンツは draft に蓄積し、公開時に publish-lesson スキルで Day 単位の PR を作成して main にマージする
- **draft の内容は何も確定していない**: draft のレッスンの内容・順番・数はいつでも変わりうる仮の状態であり、確定しているのは publish-lesson で main に公開されたものだけ。main には公開済みの情報のみを含め、未公開の情報（レッスンリンク、Day 数、カリキュラムの構成など）は一切含めないこと
- main で方針（CLAUDE.md 等）を変更した場合は、draft にマージして同期すること。draft が古い CLAUDE.md を持ったままだと、以降のセッションで方針に従えなくなる

### 許可されたブランチ

| ブランチ | 用途 | 変更してよいファイル |
|---------|------|---------------------|
| `main` | 方針・設定 | CLAUDE.md、`.claude/`、`docs/index.md`（公開済みリンクのみ）、`docs/introduction/`、`docs/.vitepress/config.mts`（公開済みのみ）、`package.json` |
| `draft` | コンテンツ | `docs/lessons/`、`curriculum.md`、`tech-versions.md`、`docs/index.md`、`docs/.vitepress/config.mts` |
| `publish/dayXX` | 公開用（一時的） | publish-lesson スキルが作成・管理する。該当 Day のファイルのみ |

**これ以外のブランチを作成・使用してはならない。** feature ブランチ、個人ブランチ等は禁止。

## 執筆方針

- **常に最新版を参照する**: コンテンツで扱うライブラリ・フレームワーク・Web 標準は、必ず執筆時点の最新安定版に基づいて書く。古いバージョンの情報を載せない。レッスン作成・修正時には以下を必ず行うこと:
  1. 公式ドキュメント（Context7 MCP または Web 検索）で最新の API・仕様を確認する
  2. `tech-versions.md`（draft ブランチ）と照合し、矛盾があれば `tech-versions.md` を更新する
  3. 同じ技術を扱う他のレッスンとの整合性を確認する
- **仕組みから語ることで引き出しの質を上げる**: 引き出しを増やすだけなら表面的な紹介で足りるが、仕組みや背景から語ることで記憶に残りやすくなり、応用が利く引き出しになる。「CSS Modules というものがある」より「CSS はグローバルスコープだから名前が衝突する。それを解決するのが CSS Modules」と語るほうが、AI に指示を出すときにも的確な言葉が出てくる
- **素の技術はフレームワークの引き出しの前提条件**: 素の HTML/CSS/JavaScript の仕組みを知らなければ、React や Next.js が何を解決しているか伝わらない。CSS のグローバルスコープ問題を知らなければ CSS Modules や Tailwind の価値がわからず、DOM 操作の煩雑さを知らなければ宣言的 UI のありがたみがわからない。基礎は「教養」ではなく、フレームワークの引き出しが意味を持つための必須の土台として位置づける
- **初心者目線**: 専門用語は初出時に必ず平易な言葉で説明する。「知っていて当然」という態度を取らない
- **コード例は正確に**: コード例は断片ではなく、文脈がわかる完全なコードを載せる。ただし手を動かすことは前提としない。読むだけで概念が伝わる構成にする
- **読むだけで完結**: 演習セクションは設けない。LT を聞く感覚で、読むだけで完結する内容にする
- **簡潔に**: 15分で読み切れるボリュームを意識する。網羅性より「今日はこれだけ覚えれば OK」を重視する
- **繋がりを意識**: 前日のレッスンで学んだ内容を活かす場面があれば積極的に触れる
- **アクセシビリティを意識**: コード例ではセマンティックな HTML を使い、適切な alt 属性、aria 属性、キーボード操作への配慮などを自然に盛り込む。「なぜそう書くのか」をアクセシビリティの観点から説明する。専用レッスンとしてではなく、各レッスンの中で当たり前の習慣として扱う
- **図は Mermaid を使う**: フローやシーケンスなどの図は ASCII アートではなく Mermaid（`\`\`\`mermaid`）で記述する。VitePress の Mermaid プラグインでレンダリングされる
