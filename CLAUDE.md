# Web Front-end Training

## プロジェクト概要

エンジニア初心者が Web フロントエンドの知識の引き出しを増やすための学習コンテンツ。
毎日15分の LT を聞く感覚で、HTML/CSS の基礎から Next.js まで幅広いトピックに触れる。
VitePress で構築されたドキュメントサイトとして提供する。

## ターゲット

- エンジニア初心者。AI 丸投げ開発の経験が多少ある程度
- これから Next.js（App Router）を使うプロジェクトに配属される予定の人
- HTML/CSS/JavaScript の基礎を自分の言葉で説明できないレベル

## 配属先の技術スタック

- Next.js（App Router、最新バージョンに追従）
- TypeScript
- Tailwind CSS

## ゴール

Web フロントエンドに関する知識の引き出しを増やすこと。何かの場面で「この話、どこかで聞いたな」と思い出せる状態を作る。

- Server Components と Client Components という概念があること、なぜ分かれているかを知っている
- データ取得にはどこで fetch するか・キャッシュをどうするかという設計観点があると知っている
- パフォーマンス、アクセシビリティ、テスト、CI/CD といった観点が存在することを知っている
- 「なぜその技術を使うのか」という問いが立てられる

これらの引き出しは、AI への指示や、AI の成果物を判断する際の土台になる。

## コンテンツの使われ方

- チームで毎日30分の枠を確保して取り組む想定
  - 前半15分: その日のレッスンを読む
  - 後半15分: Q&A・派生した議論
- 1日1レッスン、読む部分は約15分で読み切れるボリューム
- 毎日少しずつ進めてもらう（一気にやるものではない）
- 順番に進める前提（前の Day の知識を前提にしてよい）

## ブランチ運用と情報の永続化

- 決定事項や方針は必ずファイルに記録し、コミット・プッシュする。会話の中だけに情報を残さない
- **main ブランチ**: プロジェクト方針（CLAUDE.md）、スキル定義、VitePress 設定など、コンテンツの具体に関わらないもの。`docs/index.md` や `docs/introduction/` は main に置くが、未公開レッスンへのリンクや具体的なレッスン内容は含めない
- **draft ブランチ**: レッスン本文、カリキュラム（curriculum.md）、サイドバー設定など、コンテンツの具体に関わるもの
- コンテンツは draft に蓄積し、公開時に publish-lesson スキルで Day 単位の PR を作成して main にマージする
- **draft の内容は何も確定していない**: draft のレッスンの内容・順番・数はいつでも変わりうる仮の状態であり、確定しているのは publish-lesson で main に公開されたものだけ。main には公開済みの情報のみを含め、未公開の情報（レッスンリンク、Day 数、カリキュラムの構成など）は一切含めないこと
- main に変更があった場合（方針変更、レッスン公開など）は、必ず draft と open 中の全 `publish/dayXX` ブランチに同期すること。古い main の状態を持ったままだと、以降の作業やマージで不整合が起きる
- 同期は手動で行う。main への push 後、`git merge origin/main` で draft に反映し、open 中の `publish/*` PR にはチェーンの根元から順に `gh pr update-branch` を実行する。コンフリクトしたら手動で解決する

### 許可されたブランチ

| ブランチ | 用途 | 変更してよいファイル |
|---------|------|---------------------|
| `main` | 方針・設定 | CLAUDE.md、`.claude/`、`.github/`、`docs/index.md`（公開済みリンクのみ）、`docs/introduction/`、`docs/.vitepress/config.mts`（公開済みのみ）、`package.json` |
| `draft` | コンテンツ | `docs/lessons/`、`docs/drafts/`、`docs/rejected/`、`curriculum.md`、`tech-versions.md`、`docs/index.md`、`docs/.vitepress/config.mts` |
| `publish/dayXX` | 公開用（一時的） | publish-lesson スキルが作成・管理する。該当 Day のファイルのみ |

**これ以外のブランチを作成・使用してはならない。** feature ブランチ、個人ブランチ等は禁止。

### コンテンツのディレクトリ構成

| ディレクトリ | 用途 | 命名規則 |
|-------------|------|---------|
| `docs/lessons/dayXX/` | 公開済みレッスン | `dayXX`（XX は公開順の 0 埋め 2 桁） |
| `docs/drafts/<ID>/` | 候補の下書き | 3 桁ゼロパディング連番（001, 002, ...） |
| `docs/rejected/<ID>/` | 却下された候補 | 3 桁ゼロパディング連番（001, 002, ...） |

- `docs/drafts/` と `docs/rejected/` の連番は単なる ID として扱う。順序に意味はなく、欠番 OK、詰め直し不要
- 新規追加時は `docs/drafts/` 配下の最大番号 +1 を採番する
- 公開時は `docs/drafts/<ID>/` → `docs/lessons/dayXX/` に移動する（publish-lesson スキルが行う）
- 却下時は `docs/drafts/<ID>/` → `docs/rejected/<ID>/` に移動する
- `docs/rejected/` はサイドバーに出さない保管庫として扱う

## PR 運用

- **PR の説明は常に最新を保つ**: PR 作成後にコミットを追加・修正した場合や、レビューで変更が入った場合は、**必ず PR 本文を最新の内容に更新する**こと。タイトル・サマリ・変更点・テストプランすべてが最終状態を反映している必要がある。古い説明のまま放置しない
- PR 本文にはレビューに必要な情報だけを書く。作業メモや会話の経緯、内部的な注意事項は含めない

## 執筆方針

- **常に最新版を参照する**: コンテンツで扱うライブラリ・フレームワーク・Web 標準は、必ず執筆時点の最新安定版に基づいて書く。古いバージョンの情報を載せない。レッスン作成・修正時には以下を必ず行うこと:
  1. 公式ドキュメント（Context7 MCP または Web 検索）で最新の API・仕様を確認する
  2. `tech-versions.md`（draft ブランチ）と照合し、矛盾があれば `tech-versions.md` を更新する
  3. 同じ技術を扱う他のレッスンとの整合性を確認する
- **仕組みから語ることで引き出しの質を上げる**: 引き出しを増やすだけなら表面的な紹介で足りるが、仕組みや背景から語ることで記憶に残りやすくなり、応用が利く引き出しになる。「CSS Modules というものがある」より「CSS はグローバルスコープだから名前が衝突する。それを解決するのが CSS Modules」と語るほうが、AI に指示を出すときにも的確な言葉が出てくる
- **素の技術はフレームワークの引き出しの前提条件**: 素の HTML/CSS/JavaScript の仕組みを知らなければ、React や Next.js が何を解決しているか伝わらない。CSS のグローバルスコープ問題を知らなければ CSS Modules や Tailwind の価値がわからず、DOM 操作の煩雑さを知らなければ宣言的 UI のありがたみがわからない。基礎は「教養」ではなく、フレームワークの引き出しが意味を持つための必須の土台として位置づける
- **初心者目線**: 専門用語は初出時に必ず平易な言葉で説明する。「知っていて当然」という態度を取らない
- **コード例は正確に**: コード例は断片ではなく、文脈がわかる完全なコードを載せる。ただし手を動かすことは前提としない。読むだけで概念が伝わる構成にする
- **読むだけで完結**: 演習セクションは設けない。LT を聞く感覚で、読むだけで完結する内容にする
- **簡潔に**: 15分で読み切れるボリュームを意識する。網羅性より「今日はこれだけ覚えれば OK」を重視する
- **繋がりを意識**: 前日のレッスンで学んだ内容を活かす場面があれば積極的に触れる
- **レッスン単体で完結させる**: 各レッスンは他の Day を読んでいなくても違和感がないように書く。「前回学んだ〜」「Day X で〜」のような他レッスンへの参照は避ける。前提知識が必要な場合は、参照ではなくその場で簡潔に説明する
- **ピラミッド構造でトップダウンに整理する**: レッスンの構成はテーマ → 柱（3つ程度） → 各柱の詳細、の順で設計する。セクションを並べる前に「何を伝えたいか」を上位から整理し、各セクションがピラミッドのどこに位置するか明確にする
- **ペルソナの共感を軸にする**: 読者（AI で Next.js アプリを作った経験があるが中身は理解していない初心者）が共感できるかどうかを常に意識する。読者が経験していない架空のシナリオや、実務経験がないと共感できない問題設定は避ける。「見たことがある」「使ったことがある」から入る
- **アクセシビリティを意識**: コード例ではセマンティックな HTML を使い、適切な alt 属性、aria 属性、キーボード操作への配慮などを自然に盛り込む。「なぜそう書くのか」をアクセシビリティの観点から説明する。専用レッスンとしてではなく、各レッスンの中で当たり前の習慣として扱う
- **図は Mermaid を使う**: フローやシーケンスなどの図は ASCII アートではなく Mermaid（`\`\`\`mermaid`）で記述する。VitePress の Mermaid プラグインでレンダリングされる
- **インタラクティブデモはダークモード対応必須**: レッスン内に HTML デモを埋め込む場合、VitePress のダークモードでテキストが見えなくなる問題を防ぐため、以下を必ず守ること:
  - 明るい背景（`background:#f8fafc` や `background:white`）を持つコンテナには `color:#1e293b` を明示的に指定する（VitePress ダークモードでテキスト色が白に変わり、明るい背景上で見えなくなるため）
  - `<dialog>` や `popover` など独立して表示される要素にも `background:white;color:#1e293b` を指定する
  - `<details>` など子要素にも `background:white` を指定して背景を明示する
  - 補足テキストに `color:#475569` や `color:#64748b` を使う場合は、背景色とのコントラスト比が十分か確認する
- **デモの CSS クラス名・id はファイル固有プレフィックスを必須**: VitePress は全 `.md` の `<style>` ブロックを1つのグローバル CSS にまとめるため、ファイル間でクラス名が衝突するとレンダリングが壊れる（過去に Day 4 と html-candidates/13 の `.demo-base-select` が衝突し、前者で Android Chrome の表示崩れが発生）。`<style>` 内のセレクタと、対応する HTML の `class=` / `id=` / `name=` は以下の名前空間で必ず分けること:
  - レッスン: `day{NN}-` プレフィックス（例: `.day04-base-select`、`id="day04-custom-select"`）
  - HTML 候補: `c{NN}-` プレフィックス（例: `.c13-base-select`）
  - 汎用名（`.modal`、`.card`、`.box`、`.container`、`.hero`、`.tooltip`、`.trigger`、`.btn`、`.form`、`.hint`、`.item`、`.row`、`.demo-*` 等）を裸で使わない
  - `<style scoped>` は VitePress の Markdown では効かないため頼らない
  -  ```css や ```html の**コードブロック内**の例示用クラス名は対象外（実際のスタイルとして出力されないため、教育上わかりやすい名前を優先してよい）
