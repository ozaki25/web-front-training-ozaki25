# 振り返りログ（retrospective）

コンテンツ作成のループ（CLAUDE.md 参照）の **Retro フェーズ**で、ユーザーのレビュー指摘と、それに対する仕組みへの反映を記録する。記憶ではなく記録で再発を防ぐ。

> 過去（drafts 005〜103 を作った前フェーズ）の執筆・スプリント記録は `docs/drafts/writing-sprint-log.md`・`docs/drafts/theme-sprint-log.md` にある（凍結された履歴）。本ファイルは以降の振り返りを記録する。

## 書き方

新しいエントリを上に足す。各項目は次の形式:

- **指摘**（ユーザーのレビュー・受け入れで出たこと）
- **原因**（なぜ起きたか）
- **反映**（skill / CLAUDE.md / hook / 手順のどれをどう変えたか。単発の修正で終わらせない）

## 2026-07-03 — 全観点レビューの徹底（図偏重の是正）

- **指摘**: 「図だけやたらこだわってレビューしてる」「あらゆる観点でのレビューが必要」。図はレンダリングして何度も見るのに、本文の日本語・論理・まとめは 1 回書いて機械チェックを通しただけで「完璧」と判断していた。タスクの単位もレッスン 1 本と大きく、小タスクごとの改善サイクルになっていなかった
- **原因**: Review を「図の目視」に矮小化していた。本文は「書けた＝終わり」で、図と同じ密度で読み返していなかった
- **反映**:
  - 図に限らず全観点（技術・論理・日本語・構成・ペルソナ・ゴール/まとめ・見出し）を、図と同じ密度で毎回点検する。図のレンダリング目視は Review の一部であって全部ではない
  - 全観点で再レビューした結果、初回で見落としていた実質的な粗を検出・修正した:
    - 104: 「上下の2行だけ」の数え間違い（論理バグ）→ 「会員だが 3000 円未満（2 行目）が空」と正確化
    - 105: まとめの 2 文項目、「成功だけ緑でも」がテスト色の隠語で自己完結していない → 1 文化＋平易化
    - 106: まとめの 2 文項目、影響の例示（「A、と、B、では」）が読みにくい、費用対効果の言い回しが硬い → 自然化
  - **まとめの 2 文混入は 104/105/106 すべてで再発**した。まとめ点検は「各項目が 1 文か」を機械的に数える手順を Review で固定する（review-lesson C）

## 2026-07-03 — テスト計画・設計の新規 3 本（104/105/106）

- **指摘**: 「テスト計画・設計で数本」の依頼に対し、最初は既存のテスト系ドラフト（046/048/090 は技法寄り）を磨いただけで、計画・設計の中心は 049・047 の 2 本しかなかった。「手近なものを仕上げた」であって「依頼のカテゴリを狙って揃えた」ではなかった
- **原因**: Plan で「この作業が依頼のカテゴリ（計画／設計）にちゃんと当たっているか」を確認せず、既存ドラフトの消化に流れた
- **反映**:
  - 計画・設計の"あるべきセット"を定義し、欠けていた 3 本を新規作成: **104 デシジョンテーブル**・**105 状態遷移テスト**（設計技法）、**106 テスト計画＝どこまでテストするか**（計画）
  - 図は題材に最適な形で新規（決定表そのもの／状態遷移図／リスクマトリクス）。すべて色でなく位置・形・ラベルで伝え、グレースケールで確認
  - 手順として、Plan では「依頼の言葉（今回は"計画・設計"）に自分の作業が直接対応しているか」を最初に確認する

## 2026-07-03 — テスト系ドラフト 5 本（046/047/048/049/090）

- **図が最大の弱点だった（複数本で共通）**
  - 049 のピラミッド/トロフィーが Mermaid 縦積み＋色分けで「形」になっていない、047 に境界の図が無い、048 に核（同じ木の共有）の図が無い、090 は差し替え位置を示せていなかった
  - 原因: 図を「あれば足す」程度に扱い、題材に最適な図を詰めていなかった
  - 反映: 概念に最適な図を必ず検討する（ピラミッド＝幅で本数、境界＝数直線、共有＝分岐図、層＝切り込み位置）。すべて色ではなく形・幅・位置・ラベルで伝え、グレースケールで確認（review-lesson H の色非依存ルールを全図に適用）
- **まとめの各項目に 2 文入りがち**
  - 原因: 1 項目 1 文の制約を徹底できていなかった
  - 反映: まとめは各 1 文に圧縮（review-lesson C を毎回厳密に）
- **見出しに語り口調・比喩が混じる**（「で、どの割合で持つ？」「getByRole が王様」「エラーの日」など）
  - 反映: 見出しは名詞句・具体語に（review-lesson E を毎回厳密に）
- **記号 `=` / `+` の混入**（まとめ・チェック質問）
  - 反映: 言葉に置き換える（review-lesson D を毎回厳密に）
- **総括**: 「レビューが浅い」という指摘の実体は、機械チェックは通すが「初めて読む人が図・見出し・まとめで何を思うか」まで踏み込めていなかったこと。今後は Review で必ず図をレンダリングして目視し、見出し・まとめ・記号を 1 つずつ点検する

## 2026-07-03 — Day 32〜34（キャッシュ系）

- **図が色だけで区別していた**（Day 32 Router Cache）
  - 原因: 保持/取り直しを緑/赤の色のみで表現。色覚特性・モノクロで判別不能
  - 反映: 無地/斜線＋記号（✓ / ↻）など色以外の手がかりを併用する方式に変更。`review-lesson` の H に「図が色だけで区別していないか（グレースケールで判定）」を追加
- **公開レッスンのタイトルに Day 番号が抜けていた**（day06/31/32/33）
  - 原因: publish 手順のタイトル付与が記憶頼みで、機械チェックがなかった
  - 反映: `check-lesson-title.js` を追加し、publish-lesson の必須チェックに組み込み
- **技術的な誤り**（記憶での執筆が原因）
  - 「キャッシュ指定なしの fetch でページ全体が動的」→ 誤り。デフォルト fetch は静的、動的化は request 時 API か明示 no-store
  - 「revalidateTag の第2引数は必須」→ 正しくは「1 引数形が非推奨」
  - `cacheLife` の stale を「全プロファイル 5 分」→ 誤り。`seconds` は 30 秒
  - 反映: 執筆時は v16 等の一次情報で必ず裏取りする、をループの Review 手順（CLAUDE.md）に明記
- **レビューがユーザー先行だった**（色・日本語・技術の指摘が、私のレビューを素通りしてユーザーから出た）
  - 原因: レビューが「見せた後」に行われていた
  - 反映: 「見せる前に自分で粗を出し切る」Review フェーズを、コンテンツ作成のループとして CLAUDE.md に標準化
- **日本語の不自然な言い回し**（集約されます／どちらかへ倒す／消し込み 等の硬語・隠語）
  - 原因: セルフレビューで「同僚に口頭で言うか」の基準を徹底できていなかった
  - 反映: Review の `review-lesson` D を毎回徹底（既存方針の再確認）
