# レッスン追加スキル

Web Front-end Training に新しい下書きレッスンを追加します。
作業は `draft` ブランチ上で行います。

## 引数

- `$ARGUMENTS` にテーマと内容の説明が渡されます
- 例: `/add-lesson リンクと画像 aタグ、imgタグ、パス指定`

## 作業手順

以下のステップを**すべて**実行してください。

### 1. draft ブランチに切り替え

```bash
git fetch origin
git checkout draft
git pull origin draft
```

`draft` ブランチがまだ存在しない場合は `main` から作成する。

```bash
git checkout -b draft origin/main
```

### 2. 次の ID を決定

`docs/drafts/` 配下の最大連番 +1 を 3 桁ゼロパディングで採番します。

```bash
ls docs/drafts/ | sort -n | tail -1
```

例: 最大が `064` なら次は `065`。`docs/drafts/` が空なら `001`。

### 3. 既存レッスンの確認

既存の下書きや公開済みレッスンを読んで、現在の状態を把握してください。
関連するテーマのレッスンがあれば内容を確認し、重複や整合性を意識してください。

### 4. レッスンの設計（書き出す前に必ず固める）

**いきなり本文を書き始めないこと。** 薄い初稿の原因は、書き出す前に核が定まっておらず、トピックを横並びに列挙する「カタログ構成」になることにある。カタログ構成を選んだ時点で各節は「〜というものがある＋表1枚」に痩せ、15分に収めようとすると掘り下げが削れる。これを入口で防ぐため、書く前に次を固める。

- **このレッスンが答える「問い」を1文で書く**。「〜とは何か」の紹介ではなく、「なぜ〜なのか」「〜すると何が起きるのか」の形の問いにする。
  - 例（Cookie）:「なぜブラウザは勝手に値を送るのか。そしてそれがログイン維持の便利さと CSRF の危険の両方の source になるのはなぜか」
- **その問いに答える因果の鎖を1本引く**。各節は問いへの因果の一歩になっていること。「盗聴されると困る → だから暗号化する → でも相手は本物か → だから証明書」のように、前の節が次の節を呼ぶ形にする。
  - 横並びに並ぶだけで因果で繋がらない節は、**削るか、鎖のどこかに統合する**。「関連トピックだから」で足さない。
- **柱は3つ程度**に絞る。問いに答えるのに要らない柱は落とす。15分の制約は「掘るな」ではなく「**核を1つに絞れ**」の意味だと解釈する。
- **表・比較の前に、必ず具体を1つ最後まで追う節を置く**。属性表・比較表・分類表は「追った流れの整理」としてのみ許可する。表を出す前に、リクエスト1往復・ユーザー操作1回・値の一生など、**具体を1つ選んで因果を最後まで追う**節を先に書く。表だけの節が並んでいたら、その手前に具体のトレースが足りていない。

この設計をコメントやメモに書き出してから本文に進む。設計が「トピックの列挙」にしかならないなら、まだ核が絞れていない。

### 5. レッスンファイルの作成

`docs/drafts/<ID>/index.md` を作成します。

レッスンは以下の構成で、**15分程度で読み切れるボリューム**にしてください。

```markdown
# テーマ名

## 今日のゴール

- ゴール1（「〜を知る」の形式。認知・引き出しレベル）
- ゴール2

## 解説セクション（テーマに応じた見出しで複数セクション）

（概念の説明、コード例、表、ポイントなどを含む）
（読むだけで概念が伝わるように、文脈がわかる完全なコード例を載せる）

## まとめ

- 凝縮したキーワード的な箇条書き（本文の繰り返しにしない）
```

コンテンツの執筆方針は CLAUDE.md を参照すること。加えて:
- **ゴールとまとめは短く保つ**: ゴールもまとめもそれぞれ **3項目程度（多くても4）**・各1行に絞る。本文の説明を繰り返さない。項目が増えそうなときは、分割して数を増やすのではなく統合して数を減らす
- レッスンで扱うライブラリ・フレームワーク・Web 標準は、必ず Web 検索または公式ドキュメントで最新の API・仕様を確認してから書くこと。訓練データの知識だけで書かない
- `tech-versions.md` を読み、該当技術の最新バージョン・API 名・設定方法を確認すること。矛盾があれば `tech-versions.md` を更新する
- 同じ技術を扱う他のレッスンと整合性が取れているか確認すること

### 6. サイドバーの更新

`docs/.vitepress/config.mts` のサイドバーの「候補」セクションに新しい下書きを追加します。

```typescript
{ text: "テーマ名", link: "/drafts/<ID>/" },
```

### 7. ビルド確認

```bash
npx vitepress build docs
```

ビルドが通ることを確認してください。エラーがあれば修正します。

### 8. コミット & プッシュ

変更をコミットして `draft` ブランチにプッシュしてください。

コミットメッセージの形式:
```
「テーマ名」の下書きを追加 (drafts/<ID>)
```
