# レッスンレビュースキル

指定されたレッスンを、下記チェックリストの**全カテゴリ**で必ずレビューします。
場当たりに気づいた点だけ直すのは禁止。**毎回チェックリストを頭から最後まで通す**こと。

## 引数

- `$ARGUMENTS` にレビュー対象（`drafts/105`、`day26` など）
- 複数指定された場合は1本ずつ順にループを回す

## レビューのループ（仕組み）

1 本のレッスンについて、次を**指摘ゼロになるまで繰り返す**:

```
① 対象ファイルを最初から最後まで通読する
② 下のチェックリストを A→I の順に全項目チェックし、指摘を箇条書きで列挙する
③ 指摘を修正する
④ npm run docs:build でビルドが通ることを確認する
⑤ ②に戻る。指摘が出なくなったら、その時点で1周「指摘なし」を確認して終了
```

**1周でもチェックを飛ばさない。** 修正で別の問題を作ることがあるため、修正後は必ず全項目を再チェックする。

最後に、各カテゴリについて「指摘なし / 指摘あり（内容と対応）」を表で報告する。

## チェックリスト

### A. 仕組みの正確性・最新版（最優先）

- 扱う技術の仕組み・API を、公式ドキュメントまたは Web 検索で**執筆時点の最新**として確認したか（訓練データの記憶だけで書かない）
- **古い仕組み・非推奨 API・旧モデルを「現行」として説明していないか**。仕組みが変わっている場合、古い方を現行として書くのは不可
- 用語が複数ある場合、最新かつ分かりやすいものを選んでいるか
- `tech-versions.md` と整合しているか。矛盾があれば `tech-versions.md` を更新したか
- 同じ技術を扱う他レッスンと矛盾していないか

### B. 構成・設計

- **単一の核**になっているか。複数の核が混在して1本に詰め込まれていないか
- ピラミッド構造（テーマ → 柱3つ程度 → 各柱の詳細）で上位から整理されているか
- **レッスン単体で完結**しているか。「前回」「Day X で」など他レッスンへの参照がないか。前提知識はその場で簡潔に説明しているか

### C. ゴール・まとめ

- 「今日のゴール」「まとめ」がそれぞれ **3項目程度（多くても4）**か
- 各項目が **1行・1文以内**の短いフレーズか。本文のような説明を書き込んでいないか
- 本文の繰り返しになっていないか（ゴールは方向づけ、まとめは想起のトリガーに徹する）
- 項目が多いとき、分割ではなく統合で減らしているか

### D. 文章品質（直訳調・AI 詩的文体）

- 直訳調: 名詞化の硬さ（「〜の多さ」「〜の欠如」）、不自然な受動態、回りくどい修飾がないか
- 英語由来のラベル語（「落とし穴」「柱」「層」）を見出し・番号ラベルに裸で使っていないか
- AI 詩的文体: 体言止めの列挙、短文の連打でリズムを作る、問いかけ+即答、ドラマチックな一文で締める、がないか
- ダッシュ（——）を**文中**に使っていないか（見出しの区切りは可）
- 3文以上続く段落・複数論点の段落を、短文・箇条書き・表・図に分解しているか

### E. 見出し

- 名詞句で簡潔か。語りかけ調（「〜しよう」「〜ですよね」「〜してみる」）になっていないか
- 詩的・文学的な語（正体／裏側／命を吹き込む／鍵／秘密／魔法／旅／世界／扉／真実）を使っていないか。中身を表す具体語にする

### F. ペルソナ

- 読者（AI で Next.js アプリを作った経験はあるが中身は理解していない初心者）が共感できるか
- **コードを読み込んだ経験を仮定していないか**。「コードで〜を見たことがあるはず」ではなく「AI はこういうコードを書いてくる」という事実提示にする。「見たことがある」前提にしてよいのは UI など利用者としての体験のみ
- 専門用語を初出時に平易な言葉で説明しているか

### G. コード・アクセシビリティ

- コード例が断片でなく、文脈のわかる完全なコードか（読むだけで概念が伝わるか）
- セマンティックな HTML を使っているか。`alt`、`aria-*`、キーボード操作への配慮があるか
- 必要に応じて「なぜそう書くか」をアクセシビリティの観点から触れているか

### H. 図・デモ

- 図は Mermaid（` ```mermaid `）で書かれているか。ASCII アートを使っていないか
- HTML デモを埋め込む場合、ダークモード対応（明るい背景に `color:#1e293b` を明示等）がされているか
- デモの CSS クラス名・id・name にファイル固有プレフィックス（レッスンは `day{NN}-`、下書きは対応する接頭辞）が付いているか。汎用名を裸で使っていないか
- Mermaid のノードに明るい `fill` を使う場合、`color:#1e293b` を併記して文字が見えるか

### I. ビルド

- `npm run docs:build` が通るか
- 図（Mermaid）が壊れず、長い行・表が横スクロールで破綻していないか

## 出力フォーマット

各レッスンについて、最終的に次を報告する:

```
## drafts/XXX レビュー結果

| カテゴリ | 判定 | 指摘・対応 |
|---------|------|-----------|
| A 最新版 | ✅/⚠️ | ... |
| B 構成 | ... | ... |
| ...（C〜I まで全カテゴリ必ず記載） |

修正したコミット: <ハッシュ>
```

カテゴリを省略しない。指摘がなくても「✅ 指摘なし」と明記する。
