# レッスンレビュースキル

指定されたレッスンを、下記チェックリストの**全カテゴリ**で必ずレビューします。
場当たりに気づいた点だけ直すのは禁止。**毎回チェックリストを頭から最後まで通す**こと。

## 基準は CLAUDE.md

このチェックリストは運用の補助です。**品質の基準は `CLAUDE.md` の「執筆方針」セクション全体**です。

- レビュー開始時に、**毎回 `CLAUDE.md` の執筆方針を最初から最後まで読む**こと。記憶や下のチェックリストだけに頼らない
- 下のチェックリストに**載っていない方針が CLAUDE.md にあれば、それも必ず適用する**（チェックリストの記載漏れより CLAUDE.md が優先）
- CLAUDE.md の方針が更新されたら、**このスキルのチェックリストにも反映する**

## 引数

- `$ARGUMENTS` にレビュー対象（`drafts/105`、`day26` など）
- 複数指定された場合は1本ずつ順にループを回す

## レビューのループ（仕組み）

1 本のレッスンについて、次を**指摘ゼロになるまで繰り返す**:

```
① 対象ファイルを最初から最後まで通読する
② 下のチェックリストを A→I の順に全項目チェックし、指摘を箇条書きで列挙する
③ 指摘を修正する
④ npm run docs:build でビルドが通ることを確認する
⑤ ②に戻る。指摘が出なくなったら、その時点で1周「指摘なし」を確認して終了
```

**1周でもチェックを飛ばさない。** 修正で別の問題を作ることがあるため、修正後は必ず全項目を再チェックする。

最後に、各カテゴリについて「指摘なし / 指摘あり（内容と対応）」を表で報告する。

## チェックリスト

### A. 仕組みの正確性・最新版（最優先）

- 扱う技術の仕組み・API を、公式ドキュメントまたは Web 検索で**執筆時点の最新**として確認したか（訓練データの記憶だけで書かない）
- **古い仕組み・非推奨 API・旧モデルを「現行」として説明していないか**。仕組みが変わっている場合、古い方を現行として書くのは不可
- 用語が複数ある場合、最新かつ分かりやすいものを選んでいるか
- **テーマに業界標準の規格・体系がある場合、それを一次基準に照合したか**（例: テスト設計・計画は JSTQB/ISTQB シラバス、アクセシビリティは WCAG、HTTP は RFC）。標準がある分野では、用語・定義・図の構造をその標準に準拠させる（例: デシジョンテーブルは条件・アクションを行/各列＝ルール、状態遷移は状態・イベント・遷移・アクション、リスクは発生可能性×影響度）
  - **二次情報（解説記事・検索の要約）は入口にとどめ、用語の表記まで規格の条文そのもの（一次情報）で確認する**。二次情報で準拠させたつもりが条文と食い違った事故がある（決定表を Y/N → 正しくは T/F、「縮退」→「最小化」、リスクを「発生可能性/影響度」→「リスクの可能性/リスクの影響」）
- `tech-versions.md` と整合しているか。矛盾があれば `tech-versions.md` を更新したか
- 同じ技術を扱う他レッスンと矛盾していないか

### B. 構成・設計

- **単一の核**になっているか。複数の核が混在して1本に詰め込まれていないか
- **仕組み・背景から語っているか**。表面的な紹介（「〜というものがある」）で終わらず、なぜそれが必要か・どう動くかから説明しているか
- ピラミッド構造（テーマ → 柱3つ程度 → 各柱の詳細）で上位から整理されているか
- **レッスン単体で完結**しているか。「前回」「Day X で」など他レッスンへの参照がないか。前提知識はその場で簡潔に説明しているか
- **15分で読み切れるボリューム**か。網羅性より「今日はこれだけ覚えれば OK」を優先しているか
- **演習セクションを設けていない**か（読むだけで完結する構成か）
- **分類・一覧・地図系のレッスンは、挙げた項目すべてに実体があるか**（例: 「3 系統」と分類したら各系統の説明が、「代表技法は A・B・C」と挙げたらそれぞれに本文かレッスンが対応しているか）。地図に載せたのに中身が無い、という不整合を作らない
- **既存レッスンをセットに組み込むときは、新規と同じ基準で再監査したか**（用語の条文照合・図のレンダリング目視まで。「既存だから」で素通りさせない）

### C. ゴール・まとめ

- 「今日のゴール」「まとめ」がそれぞれ **3項目程度（多くても4）**か
- 各項目が **1行・1文以内**の短いフレーズか。本文のような説明を書き込んでいないか
  - **句点「。」の数を機械的に数える**。1 項目に「。」が 2 つ以上、または「。」で区切られた 2 文が入っていたら不可（例:「全部はテストできない。だから濃淡をつける」→「全部はテストできないから濃淡をつける」と 1 文に繋ぐ）。104/105/106 で 3 本連続で再発した癖なので、目視の印象で流さず必ず数える
- 本文の繰り返しになっていないか（ゴールは方向づけ、まとめは想起のトリガーに徹する）
- 項目が多いとき、分割ではなく統合で減らしているか

### D. 文章品質（直訳調・AI 詩的文体）

- 直訳調: 名詞化の硬さ（「〜の多さ」「〜の欠如」）、不自然な受動態、回りくどい修飾がないか
- 英語由来のラベル語（「落とし穴」「柱」「層」）を見出し・番号ラベルに裸で使っていないか
- **カタカナ専門用語の安易な採用**がないか。前文で平易に言い換えた直後に、急にカタカナ・漢語へ切り替えていないか
- AI 詩的文体: 体言止めの列挙、短文の連打でリズムを作る、問いかけ+即答、ドラマチックな一文で締める、がないか
- **「説明の足場」が残っていないか**（CLAUDE.md 参照）。メタな前置き／数え上げの宣言（「違いは大きく N つです」→「1 つ目は」）／番号ラベル見出し（「違い 1:」）／教科書的な締め（「〜という設計です」）／不自然な比喩（「〜の地図」「正体」）。**各段落を「同僚に口頭でこう言うか」で判定する**。言わない言い回しは足場なので削る
- ダッシュ（——）を**文中**に使っていないか（見出しの区切りは可）
- **括弧書き・記号（＝・＋）を多用していないか**。装飾的な「（…）」は文に溶かす。括弧は「後で使う用語・コードや会話で出会う語の説明」「短い例の列挙」に限る（一度きりで平易な語で足りるなら出さない）。「A＝B」「A ＋ B」は言葉（「A は B」「A に B を足す」）で書く
- 3文以上続く段落・複数論点の段落を、短文・箇条書き・表・図に分解しているか

### E. 見出し

- 名詞句で簡潔か。語りかけ調（「〜しよう」「〜ですよね」「〜してみる」）になっていないか
- 詩的・文学的な語（正体／裏側／命を吹き込む／鍵／秘密／魔法／旅／世界／扉／真実）を使っていないか。中身を表す具体語にする

### F. ペルソナ

- 読者（AI で Next.js アプリを作った経験はあるが中身は理解していない初心者）が共感できるか
- **コードを読み込んだ経験を仮定していないか**。「コードで〜を見たことがあるはず」という書き方はしない。「見たことがある」前提にしてよいのは UI など利用者としての体験のみ
- **未観測の AI 出力を前提にしていないか（重要）**。読者は AI でアプリを作った経験はあっても、AI の生成物（特にテストコードなど）を精査した経験は無い。だから「**AI に頼むと〜な出力が出る／〜が抜ける**」「**AI は〜に偏る**」のように、読者が見たこともない AI の出力・欠落を前提にした導入は共感されず逆効果（テスト系全般でこれをやって「ついていけない・迷惑」と指摘された）。
  - 導入は**概念・問題そのもの**から始める（「AI に頼むと」で始めない）
  - AI に触れるなら「**AI にどう指示するか**（語彙）」「**壁打ちにどう使うか**（協働）」に寄せる。「AI に『〜で書いて』と指示できる」は可。「AI が〜を出す／抜かす」は不可
  - 「できたテストを〜の観点で確認する」のように、**自分が書いても AI が書いても成り立つ**中立な書き方にする（特定の AI の癖を前提にしない）
- **「AI に〜」という枕を共感できない場面に被せていないか**。仕組み・デフォルト挙動など AI 体験と無関係に成り立つ話の枕に「AI に任せると」を使っていないか。一方「AI にどう指示するか・出力をどう評価するか」の語彙づくりは積極的に使ってよい
- 専門用語を初出時に平易な言葉で説明しているか

### G. コード・アクセシビリティ

- コード例が断片でなく、文脈のわかる完全なコードか（読むだけで概念が伝わるか）
- セマンティックな HTML を使っているか。`alt`、`aria-*`、キーボード操作への配慮があるか
- 必要に応じて「なぜそう書くか」をアクセシビリティの観点から触れているか

### H. 図・デモ

- 図は Mermaid（` ```mermaid `）で書かれているか。ASCII アートを使っていないか
- **図はコードを読むだけで判定せず、実際に画像へレンダリングして目視する**。ヘッドレスブラウザ等で SVG/Mermaid を書き出し、ライトとグレースケールの両方で、重なり・はみ出し・ラベルの読みやすさ・色依存を目で確認する（座標や属性の読みだけで「大丈夫」と判断しない。過去に描画確認を飛ばして崩れを見落とした）
- **図が色だけで区別していないか**。意味の区別（保持/取り直す、起点/波及など）は色に依存させず、必ず形・記号・パターン・位置・ラベルなど色以外の手がかりを併用する。グレースケールにしても読めるかで判定する。凡例・本文も「緑/赤」ではなく形や語で説明する（色覚特性やモノクロ表示でも伝わるように）
- HTML デモを埋め込む場合、ダークモード対応（明るい背景に `color:#1e293b` を明示等）がされているか
- デモの CSS クラス名・id・name にファイル固有プレフィックス（レッスンは `day{NN}-`、下書きは対応する接頭辞）が付いているか。汎用名を裸で使っていないか
- Mermaid のノードに明るい `fill` を使う場合、`color:#1e293b` を併記して文字が見えるか

### I. ビルド・描画

- `npm run docs:build` が通るか
- **`node .claude/hooks/check-style.js <file>` で型の欠陥（まとめの複文・詩的見出し・全角記号 ＝＋・足場フレーズ）を機械検出したか**。候補が出たら引用の格言・概念語・事実かを人が判断して、該当すれば直す（無引数で全 index.md を一括スキャンできる）
- 図（Mermaid）が壊れず、長い行・表が横スクロールで破綻していないか
- **太字 `**` が素通りしていないか**。markdown-it は全角約物（（）「」など）に隣接した `**` を、文中（前後が日本語）にあるとき太字にしないため、目視では気づけない。`node .claude/hooks/check-bold.js docs/drafts/XXX/index.md`（複数ファイル可）か `cat ... | node .claude/hooks/check-bold.js` で検査し、出力が空であることを確認する。**引数の渡し方を間違えると無言で通過するので、既知の崩れ（例: `テスト**強調（補足）**を置く`）で検査自体が exit 1 になることをたまに確認する**。崩れたら約物を `**` の外に出す（例: `**目次（一覧）**` → `**目次**（一覧）`）

## 出力フォーマット

各レッスンについて、最終的に次を報告する:

```
## drafts/XXX レビュー結果

| カテゴリ | 判定 | 指摘・対応 |
|---------|------|-----------|
| A 最新版 | ✅/⚠️ | ... |
| B 構成 | ... | ... |
| ...（C〜I まで全カテゴリ必ず記載） |

修正したコミット: <ハッシュ>
```

カテゴリを省略しない。指摘がなくても「✅ 指摘なし」と明記する。
