# エンプティステート — 0 件の画面を放置しない

## 今日のゴール

- 中身が空の画面はそれ自体が設計対象だと知る
- 空になる場面には初回・検索 0 件・全部完了の 3 種類があると知る
- 状況の説明と次の一歩を示すのがエンプティステートの基本形だと知る

## 空の画面は壊れて見える

アプリで何かを検索して、画面に「データがありません」の 1 行だけが出てきたときのことを思い出してください。多くの人は一瞬こう考えます。

- 読み込みに失敗した？
- 通信が切れている？
- 操作を間違えた？

実際には検索条件に合うものがなかっただけでも、素っ気ない空の画面は「壊れているのでは」という不安を生みます。不安が解けたとしても、今度は「ここから何をすればいいのか」という迷いが残ります。

こうした中身が空の画面の見せ方は、UI デザインの定番パターンとして**エンプティステート**（empty state）という名前が付いています。

> 空の画面は「表示するものがないから何も出さない」で済むものではなく、何を出すかを考える対象

## 空になる 3 つの場面

一覧が空になる場面は、だいたい 3 つに分けられます。

| 場面 | 例 | 利用者の状況 |
|---|---|---|
| 初めて使う | 登録直後で Todo も投稿も 1 件もない | 使い方がまだ分からない |
| 検索の結果が 0 件 | 条件に一致する商品がない | 目当ての物を探している途中 |
| 全部片付けた | Todo を全部完了にした・通知を全部読んだ | やることを終えた |

同じ 0 件でも、利用者の状況はまったく違います。

- **初めて使う人**: 何をすればいいか知りたい
- **検索中の人**: 目当ての物にたどり着きたい
- **全部片付けた人**: もうやることがないと確認できれば十分

見せるべきものも場面ごとに変わります。

紛らわしいのが、通信の失敗などでデータを表示できない場面です。

- これは中身が空なのではなく、あるかどうか分からない状態。エンプティステートではなく**エラー**として扱う
- エラーまで「データがありません」で済ませると、利用者は「本当にないのか、取れなかったのか」を区別できない

## 状況の説明と次の一歩

良いエンプティステートに入れるものは、場面によらず共通しています。

- **状況の説明**: なぜ空なのかを伝える。「まだ投稿がありません」「この条件に一致する商品はありません」のように、壊れていないことが分かる言葉にする
- **次の一歩**: 空の状態から抜け出す行動を示す。「最初の投稿を作成する」ボタンや、「検索条件を変えてみてください」という提案

同じ 0 件の画面を並べてみます。

作る手間の差はごくわずかですが、初めて開いた人の体験は大きく変わります。

## 場面に合わせたトーン

説明と次の一歩という骨組みは同じでも、言葉のトーンは場面で変えます。

- **初めて使う**: 前向きに始め方を案内する。「さあ、最初の Todo を追加しましょう」のような誘い方が合う
- **検索 0 件**: 淡々と。探し物が見つからず困っている人に、にぎやかなイラストや軽口は的外れになる。事実と代替案（キーワードを変える・条件を減らす）だけを短く伝える
- **全部片付けた**: 達成をねぎらってよい。「すべて完了しました」は、その画面では良い知らせ

初回向けの明るい文言を検索 0 件の画面に使い回すと、ちぐはぐになります。トーンを場面ごとに指定するのも設計のうちです。

## コンポーネントの例

Todo 一覧が空のときにエンプティステートを出す React コンポーネントです。

```tsx
type Todo = { id: number; title: string };

type Props = {
  todos: Todo[];
  onCreate: () => void;
};

function TodoList({ todos, onCreate }: Props) {
  if (todos.length === 0) {
    return (
      <section aria-label="Todo 一覧">
        <span aria-hidden="true">📝</span>
        <h2>まだ Todo がありません</h2>
        <p>最初の Todo を追加して、今日やることを整理しましょう。</p>
        <button type="button" onClick={onCreate}>
          Todo を追加する
        </button>
      </section>
    );
  }

  return (
    <ul aria-label="Todo 一覧">
      {todos.map((todo) => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  );
}
```

- アイコンは飾りなので `aria-hidden="true"` を付けて、読み上げの対象から外す
- 見出しと本文で状況を説明し、`button` で次の一歩を示す。ボタンの文言は「OK」ではなく「Todo を追加する」のように行動そのものを書く
- 空のときも `aria-label` で領域に名前を付けておくと、支援技術にもどの一覧が空なのか伝わる
- 検索結果のように操作のたびに中身が入れ替わる場所なら、`role="status"` を付けると 0 件になったことが読み上げでも伝わる

## 開発中には見えない画面

エンプティステートが後回しにされるのには、はっきりした理由があります。

- 開発中の動作確認は、テスト用のデータを入れた状態で行う
- だから作っている本人は、空の画面をほとんど見ない

一方、実際の利用者はこの画面に必ず出会います。

- 登録した直後は、何を開いても 0 件
- 検索を使えば、遅かれ早かれ 0 件を引く

> 第一印象を決めるはずの画面ほど、作った人が一度も見ていない画面になりやすい

依頼や仕様の側にも原因があります。

- 「一覧を表示する」とだけ書かれ、空のときにどう見せるかまでは書かれないことがほとんど
- 書かれていない画面は作り込まれず、「データがありません」の 1 行で実装されても仕様どおり

だから、一覧を作るときは最初からエンプティステートを設計に入れます。

## AI への指示

一覧画面を頼むときに、そのまま使える語彙です。

- 「一覧が 0 件のときは、ただの『データなし』ではなく、なぜ空なのかの説明と次にできる行動を示すエンプティステートにして」
- 「初回のゼロ状態・検索 0 件・全部完了で文言を変えて。検索 0 件には検索条件を変える案内を付けて」

できあがった画面を確かめるときも、この 3 場面を順に作って眺めます。

- 登録直後の状態で開く
- ヒットしない言葉で検索する
- 全部完了にする

それぞれの画面が状況を説明し、次の一歩を示していれば合格です。

## まとめ

- 空の画面は放置すると壊れているように見え、次の行動も示せない
- 空になる場面は初回・検索 0 件・全部完了の 3 つで、エラーとは区別する
- 状況の説明と次の一歩が基本形で、トーンは場面に合わせて変える
