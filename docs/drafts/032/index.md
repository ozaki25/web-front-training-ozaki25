# HTTP とステータスコード — fetch が送って、受け取っているもの

## 今日のゴール

- HTTP のリクエストとレスポンスの「形」を知る
- ステータスコードを先頭の数字で読めるようになる
- fetch が 404 でもエラーにならない理由を知る

## 通信の中身は、定型の手紙

`fetch("/api/users")` の 1 行の裏で、ブラウザとサーバーは **HTTP** という約束事でやり取りしています。中身は意外なほど素朴で、**定型フォーマットのテキストの往復**です。

ブラウザが送る**リクエスト**は、おおよそこんな形をしています。

```
POST /api/users HTTP/1.1        ← メソッドと宛先
Content-Type: application/json  ← ヘッダー（メタ情報）
Authorization: Bearer abc123

{"name":"田中"}                  ← ボディ（本文）
```

サーバーが返す**レスポンス**も同じ構造です。

```
HTTP/1.1 201 Created            ← ステータス（結果の番号）
Content-Type: application/json  ← ヘッダー

{"id":42,"name":"田中"}          ← ボディ
```

部品は 3 種類だけです。

| 部品 | 役割 |
|------|------|
| 1 行目 | リクエストなら「何をどこに」、レスポンスなら「結果はどうだったか」 |
| ヘッダー | 本文の形式、認証情報、キャッシュ指示などのメタ情報 |
| ボディ | 本文（JSON など）。無いこともある |

（新しい HTTP/2 や HTTP/3 では通信効率のため形式が変わっていますが、この 3 部品の考え方は同じです）

## メソッド — 「何をしたいか」の動詞

リクエストの 1 行目にある**メソッド**は、操作の意図を表す動詞です。

| メソッド | 意図 | 例 |
|---------|------|---|
| GET | 取得する（何も変えない） | 一覧や詳細の表示 |
| POST | 新しく作る | ユーザー登録、注文の作成 |
| PUT / PATCH | 更新する | プロフィールの変更 |
| DELETE | 削除する | 記事の削除 |

大事なのは「GET は**何も変えない**約束」だということです。ブラウザやキャッシュの仕組みは「GET なら何度送っても安全」という前提で動いています。データを変更する処理を GET で作ると、先読みやリトライで**意図せず複数回実行される**事故につながります。AI が生成した API の設計でも、「変更系が GET になっていないか」は見る価値があります。

## ステータスコード — 先頭の数字で読む

レスポンスの**ステータスコード**は結果を表す 3 桁の番号です。全部覚える必要はありません。**先頭の 1 桁**だけで大意が取れます。

| 先頭 | 意味 | 雰囲気 |
|------|------|--------|
| 2xx | 成功 | 「できました」 |
| 3xx | リダイレクト | 「別の場所を見てください」 |
| 4xx | **あなた（クライアント）側**の問題 | 「依頼がおかしいです」 |
| 5xx | **サーバー側**の問題 | 「こちらの故障です」 |

そのうえで、開発で頻繁に会う顔ぶれだけ覚えれば十分です。

| コード | 意味 | 補足 |
|--------|------|------|
| 200 OK | 成功 | いちばん見る |
| 201 Created | 作成成功 | POST の成功でよく返る |
| 301 / 302 | 引っ越し（恒久 / 一時） | ブラウザが自動で追いかける |
| 400 Bad Request | リクエストの形式が不正 | 送った JSON が壊れている等 |
| 401 Unauthorized | **未認証**（誰だか分からない） | ログインしていない |
| 403 Forbidden | **権限なし**（誰かは分かるがダメ） | ログイン済みでも見られない |
| 404 Not Found | 見つからない | URL の打ち間違い、データの不存在 |
| 500 Internal Server Error | サーバー内部のエラー | サーバーのコードのバグなど |
| 503 Service Unavailable | 一時的に応答不能 | 過負荷・メンテナンス |

特に **401 と 403 の違い**は会話で頻出です。401 は「ログインしてください」、403 は「あなたには権限がありません」。エラー画面の文言もこの区別で変わります。

「4xx ならこちらの送り方を直す、5xx ならサーバー側を調べる」。この切り分けが頭にあるだけで、エラー調査の最初の一歩が速くなります。

## fetch の落とし穴 — 404 はエラーじゃない？

ステータスコードに関して、fetch には知らないと必ず踏む仕様があります。**404 や 500 が返っても、fetch は失敗扱いになりません**。

```ts
try {
  const res = await fetch("/api/users/999"); // 404 が返っても…
  const data = await res.json();             // ここまで普通に進む
} catch (e) {
  // ここに来るのは「通信自体ができなかった」とき
  // （ネットワーク断、DNS 解決失敗など）だけ
}
```

fetch にとっての「失敗」は**通信が成立しなかったこと**だけで、「通信は成立したが結果が 404」は成功（レスポンスを受け取れた）なのです。

ステータスを見たいときは、自分で確認します。

```ts
const res = await fetch("/api/users/999");
if (!res.ok) {
  // res.ok は「ステータスが 2xx かどうか」
  throw new Error(`取得に失敗しました: ${res.status}`);
}
const data = await res.json();
```

AI の書いた fetch コードに `res.ok` のチェックが無ければ、**404 でも気づかず JSON の解析に突き進み、無関係なエラーで落ちる**コードです。「`res.ok` を見てる？」の一言で指摘できます。

## ヘッダー — 顔ぶれだけ知っておく

ヘッダーは深入りせず、よく見る 3 つの顔だけ紹介します。

- `Content-Type`: ボディの形式。`application/json` なら JSON
- `Authorization`: 認証情報。`Bearer トークン` の形をよく見る
- `Cache-Control`: 「この応答をどれくらい使い回してよいか」のキャッシュ指示

開発者ツールの Network タブを開くと、すべてのリクエストでこの 3 部品（1 行目・ヘッダー・ボディ）が実際に見られます。「通信の中身は定型の手紙」と知ってから眺めると、Network タブは一気に読める画面になります。

## まとめ

- HTTP は「1 行目 + ヘッダー + ボディ」のテキストの往復
- メソッドは意図の動詞。GET は「何も変えない」約束
- ステータスは先頭の数字で読む。4xx はこちら、5xx はサーバー側。401 は未認証、403 は権限なし
- fetch は 404 でも失敗にならない。`res.ok` の確認が必須
