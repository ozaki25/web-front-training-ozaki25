# スクリーンリーダーから見た Web — 自分のアプリは「聞く」とどうなるか

## 今日のゴール

- スクリーンリーダーがページを「構造」で読んでいることを知る
- 同じ見た目でも、書き方で読み上げが激変することを知る
- 自分の環境でスクリーンリーダーを起動できるようになる

## 画面を見ずに使うユーザー

**スクリーンリーダー**は、画面の内容を音声で読み上げるソフトウェアです。視覚障害のある人を中心に、Web を「目で見る」代わりに「耳で聞いて」使うための支援技術です。

- macOS / iPhone: **VoiceOver**（標準搭載）
- Windows: **NVDA**（無料）など
- Android: **TalkBack**（標準搭載）

つまりあなたの作った（AI に作らせた）アプリにも、**音で使っているユーザーがいる**可能性が常にあります。そのときアプリがどう聞こえているかは、HTML の書き方で決まります。

## スクリーンリーダーは「見た目」を読んでいない

スクリーンリーダーは画面の**ピクセルを読んでいるのではなく、HTML の構造を読んでいます**。

同じ見た目のボタンを、2 通りの HTML で作ってみます。

```html
<!-- 書き方 A: div を CSS でボタンっぽくした -->
<div class="btn" onclick="save()">保存</div>

<!-- 書き方 B: button 要素 -->
<button onclick="save()">保存</button>
```

見た目は CSS でまったく同じにできます。しかし読み上げはこう変わります。

| | 読み上げ |
|---|---------|
| 書き方 A | 「保存」 |
| 書き方 B | 「**保存、ボタン**」 |

書き方 A は「保存」という**ただのテキスト**として読まれます。押せることが伝わらず、そもそもキーボードでフォーカスを当てることもできません。書き方 B は「ボタン」という**役割**（role）付きで読まれ、Tab キーで到達でき、Enter で押せます。

スクリーンリーダーは各要素について、おおよそ 3 点セットで伝えます。

- **名前**: 「保存」「メールアドレス」
- **役割**: ボタン、リンク、見出し、編集テキスト
- **状態**: 押されている、展開済み、チェック済み

`<button>` や `<a>` のような**意味を持つタグを使うだけで、役割と状態の大半は自動で付きます**。セマンティックな HTML が大事だと言われ続ける理由は、ここで音になって現れます。

## 読み上げユーザーは「飛び石」で移動する

もう 1 つの大事な事実。スクリーンリーダーのユーザーは、ページを上から全部聞いているわけではありません。**構造を使ってジャンプしながら**目的の場所を探します。

- **見出しジャンプ**: 見出し（h1〜h6）だけを順に聞いて、目的のセクションに飛ぶ
- **ランドマークジャンプ**: `<nav>`（ナビゲーション）や `<main>`（本文）といった領域単位で飛ぶ
- **リンク一覧・フォーム部品一覧**: ページ内のリンクだけを一覧で聞く

ここで効いてくるのが、ふだん何気なく書いている構造です。

| 書き方 | 音の世界での意味 |
|--------|----------------|
| 見出しを `<h2>` で書く | 目次に載る。ジャンプで到達できる |
| 太字の `<div>` を見出し代わりにする | 目次に載らない。**そのセクションは実質、見つけられない** |
| `<main>` `<nav>` で領域を分ける | 「本文へジャンプ」が使える |
| 全部 `<div>` | ページ全体がのっぺりした 1 本のテキスト |

見た目には現れない差が、音の世界では「使える / 使えない」の差になります。

## フォームと画像 — 名前が無いとこう聞こえる

**ラベルの無い入力欄**は、こう読まれます。

> 「編集テキスト」

何を入力する欄なのか、一切伝わりません。`<label>` で名前を結びつけると、「メールアドレス、編集テキスト」になります。

```html
<label for="signup-email">メールアドレス</label>
<input id="signup-email" type="email" />
```

**alt の無い画像**は、ファイル名（「IMG_2034.jpg」など）が読まれたり、無言で飛ばされたりします。`alt="集合写真。チームメンバー5人が…"` のような説明があれば、それが画像の「聞こえ方」になります。

**動的な変化**も要注意です。「保存しました」のメッセージが画面の隅に出ても、視線の無い世界では**何も起きていません**。`aria-live="polite"` を付けた領域に出すことで、「画面が変わったら読み上げて」と伝えられます。

## 5 分でできる体験

知識として知るのと、聞くのとでは衝撃が違います。一度自分のアプリを聞いてみてください。

- **macOS**: Cmd + F5 で VoiceOver を起動 / 終了。Tab や矢印キーで移動
- **iPhone**: 設定 → アクセシビリティ → VoiceOver
- **Windows**: NVDA を無料でインストール

そして 2 つだけ試します。

1. **目を閉じて**（または画面を伏せて）、自分のアプリの目的の操作ができるか
2. Tab キーだけで、押したいボタンまでたどり着けるか

「編集テキスト」しか言わない入力欄、無言の画像、たどり着けないボタン。自分のアプリの聞こえ方を一度知ると、コードの見え方が変わります。

## AI のコードを聞く耳で見る

AI は見た目の再現は得意ですが、音の世界はプロンプトに無ければ後回しにしがちです。チェックは 3 つに絞れます。

- 押せるものは `<button>` / `<a>` になっているか（`div` + onClick になっていないか）
- 入力欄に `<label>`（または `aria-label`）が結びついているか
- 操作の結果メッセージは `aria-live` の領域に出ているか

そして指示の言葉も変わります。「アクセシビリティに配慮して」という曖昧な依頼より、「**スクリーンリーダーで役割と名前が読まれるように**」と言えるほうが、AI の出力は確実に良くなります。

## まとめ

- スクリーンリーダーは見た目でなく HTML の構造（名前・役割・状態の 3 点セット）を読む
- ユーザーは見出しとランドマークで「飛び石」移動するので、構造の手抜きは到達不能を生む
- ラベル無し入力欄は「編集テキスト」としか言わず、動的変化は aria-live で伝える
- まず一度、自分のアプリを聞いてみる（macOS は Cmd + F5）
