# Service Worker — ブラウザの中に住むもう 1 つのプログラム

## 今日のゴール

- Service Worker が「ページとサーバーの間に割り込むプロキシ」だと知る
- リクエストの横取りでオフライン対応やキャッシュが成立する仕組みを知る
- PWA の「インストール」の正体を知る

## ページを閉じても残るもの

通常の JavaScript は、ページを開いている間だけ動きます。ページを閉じれば止まり、オフラインなら何もできません。

しかし **Service Worker** は違います。ページとは**別のスレッド**でブラウザの中に常駐し、ページを閉じていても、オフラインでも、一定の仕事ができます。

Service Worker は「ブラウザの中に住むもう 1 つのプログラム」と捉えるのが分かりやすい位置づけです。

## リクエストに割り込む「プロキシ」

Service Worker のいちばん強力な機能は、**ページからのネットワークリクエストに割り込めること**です。

```mermaid
flowchart LR
  A["ページ<br>（fetch で API 呼び出し）"] --> B["Service Worker<br>（ここで横取りできる）"]
  B -- "そのまま通す" --> C["サーバー"]
  B -- "キャッシュから返す" --> A
  style A fill:#dbeafe,color:#1e293b,stroke:#3b82f6
  style B fill:#fef3c7,color:#1e293b,stroke:#f59e0b
  style C fill:#dcfce7,color:#1e293b,stroke:#22c55e
```

ページが `fetch("/api/data")` を呼ぶと、そのリクエストはサーバーに届く前に Service Worker を通ります。Service Worker は以下の選択ができます。

- **そのままサーバーに転送する**（何もしない）
- **キャッシュにある結果を即座に返す**（オフライン対応の要）
- **リクエストを加工してからサーバーに送る**

この「割り込み」のおかげで、**オフラインでもページが表示できる**ようになります。ネットワークが無くても、Service Worker が保存済みの HTML や画像をキャッシュから返せるからです。

## ライフサイクル — 登録してから効くまで

Service Worker はページの JavaScript とは別に管理されます。使い始めるには**登録**が必要で、すぐには効きません。

```mermaid
flowchart LR
  A["登録<br>（初回アクセス時）"] --> B["インストール<br>（ファイルをキャッシュに保存）"]
  B --> C["待機<br>（古い SW がまだ動いていたら待つ）"]
  C --> D["アクティブ<br>（リクエストに割り込める状態）"]
  style A fill:#dbeafe,color:#1e293b,stroke:#3b82f6
  style B fill:#fef3c7,color:#1e293b,stroke:#f59e0b
  style C fill:#f1f5f9,color:#1e293b,stroke:#94a3b8
  style D fill:#dcfce7,color:#1e293b,stroke:#22c55e
```

初回アクセスでは**まだ割り込めません**。Service Worker が効くのは 2 回目以降のアクセスからです。これは「ページを開いてから登録する」という順番の制約です。

## PWA の構成要素

ホーム画面にアプリを追加する「**PWA**（Progressive Web App）」は、Service Worker を軸にした仕組みの集合体です。

- **Service Worker** がオフライン対応とバックグラウンド処理を担当
- **マニフェストファイル**（`manifest.json`）がアプリ名・アイコン・起動設定を宣言
- ブラウザがこの 2 つを検出すると「**インストール**」の選択肢を提示する

「インストール」と言っても、ネイティブアプリのようなインストーラは動きません。実体は「ホーム画面にショートカットを作り、Service Worker を常駐させる」だけです。アプリストアを通さずに、Web ページをアプリのように使えるのが PWA の核心です。

## 現在の位置づけ — 強力だが慎重に

Service Worker は強力ですが、現在の開発ではいくつかの注意点があります。

- **キャッシュの制御が難しい**: 古い Service Worker がキャッシュを握ったまま居座ると、新しいバージョンのアプリが表示されない事態になる。「更新したのに反映されない」の犯人が Service Worker だったというトラブルは定番
- **Next.js との関係**: Next.js は App Router のキャッシュ機構を持っており、Service Worker によるキャッシュと組み合わせると制御が複雑になる。使う場合は `next-pwa` のようなライブラリが仲介する
- **すべてのアプリに必要なわけではない**: オフライン対応やプッシュ通知が不要なアプリに Service Worker を入れても、キャッシュの管理コストだけが増える

「Service Worker でオフライン対応を」と AI が提案してきたら、「**このアプリはオフラインで使う場面があるか？**」が最初に問うべきことです。答えが No なら、入れるメリットは薄い。

## まとめ

- Service Worker はページと別のスレッドで常駐する「もう 1 つのプログラム」
- リクエストに割り込んで、キャッシュからの応答やオフライン対応を実現する
- PWA の「インストール」の実体は、ショートカット + Service Worker の常駐
- 強力だが、キャッシュの制御が難しい。オフラインが必要ないなら慎重に
