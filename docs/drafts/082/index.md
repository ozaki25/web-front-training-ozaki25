# フロントエンドの可観測性 — ユーザーの画面で何が起きたかを知る

## 今日のゴール

- 「手元では再現しない」問題の構造を知る
- エラートラッキングと RUM という 2 つの観測手段を知る
- 観測データが「直す優先順位」を決める材料になると知る

## 「動いてるはずなんですけど」の限界

開発チームに入ると、チャットに「エラー通知」のチャンネルがあり、Sentry や Datadog といった名前のツールから自動の警報が流れています。あれが今日の主題です。

リリースしたアプリについて、こんな報告が届きます。

> 「ユーザーから、ボタンを押しても反応しないと問い合わせが来ています」

手元で試すと、動く。同僚の環境でも、動く。でもユーザーのところでは動かない。**ユーザーの画面は、開発者からは見えない**のです。

サーバーなら、ログを見れば何が起きたか分かります。しかしフロントエンドのコードは**ユーザーのブラウザの中**で動いています。そこで起きたエラーは、誰かが教えてくれない限り、存在すら知ることができません。

この「見えない場所を見えるようにする」仕組みが、フロントエンドの**可観測性**（observability）です。柱は 2 つあります。

## 柱 1: エラートラッキング — ユーザーの画面のエラーを集める

**エラートラッキング**は、ユーザーのブラウザで発生した JavaScript のエラーを**自動で収集して、開発チームに送る**仕組みです。代表的なサービスに Sentry や Datadog があります。

アプリに計測用のコードを仕込んでおくと、ユーザーの画面でエラーが起きた瞬間に、こんな情報が送られてきます。

| 情報 | 分かること |
|------|-----------|
| エラーメッセージとスタックトレース | どのコードの何行目で何が起きたか |
| ブラウザと OS | Safari だけで起きているのか、全環境か |
| 発生回数と影響ユーザー数 | 1 人の環境依存か、全員に起きているか |
| 直前の操作の記録 | エラーに至るまでにユーザーが何をしたか |

「ボタンが反応しない」という曖昧な報告が、「**iOS の Safari でだけ、この関数が undefined を参照して落ちている。影響は 1 日 200 人**」という具体的な事実に変わります。再現に悩む時間が、原因の特定から始められる時間になります。

### リリースとの紐付け

エラートラッキングの実務でいちばん効くのが、**デプロイとエラーの突き合わせ**です。「このバージョンをリリースした直後から、このエラーが急増した」が見えれば、犯人のコミットはほぼ特定できたも同然です。逆に言うと、リリース直後はエラーのグラフを見る時間。これが「リリースしっぱなしにしない」の実体です。

## 柱 2: RUM — 実ユーザーの体感速度を測る

**RUM**（Real User Monitoring、実ユーザー監視）は、エラーではなく**パフォーマンスを実ユーザーの環境で測る**仕組みです。

開発マシンの Lighthouse 計測（ラボデータ）は、高性能なマシンと速い回線での 1 回の結果にすぎません。RUM は、**実際のユーザー全員の環境から** LCP や INP などの指標を集めます。

すると、ラボでは見えなかった事実が出てきます。

- 開発マシンでは LCP 1.2 秒。しかし**ユーザーの中央値は 3.8 秒**（低速回線と古い端末が多数派だった）
- 日本からは速いのに、**海外ユーザーだけ遅い**（CDN の設定漏れ）
- 特定のページだけ INP が悪い（重い処理がクリックをブロックしている）

「速くなった気がする」ではなく、「**ユーザーの 75 パーセンタイルで LCP が 3.8 秒 → 2.4 秒に改善した**」と数字で言える。改善の効果測定が、感想から測定に変わります。

## 観測は「直す優先順位」を決める道具

エラートラッキングと RUM が揃うと、開発の判断が変わります。

| 観測が無いと | 観測があると |
|------------|------------|
| 問い合わせが来たバグから直す（声の大きい順） | **影響ユーザー数の多い順**に直す |
| 「たぶん速くなった」 | 指標の改善幅で効果を確認する |
| リリース後の問題発覚は数日後 | リリース直後のグラフで即日検知 |

限られた時間で何を直すか。その意思決定の材料を供給するのが可観測性の本当の価値です。

## 入れ方の現実

Sentry のようなサービスは、Next.js 用のセットアップが整備されており、導入自体は難しくありません。AI に「Sentry を導入して」と頼べば設定コードは出てきます。

そのときに人間が判断すべきことは、設定よりも運用です。

- **個人情報を送っていないか**: エラー情報にはユーザーの入力値が混ざりうる。マスキングの設定は人間が確認する
- **ノイズの除去**: ブラウザ拡張機能由来のエラーなど「自分たちのコードのせいではないもの」を除外しないと、本物の警報が埋もれる
- **見る習慣**: 集めても誰も見なければ存在しないのと同じ。リリース後とウィークリーで見る、のような運用の取り決めが本体

## まとめ

- フロントエンドのコードはユーザーのブラウザの中。観測しなければ何も見えない
- エラートラッキングは「どこで何が何人に起きたか」を自動収集する
- RUM は実ユーザーの環境でパフォーマンスを測る。ラボデータとの乖離が見える
- 観測の価値は「直す優先順位」を数字で決められること。集めたら見る運用まで含めて可観測性
