# デザインシステムの実装 — 「決まっている」現場で何が嬉しいか

## 今日のゴール

- デザインシステムが「UI の部品と規約の揃った棚」だと知る
- トークン → コンポーネント → ドキュメントの 3 層構造を知る
- 実装者にとっての「嬉しさ」と「辛さ」を知る

## 毎回ゼロから作る苦痛

ボタン 1 つ取っても、サイズ・色・角丸・ホバー時の変化・無効時の見た目・フォーカスリング。決めることが大量にあります。ページごとに毎回これをゼロから作ると、2 つの問題が起きます。

- **バラバラになる**: ページ A のボタンは角丸 4px、ページ B は 8px。同じアプリなのに統一感が無い
- **毎回悩む**: 「この余白は 16px？ 24px？」を毎回ゼロから判断する時間が積み上がる

「色もサイズもボタンの形も、あらかじめ決まっていて、棚から取り出して使うだけ」。この棚が**デザインシステム**です。

## 3 層構造 — トークン・コンポーネント・ドキュメント

デザインシステムは、だいたい 3 つの層で構成されています。

### 層 1: デザイントークン — 値に名前を付ける

色、サイズ、余白、フォントなどの**最小の値に名前を付けたもの**です。

```css
:root {
  --color-primary: #0f6fff;
  --color-danger: #dc2626;
  --spacing-sm: 8px;
  --spacing-md: 16px;
  --radius-md: 8px;
}
```

`#0f6fff` と書く代わりに `var(--color-primary)` と書く。色を変えたいときはトークンの定義を 1 か所変えるだけで、アプリ全体に反映されます。Tailwind CSS の `bg-blue-600` も、実質的にはトークンの一種です。

### 層 2: コンポーネント — トークンで組み立てた部品

トークンを使って組み立てた、再利用可能な UI 部品です。

```tsx
type ButtonProps = {
  children: React.ReactNode;
  variant?: "primary" | "danger";
  disabled?: boolean;
};

export function Button({ children, variant = "primary", disabled }: ButtonProps) {
  return (
    <button
      disabled={disabled}
      className={`rounded-md px-4 py-2 text-white font-medium
        focus-visible:outline-2 focus-visible:outline-offset-2
        ${variant === "primary" ? "bg-blue-600 hover:bg-blue-700 focus-visible:outline-blue-600" : ""}
        ${variant === "danger" ? "bg-red-600 hover:bg-red-700 focus-visible:outline-red-600" : ""}
        ${disabled ? "opacity-50 cursor-not-allowed" : ""}`}
    >
      {children}
    </button>
  );
}
```

このボタンは `variant` で色を切り替えますが、使える色はあらかじめ決まっています。「好きな色を渡せる」のではなく「**決められた選択肢から選ぶ**」のがポイントです。選択肢が絞られているから、ページごとにバラバラにならない。

### 層 3: ドキュメント — 使い方のガイド

コンポーネントをどう使うか、どの場面でどの variant を選ぶか。ガイドラインがないと、棚の部品を間違った場面で使ってしまいます。

- 「primary は 1 画面に 1 つだけ」
- 「削除操作には danger を使う」
- 「テキストリンクとボタンを混ぜない」

大企業のデザインシステム（Google の Material Design、Apple の Human Interface Guidelines）には、この層が特に厚く用意されています。

## 実装者にとっての嬉しさ

### 嬉しさ 1: 判断が減る

「余白は何 px？」「この青は何色？」を毎回考えなくてよくなります。トークンとコンポーネントが答えを持っているので、**選ぶだけ**。判断の回数が激減し、本来の機能実装に集中できます。

### 嬉しさ 2: AI との相性が良い

AI に「デザインシステムの Button コンポーネントを使って」と言えば、色もサイズも角丸もフォーカスリングも**自動で揃います**。デザインシステムが無いプロジェクトで AI にボタンを作らせると、毎回微妙に違うボタンが量産されます。

### 嬉しさ 3: 一括変更ができる

ブランドカラーを変えたい。トークンの `--color-primary` を 1 か所変えれば、すべてのボタン、リンク、バッジの色が一斉に変わります。トークンを使わず色を直書きしていたら、grep で探して 1 つずつ直す作業が待っています。

## 辛さも知っておく

### 辛さ 1: 「微妙に合わない」との戦い

デザイナーの新しい画面案に、デザインシステムには無い微妙なバリエーション（少し大きいボタン、少し違う角丸）が入ってくることがあります。「既存部品で済ませる」か「新しいバリエーションを足す」かの判断が、日常的に発生します。

### 辛さ 2: 作りすぎると重荷になる

50 種類のコンポーネントを作っても、使われるのが 10 種類なら残り 40 はメンテナンスの重荷です。**必要になったものから作る**のが、小さなチームでの現実的な進め方です。

## まとめ

- デザインシステム = トークン（名前付きの値）+ コンポーネント（決められた部品）+ ドキュメント（使い方のガイド）
- 判断が減る、AI と相性が良い、一括変更ができる
- 「微妙に合わない」との判断が日常的に発生する。必要なものから作る
- トークンの 1 か所変更でアプリ全体に反映される仕組みが核
