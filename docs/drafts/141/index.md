# ローディングの見せ方 — スピナーとスケルトンの使い分け

## 今日のゴール

- 読み込み中の見せ方しだいで同じ待ち時間でも体感速度が変わると知る
- スピナーとスケルトンとプログレスバーの向き不向きを知る
- 読み込み中であることを支援技術に伝える書き方を知る

## 待ち時間と体感速度

アプリで一覧画面を開くと、データがサーバーから届くまでの待ち時間が必ず発生します。通信を速くする工夫とは別に、待っている間に何を見せるかでも、利用者が感じる速さは変わります。この感じ方の速さを**体感速度**と呼びます。真っ白な画面で 1 秒待つのと、これから出てくるものの形が見えている画面で 1 秒待つのとでは、実際の時間は同じでも印象が違います。

サーバーの応答を待たずに、成功を仮定して画面を先に変えてしまう楽観的更新という手法もあります。ただしあれが使えるのは、いいねの数字のように結果を先取りできる操作だけです。初回のデータ取得のように、届くまで表示するものが何もない場面では、待つこと自体は避けられません。今日はその避けられない待ち時間に何を見せるか、という話です。

## 読み込み中の見せ方

### 何も表示しない

一番シンプルなのは、何も出さずにデータが届いてから描画することです。ごく短時間で終わる処理なら、これで問題ありません。むしろ 0.2 秒で終わる処理にスピナーを出すと、一瞬ちらついてかえって落ち着きません。

問題は少しでも長引いたときです。画面が変わらない時間が続くと、利用者には操作が受け付けられたのかどうかが分からず、「固まったのでは」という不安が生まれます。読み込み中とフリーズの区別がつかないことが、何も表示しない場合の一番の弱点です。

### スピナー

くるくる回るアイコンです。いま読み込み中であることは確実に伝わり、フリーズとの区別がつきます。実装も単純なので、ローディング表示の定番になっています。

弱点は、あとどれくらい待つのかがまったく伝わらないことです。回り続けるアイコンを眺める時間が長くなるほど、「本当に終わるのか」という不安が募ります。数秒以内に終わる短い待ちに向いた見せ方です。

### プログレスバー

進み具合を横棒の伸びで見せる方法です。残りがどれくらいか見通せるので、長めの待ちでも安心して待てます。ファイルのアップロードのように、全体量と現在位置が数値で分かる処理に向いています。

注意したいのは、多くの API 通信では本当の進捗が取れないことです。進み方が分からないのに、それらしく伸びる棒を見せると、実際の進みと合わない「嘘の進捗」になります。90% あたりでずっと止まっている棒を見たことがあれば、それがこの状態です。

### スケルトンスクリーン

実際のコンテンツが出てくる前に、その形だけを薄い色の四角や線で先に表示しておく手法です。動画サイトや SNS を開いた直後、灰色の四角がちらっと見えてから中身に置き換わる、あの表示です。

中身はまだなくても「もうすぐこの形のものがここに出る」と伝わるので、視線の置き場が定まり、同じ待ち時間でもスピナーより短く感じられやすいとされています。実際のコンテンツと同じ位置・同じ大きさで出しておけば、表示が切り替わる瞬間に画面がガタッとずれるのを防ぐ効果もあります。一覧やカードのように、完成形のレイアウトが決まっている画面と相性のよい見せ方です。

スピナーは「待たされていること」に意識が向き、スケルトンは「これから出てくるもの」に意識が向きます。待ち時間はどちらも同じなのに、後者のほうが画面が進んでいるように見えるのが分かると思います。

## 使い分けの目安

| 場面 | 向いている見せ方 |
|------|-----------------|
| ボタンを押した後の短い処理 | スピナー。ボタンの中に小さく出し「送信中」のラベルを添える |
| 一覧やカードなど完成形のレイアウトが決まっている画面 | スケルトンスクリーン。実物と同じ形・同じ位置で出す |
| アップロードなど進み具合が数値で分かる処理 | プログレスバー。本当の進捗だけを見せる |
| ごく短時間で確実に終わる処理 | 何も出さない。一瞬のちらつきを避ける |

この使い分けは、そのまま AI への指示の語彙になります。「記事一覧のローディングはスケルトンにして。カードと同じレイアウトで 3 件分」「送信ボタンは処理中に無効化して、中にスピナーと送信中のラベルを出して」のように見せ方まで指定すると、画面ごとにローディング表示がばらばらになるのを防げます。

Next.js の App Router なら、ページと同じ階層に `loading.tsx` を置くと、ページの準備ができるまでその中身が自動で表示されます。スケルトンの定番の置き場所なので、「この一覧ページに loading.tsx を追加して、スケルトンを表示して」という指示も通じます。

## やってはいけない待たせ方

### 押しても見た目が変わらないボタン

二重送信を防ぐためにボタンを無効化するのは正しい対応ですが、変化がそれだけだと、利用者には押せたのかどうかが分かりません。反応がないと思ってもう一度押し、無効化が漏れている箇所では同じリクエストが二重に飛びます。無効化とあわせて、スピナーや「送信中」のラベルで、受け付けたことを必ず見せます。

### 実際より速く見せかける演出

本当の進捗が分からないのに 90% まで一気に伸びるプログレスバーや、終わっていない処理を完了したように見せる表示は、その場の印象はよくても、あとで実際の状態との辻褄が合わなくなります。楽観的更新が失敗時の巻き戻しまで設計して成立しているのに対し、ただ速く見せるだけの演出には埋め合わせがありません。見せ方で工夫してよいのは印象までで、状態を偽らないのが線引きです。

## 読み込み中を支援技術に伝える

ここまでの話は、すべて見た目のアニメーションです。スクリーンリーダーの利用者には、スピナーが回っていても灰色の四角が明滅していても、何も伝わりません。読み込み中であることは、見た目とは別にテキストで伝えます。

```tsx
"use client";

import { useEffect, useState } from "react";

type Article = { id: string; title: string; summary: string };

export function ArticleList() {
  const [articles, setArticles] = useState<Article[] | null>(null);

  useEffect(() => {
    fetch("/api/articles")
      .then((res) => res.json())
      .then((data: Article[]) => setArticles(data));
  }, []);

  // 読み込み中はスケルトンを表示する
  if (articles === null) {
    return (
      <div aria-busy="true">
        {/* 意味はテキストで伝える。画面上は CSS で視覚的に隠す（いわゆる sr-only） */}
        <p role="status" className="sr-only">
          記事を読み込んでいます
        </p>
        {/* スケルトンの四角は飾りなので読み上げ対象から外す */}
        <div aria-hidden="true">
          {[1, 2, 3].map((n) => (
            <div key={n} className="skeleton-card">
              <div className="skeleton-line skeleton-title" />
              <div className="skeleton-line" />
            </div>
          ))}
        </div>
      </div>
    );
  }

  return (
    <ul>
      {articles.map((article) => (
        <li key={article.id}>
          <h3>{article.title}</h3>
          <p>{article.summary}</p>
        </li>
      ))}
    </ul>
  );
}
```

役割分担はこうなっています。

- `role="status"` の要素は通知領域として扱われ、中のテキストをスクリーンリーダーが控えめに読み上げる（`aria-live="polite"` を指定したのと同じ扱い）
- スケルトンの四角そのものは意味を持たない飾りなので、`aria-hidden="true"` で読み上げ対象から外す
- `aria-busy="true"` は「この領域は更新中で、内容がまだ揃っていない」という印になる

「読み込んでいます」のテキストは、画面を見ている人にはスケルトンと二重になるので、視覚的には隠しておくのが定番です。AI にローディング表示を頼むときも、「role="status" で読み込み中を伝えて、スケルトン自体は aria-hidden にして」と一言添えると、見た目だけのローディングになるのを避けられます。

## まとめ

- 待ち時間が同じでも、見せ方しだいで体感速度は変わる
- 短い処理はスピナー、形が決まった画面はスケルトン、進捗が数値で分かる処理はプログレスバー
- 読み込み中の意味はアニメーションではなく role="status" のテキストで伝える
