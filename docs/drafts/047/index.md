# 同値分割と境界値 — 試す値を代表と境目に絞る

## 今日のゴール

- テストの価値は「何を試すか」で決まると知る
- 同値分割と境界値分析という古典的な技法を知る
- カバレッジ 100% でも安心できない理由を知る

## 緑のチェックとバグの有無

テストは、書けばたいてい緑（成功）になります。でも「緑になった」ことと「バグが無い」ことは、同じではありません。

```ts
// テスト対象: 送料を計算する関数。3000 円以上は送料無料
export function calcShipping(total: number): number {
  return total >= 3000 ? 0 : 500;
}
```

```ts
// よく書かれるテスト
test("1000 円の送料は 500 円", () => {
  expect(calcShipping(1000)).toBe(500);
});

test("5000 円の送料は無料", () => {
  expect(calcShipping(5000)).toBe(0);
});
```

2 本とも通り、チェックは緑になりますが、このテストには穴があります。

仮に実装が `total >= 3000` ではなく `total > 3000` と間違っていたら（3000 円ちょうどの人から送料を取ってしまうバグ）、**この 2 本はどちらも緑のまま**です。テストの数でも緑の数でもなく、**どの値を試すか**でテストの価値が決まります。その値の選び方を昔から整理してきたのが、同値分割や境界値分析といった技法です。

## 同値分割で代表だけを試す

入力の候補は無限にあります（1 円、2 円、…）。しかし `calcShipping` にとって、1000 円と 1500 円は**同じ扱いになる仲間**です。どちらも「3000 未満 → 500 円」のグループに属します。

このように、**同じ振る舞いになる入力をグループ（同値パーティション）に分け、各グループから代表を 1 つだけ試す**のが**同値分割**です。

| グループ | 代表値 | 期待 |
|---------|--------|------|
| 3000 未満 | 1000 | 500 円 |
| 3000 以上 | 5000 | 0 円 |

冒頭の 2 本は、実は同値分割として正しい選び方です。同じグループから何本も試しても水増しにしかなりません。「テストを 20 本書きました」が誇れる報告とは限らない、という判断もこの技法でできます。

ここまでは**有効な値**のグループでした。実際には、拒否・無視すべき**無効な値**のグループ（無効パーティション）もあります。送料計算なら、マイナスの金額や `NaN` がそれです。網羅の目安は、無効も含めて全グループを 1 回ずつ通すことです。

しかし、同値分割だけでは足りません。

## 境界値分析で境目の両側を試す

先ほどのバグ（`>=` と `>` の取り違え）が緑をすり抜けたことを思い出してください。バグが現れるのは、グループとグループの**境目**でした。

経験則として、**バグは圧倒的に境界に集中します**。「以上と超」「以下と未満」「±1」の間違いは、人間も AI も日常的にやらかすからです。そこで、**境界の両側を狙い撃ちで試す**のが**境界値分析**です。

| 試す値 | 期待 | 仕留められるバグ |
|--------|------|----------------|
| 2999 | 500 円 | 境界が下にズレていないか |
| **3000** | **0 円** | **`>=` を `>` と書き間違えていないか** |
| 3001 | 0 円 | 境界が上にズレていないか |

3000 ちょうどの 1 本があれば、先ほどのバグは赤くなって捕まりました。**仕様書の「〜以上」「〜まで」という言葉を見たら、その数字の前後を試す**。これを意識するだけで、テストで捕まえられるバグが変わってきます。

なお、境界の 3000 と両隣（2999・3001）を試すのが 3 値、境界とすぐ外側（3000・2999）だけなら 2 値です。最低限は 2 値でも押さえられます。

## 定番の「意地悪な値」リスト

境界と並んで、どんな入力にも潜む定番のエッジケースがあります。テストを読むときは、これらが試されているかを見ます。

- **空**: 空文字、空配列、何も選んでいない
- **ゼロとマイナス**: 0 円の注文、-1 個
- **null / undefined**: データがまだ無い
- **極端に長い**: 1 万文字の名前、10 万件の一覧
- **文字の罠**: 全角数字、絵文字、前後の空白

**仕様の境界（3000 円のような業務の数字）は、コードには書かれていません**。だから機械的には拾えず、外しやすいところです。「この関数の仕様で大事な境目はどこか」を知っているのは、コードではなく人間側です。

## カバレッジの数字だけでは足りない

テストの充実度の指標として**カバレッジ**（テスト実行で通過したコードの行の割合）がよく使われます。便利な数字ですが、これだけでは判断できません。

```ts
test("送料が計算できる", () => {
  calcShipping(1000); // 呼んだだけ。結果を確かめていない
});
```

このテストでもカバレッジは上がります。でも**実行された**ことと**正しさを確かめた**ことは別物です。`expect`（期待値の照合）が弱ければ、カバレッジ 100% のコードにもバグは平気で残ります。照合の薄いテストは数を増やしても意味がないので、**数字より expect の中身**を見てください。

## 人と AI の役割分担

テストで人間がやることは、そんなに多くありません。

1. **仕様の境界を伝える**: 「3000 円以上で送料無料。2999 / 3000 / 3001 を試して」と指示する。技法の言葉（境界値）を知っていれば一言で済む
2. **できたテストを技法で確かめる**: 同値パーティションの代表は揃っているか、境界の両側はあるか、意地悪な値はあるか、expect は仕事をしているか

テストコードを自分で量産する時代は終わりつつあります。それでも、**「何を試すべきか」を見極める力**はこれまで以上に大事になります。

## まとめ

- テストの価値は本数や緑の数でなく「どの値を試すか」
- 同値分割で代表に絞り、境界値分析で境目の両側を狙う（バグは境界に集中する）
- 空・ゼロ・null・極端・全角の意地悪リストも定番
- カバレッジは「通った」だけで、確かめているのは expect の中身
