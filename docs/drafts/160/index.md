# HTTP の進化 — なぜ今のサイトは速いのか

## 今日のゴール

- HTTP のバージョン差が「1 本の接続で何本運べるか」だと知る
- 昔の高速化テク（画像の結合・ドメイン分割）が不要になった理由を知る
- HTTP/3 が TCP をやめて速くなった仕組みを知る

## 1 ページに何十個ものリクエスト

今のサイトは、1 ページ表示するのに HTML・CSS・JS・画像など何十個ものファイルを取りに行きます。この「何十個」をどう効率よく運ぶかで、HTTP はバージョンを重ねてきました。

## HTTP/1.1 — 1 本の接続で 1 個ずつ

HTTP/1.1 では、1 つの接続で 1 度に 1 つのやり取りしかできません。前のファイルが返るまで、次が待たされます。

ブラウザは接続を数本開いて並行させますが、本数には上限があります。だから昔は、次のような工夫が使われていました。

- 小さな画像を 1 枚の大きな画像に結合する（スプライト）
- 画像用に別ドメインを用意して接続数を稼ぐ（ドメインシャーディング）

今のコードで見かけると不思議ですが、当時は接続数を絞り出すための定番でした。

## HTTP/2 — 1 本の接続で同時に何本も

HTTP/2 は、1 つの接続の中で複数のやり取りを同時に流せるようにしました。これを多重化と呼びます。

おかげで接続を増やす小細工は要らなくなり、スプライトもドメイン分割も不要になりました。

ただし土台の TCP には弱点が残ります。途中で 1 つのパケットが失われると、後続がまとめて待たされます。

この詰まりをヘッドオブラインブロッキングと呼びます。

## HTTP/3 — TCP をやめる

HTTP/3 は、土台を TCP から、UDP をベースにした **QUIC** に変えました。やり取りごとに独立した通り道を持つので、1 つでパケットが失われても、他は止まりません。

CDN を経由するサイトの多くは、すでに HTTP/3 で配信されています。企業のファイアウォールが UDP を止める環境では、HTTP/2 に自動で戻ります。

## まとめ

- HTTP/1.1 は 1 接続 1 個ずつで、接続数を稼ぐ小細工が要った
- HTTP/2 は多重化で同時並行にし、スプライトやドメイン分割を不要にした
- HTTP/3 は TCP をやめ、パケット損失が他を止めない仕組みにした
