# アクセシビリティ検査の原理 — 機械に分かること、分からないこと

## 今日のゴール

- axe や Lighthouse が何を自動検出できるか知る
- 自動検査で見つかるのは問題の一部だけ、という限界を知る
- 「機械 + 1 分の手動チェック」という現実的な検査の型を持つ

## 「アクセシビリティ、大丈夫？」と聞かれたら

チームでアクセシビリティの話題が出たとき、「ちゃんとやれている」をどう確かめればよいでしょうか。全ページを目視で？ 専門家に依頼？

実は、最初の一歩は無料で数秒です。**自動検査ツール**が、定番の問題を機械的に検出してくれます。

ただし先回りして言うと、AI に「アクセシビリティ対応して」と頼むと「Lighthouse のスコアを 100 にしました」という形の報告が返ってきがちです。**そのスコアが何を意味し、何を意味しないのか**を知らないと、この報告を評価できません。それが今日の本題です。

## 自動検査の道具立て

### Lighthouse — まず全体のスコアを見る

ブラウザの開発者ツールに入っている **Lighthouse** は、ページを分析して Accessibility のスコア（0〜100）とともに、検出した問題を一覧してくれます。

- 画像に alt が無い
- ボタンやリンクに名前（読み上げられるテキスト）が無い
- 文字色と背景色のコントラスト不足
- フォーム部品にラベルが無い

それぞれの指摘には「なぜ問題か」「どう直すか」のドキュメントへのリンクが付いており、**学習教材としても優秀**です。

### axe — 開発フローに組み込む

Lighthouse の a11y 検査の中身は、実は **axe-core** という検査エンジンです。これは単体でも使えて、組み込み先が豊富です。

| 使い方 | 場面 |
|--------|------|
| ブラウザ拡張（axe DevTools） | 開発中に手元のページを検査 |
| ESLint プラグイン（eslint-plugin-jsx-a11y） | **コードを書いた瞬間**に JSX の問題を指摘 |
| テストへの組み込み（jest-axe / vitest-axe） | コンポーネントのテストで自動検査し、CI で常時チェック |

特に ESLint プラグインは強力です。`<img>` に alt が無い、`onClick` だけの `div` がある、といった問題を、**AI がコードを生成したその場で**赤線が教えてくれます。Next.js のデフォルトの ESLint 設定には、この jsx-a11y の主要ルールが最初から含まれています。

## 限界 — 機械に分かるのは「形」だけ

ここからが今日の本題です。自動検査が検出できるのは、アクセシビリティ問題の**一部にすぎない**ことが、複数の調査で知られています（よく引用される目安は 3〜5 割程度）。

なぜか。機械が判定できるのは「**形式的に判定可能なもの**」だけだからです。

| 機械に分かる（形の問題） | 機械に分からない（意味の問題） |
|------------------------|------------------------------|
| alt 属性が**無い** | alt の中身が**画像の説明として適切か**（`alt="画像"` は検査を通る） |
| ボタンに名前が無い | 名前が**機能と一致しているか**（「ここをクリック」ばかりでも通る) |
| コントラスト比の不足 | **読む順序が自然か**、見出しの構造が内容と合っているか |
| ラベルの欠落 | フォームの**エラーの伝え方が分かりやすいか** |
| | **キーボードだけで実際に操作を完遂できるか** |

つまり「自動検査が全部緑 = アクセシブル」ではありません。`alt="image123"` でも、意味不明な見出し構造でも、検査は通ります。**スコア 100 は「形式的な問題が無い」という意味**であって、使えることの証明ではないのです。

## 現実的な型 — 機械 + 1 分の手動

とはいえ、専門家による全数監査を毎回やるのは現実的ではありません。実務で機能するのは、この組み合わせです。

### 機械にやらせること（常時・自動）

1. ESLint（jsx-a11y）を有効にしておく。書いた瞬間に形の問題を潰す
2. CI のテストに axe を組み込み、デグレード（直したのに再発）を防ぐ

### 人間がやること（画面を作ったら 1 分）

1. **Tab キーだけ**で目的の操作ができるか。フォーカスがどこにあるか見えるか
2. **見出しだけ**を拾い読みして、ページの構造が伝わるか
3. 画像の alt を読み上げたと想像して、**情報が欠けないか**

この 3 つは道具なしで 1 分でできて、機械の苦手な「意味の問題」を最低限カバーします。

## AI 時代の使い方

AI に「アクセシビリティを直して」と頼むと、機械的に検出可能な問題（alt 追加、ラベル追加）は上手に直します。一方で、alt の**中身**が「画像」「写真」のような無意味な文字列になることがあります。形は直せても、意味はコンテキストを知る人間の領分だからです。

実践的な分担はこうなります。

- **検出は機械**（Lighthouse / axe / ESLint）に任せる
- **修正の実装は AI** に任せる（「この axe の指摘を直して」と検査結果を渡す）
- **意味の確認は人間**がやる（alt の中身、読み上げ順、操作の完遂）

「ツールを回して終わり」でも「全部人力」でもない、この三者の分担が現在の最適解です。

## まとめ

- Lighthouse / axe / ESLint（jsx-a11y）が形式的な問題を自動検出してくれる
- 機械に分かるのは「形」だけ。スコア 100 はアクセシブルの証明ではない
- 機械の常時検査 + Tab・見出し・alt の 1 分手動チェックが現実的な型
- 検出は機械、実装は AI、意味の確認は人間という分担
