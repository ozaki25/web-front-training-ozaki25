# 命名という設計 — 名前がコードの読みやすさを決める

## 今日のゴール

- 名前がコードの「読みやすさ」に直結する理由を知る
- 良い名前の条件を 3 つ知る
- AI のコードの命名を見る目を持つ

## コードの大部分は、名前を読む時間

コードを書く時間より、読む時間のほうがずっと長い。これは経験則としてよく言われますが、「読む」時間の多くは**名前を読んでいる**時間です。

変数名、関数名、コンポーネント名、ファイル名、props 名。コードの大半は名前でできています。名前が的確なら読むだけで意味が分かり、名前が曖昧なら毎回中身を確認しに行く必要があります。

```tsx
// 名前だけで何をするか分かる
const overdue = tasks.filter((t) => t.deadline < now && !t.done);

// 何も分からない。中身を読まないといけない
const arr2 = list.filter((x) => x.d < n && !x.f);
```

2 つのコードは**まったく同じ処理**ですが、読む体験は別物です。名前は設計の一部であり、**未来の読み手（1 か月後の自分を含む）への説明書**です。

## 良い名前の 3 条件

### 1. 中身が想像できる

名前を読んだだけで、中に何が入っているか（変数）、何をするか（関数）が**想像できる**のが良い名前です。

| 悪い例 | 良い例 | 理由 |
|--------|--------|------|
| `data` | `users` | data は何のデータか不明 |
| `flag` | `isVisible` | flag は何の旗か不明 |
| `handleClick` | `handleAddToCart` | 「何のクリック」か分からないと読み進めないと意味が取れない |
| `tmp` | `formattedDate` | 一時変数でも用途を書く |

特に `data`、`info`、`result`、`item`、`value` は**万能すぎて何も伝えない**名前の代表格です。AI のコードにもよく登場します。

### 2. 粒度が合っている

スコープ（名前が使われる範囲）の広さに応じて、名前の具体性を調整します。

- **ローカルの短い範囲**: `users.map((u) => u.name)` の `u` は許容。読む人の目に留まっている間しか使われない
- **ファイルをまたぐ export**: `getUsers()` より `fetchActiveUsers()` のほうが、呼ぶ側で意味が取れる

「3 行先で使われるだけ」の変数と「20 ファイルから import される」関数では、求められる説明の密度が違います。

### 3. 嘘をつかない

**名前の約束と中身が違う**のが最悪です。

```tsx
// getUser と言いながら、ユーザーを新規作成して返している
async function getUser(email: string) {
  let user = await db.user.findByEmail(email);
  if (!user) {
    user = await db.user.create({ email }); // 副作用がある
  }
  return user;
}
```

`get` は「取得する」の約束なので、呼ぶ側は副作用（データの変更）を想定しません。`getOrCreateUser` なら約束と中身が合います。

嘘の名前はバグの温床です。名前を信じて使った結果、想定外の副作用が起きる。しかも名前は正しそうに見えるので、原因にたどり着くのに時間がかかります。

## React / Next.js の命名の慣習

慣習に従うと、名前を見ただけで「何の種類か」が分かります。

| パターン | 例 | 意味 |
|---------|---|------|
| `is〜` / `has〜` / `should〜` | `isOpen`、`hasError` | 真偽値（boolean） |
| `handle〜` | `handleSubmit` | イベントハンドラ |
| `on〜` | `onSubmit` | props で受け取るコールバック |
| `use〜` | `useAuth` | カスタムフック |
| 大文字始まり | `ProductCard` | コンポーネント |
| `〜Context` | `UserContext` | React Context |

`isOpen` を見れば boolean、`handleSubmit` を見ればイベントハンドラだと、**型注釈を確認しなくても**分かります。

## AI の命名を見るポイント

AI は「動く名前」は付けますが、「読みやすい名前」は怪しいことがあります。

1. **汎用すぎる名前**（`data`、`result`、`info`）を見たら → 「何の data？」と具体化する
2. **略語が過剰**（`usr`、`btn`、`cfg`）を見たら → 検索しにくくなるので、省略しない正式名に
3. **動詞と実際の動作のズレ**を見たら → `get` なのに作成する、`update` なのに削除する、は嘘の名前

AI に「変数名を見直して、何のデータか分かるようにして」と頼むだけで、コードの読みやすさは一段上がります。

## まとめ

- コードの大部分は名前を読む時間。名前の質が読みやすさを決める
- 良い名前は「中身が想像できる」「粒度が合っている」「嘘をつかない」
- React の is〜 / handle〜 / use〜 の慣習を知ると、型を見ずに種類が分かる
- AI の汎用名・略語・動詞のズレを見つけたら、具体化を指示する
