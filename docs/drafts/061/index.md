# next/image と next/font — 重い素材を Next.js が自動で軽くする

## 今日のゴール

- 画像とフォントがパフォーマンスに与える影響の大きさを知る
- next/image の自動最適化（リサイズ・形式変換・遅延読み込み）を知る
- next/font がフォント読み込みの「ガタッ」を消す仕組みを知る

## ページの重さの半分は画像

Web ページの平均的な構成を見ると、**転送量の約半分を画像が占めています**。2,000px 幅の写真を 400px の枠に `<img>` で貼れば、残り 1,600px 分は無駄にダウンロードされた通信です。

素の `<img>` タグは何も最適化しません。そこで Next.js は **`next/image`** という Image コンポーネントを用意し、画像の配信を自動で最適化します。

## next/image — 書くだけで 3 つの最適化

```tsx
import Image from "next/image";

export function ProductCard() {
  return (
    <Image
      src="/products/coffee.jpg"
      alt="コーヒー豆のパッケージ"
      width={400}
      height={300}
    />
  );
}
```

`<img>` の代わりに `<Image>` を使うだけで、3 つの最適化が自動で効きます。

### 1. リサイズ — 必要なサイズだけ配信

指定した `width` / `height` やデバイスの画面幅に合わせて、**サーバー側で画像を縮小して**から配信します。2,000px の元画像を 400px 枠に表示するなら、400px 分のデータだけ送る。通信量が劇的に減ります。

### 2. 形式変換 — WebP / AVIF で軽く

JPEG のまま送る代わりに、ブラウザが対応していれば **WebP や AVIF** といった高圧縮形式に自動変換します。見た目はほぼ同じで、ファイルサイズが数分の一になります。

### 3. 遅延読み込み — 見えるまで取りに行かない

画面外の画像は、**スクロールして見える直前まで読み込みを遅らせます**（lazy loading）。ページの最初に必要なリソースを減らし、初期表示を速くします。

### width / height が必須な理由

`<Image>` に `width` と `height` を指定するのは、「画像の場所を先に確保する」ためです。確保しないと、画像が読み込まれた瞬間にレイアウトがガタッとズレます（CLS の悪化）。AI が `<img>` を使っていたら、「`next/image` にできない？」と聞くのがいちばん手っ取り早い最適化です。

### fill モード — 親要素に合わせる

サイズが固定でない場合（親の幅に合わせたい場合）は、`fill` を使います。

```tsx
<div style={{ position: "relative", width: "100%", aspectRatio: "4/3" }}>
  <Image src="/hero.jpg" alt="メインビジュアル" fill sizes="100vw" />
</div>
```

`fill` を付けると親要素いっぱいに広がります。`sizes` はレスポンシブ対応で「この画像は画面幅の何割を占めるか」をブラウザに伝え、適切なサイズの画像を選ばせるヒントです。

## next/font — フォントの「ガタッ」を消す

カスタムフォント（Google Fonts など）にも問題があります。

1. ページが開くと、まず OS のフォント（フォールバック）で表示される
2. カスタムフォントのダウンロードが完了すると、文字の形や幅が**切り替わる**
3. 見出しの幅が変わってレイアウトがズレる（CLS）。または切り替わるまで文字が非表示（FOIT）

**`next/font`** はこの問題を 2 つの方法で解決します。

```tsx
import { Noto_Sans_JP } from "next/font/google";

const notoSansJP = Noto_Sans_JP({
  subsets: ["latin"],
  weight: ["400", "700"],
  display: "swap",
});

export default function Layout({ children }: { children: React.ReactNode }) {
  return <body className={notoSansJP.className}>{children}</body>;
}
```

### 1. ビルド時にフォントを取得して自前配信

Google Fonts に外部リクエストを飛ばす代わりに、**ビルド時にフォントファイルをダウンロードして、自分のサーバーから配信**します。DNS 解決やサードパーティの遅延が消え、CDN のキャッシュも自分で制御できます。

### 2. フォールバックの自動調整

フォールバックフォント（OS のゴシック体など）の文字幅を、カスタムフォントに近くなるよう**自動で微調整した CSS** を生成します。切り替え時の幅のズレが最小限になり、CLS が抑えられます。

## AI のコードを見るポイント

| 見かけたら確認 | 問い |
|-------------|------|
| `<img>` タグ | `next/image` で書き直せないか |
| `<Image>` に `priority` が付いていない LCP 候補 | ファーストビューのメイン画像には `priority` を付けると遅延読み込みが解除され、LCP が速くなる |
| `<link>` で Google Fonts を読み込んでいる | `next/font` で書き直せば、外部リクエストが消えてフォントのガタつきも抑えられる |
| `fill` モードで `sizes` が無い | ブラウザが最大幅を仮定して巨大な画像を取得してしまう |

## まとめ

- 画像は転送量の約半分で、next/image はリサイズ・形式変換・遅延読み込みを自動で行う
- width / height の指定はレイアウトシフト（CLS）の防止で、LCP 候補には `priority`
- next/font はビルド時にフォントを取得し、切り替え時の幅ズレを自動調整する
- `<img>` や `<link>` の Google Fonts を見たら、Next.js の専用コンポーネントへの置き換えを提案する
