# Web Vitals — 「速い」を数字にする

## 今日のゴール

- パフォーマンスを測る 3 つの指標（LCP / CLS / INP）を知る
- それぞれが「ユーザー体験のどの瞬間」を測っているか知る
- Lighthouse の点数より、個別指標を読めるようになる

## 「遅い気がする」では改善できない

アプリが遅いと感じたとき、「もっと速くして」と AI に頼んでも、闇雲な最適化になりがちです。どこが遅いのかが分からなければ、直す場所も分かりません。

Google が定めた **Web Vitals** は、「速い」を**ユーザー体験の 3 つの瞬間**に分解して数値化した指標です。検索順位にも影響するため、チームの会話で頻繁に出てきます。

## 3 つの指標 — 見える・安定・反応する

| 指標 | 測っている瞬間 | ユーザーの気持ち |
|------|---------------|----------------|
| **LCP** (Largest Contentful Paint) | 画面の**メインコンテンツ**が見えるまでの時間 | 「中身が出た」 |
| **CLS** (Cumulative Layout Shift) | 表示後に要素が**ガタッとズレた**量の合計 | 「押そうとしたら逃げた」 |
| **INP** (Interaction to Next Paint) | 操作してから画面が**反応する**までの時間 | 「押したのに動かない」 |

### LCP — いつ「見えた」か

ページを開いてから、**いちばん大きなコンテンツ**（メイン画像、見出しのテキストブロックなど）が描画されるまでの時間です。

| 判定 | 時間 |
|------|------|
| 良好 | **2.5 秒以内** |
| 要改善 | 2.5〜4 秒 |
| 悪い | 4 秒超 |

LCP が遅い原因の定番は、**巨大な画像の読み込み待ち**と**サーバーからの応答が遅い**の 2 つです。Next.js の `next/image` が画像を自動で最適化するのは、この指標を改善するためです。

### CLS — どれだけ「ガタガタ」したか

ページが表示された後に、要素の位置が**予告なくズレた**量の合計です。単位は独自のスコア（無次元）で、低いほど安定しています。

| 判定 | スコア |
|------|--------|
| 良好 | **0.1 以下** |
| 要改善 | 0.1〜0.25 |
| 悪い | 0.25 超 |

CLS が悪化する定番は 3 つです。

- **画像にサイズ指定がない**: 読み込み完了時に突然スペースが確保され、下の要素が押し下げられる
- **Web フォントの読み込みで文字幅が変わる**: フォールバックフォントから Web フォントに切り替わるとき
- **広告や動的コンテンツが後から挿入される**: 読んでいる途中に画面が下にズレる

「リンクを押そうとしたら画面がズレて、別のリンクを押してしまった」。CLS はまさにこの事故を測っています。

### INP — 押してから「反応する」まで

ユーザーの操作（クリック、タップ、キー入力）から、画面が**次の描画を完了する**までの時間です。ページ滞在中のすべての操作のうち、最も遅かったものが代表値になります。

| 判定 | 時間 |
|------|------|
| 良好 | **200 ミリ秒以内** |
| 要改善 | 200〜500 ミリ秒 |
| 悪い | 500 ミリ秒超 |

INP が悪い典型は、**クリックのハンドラの中で重い処理を同期的に実行している**場合です。イベントループの回で見た「コールスタックを占領すると画面が固まる」がそのまま数値に現れます。

## 測り方 — 道具は揃っている

### 開発中のラボデータ

- **Chrome DevTools の Lighthouse**: ページを分析して 3 指標を含むレポートを出す
- **Chrome DevTools の Performance タブ**: 操作を録画して、どの処理に何ミリ秒かかったかを詳細に見る

### 本番のフィールドデータ

- **Chrome User Experience Report (CrUX)**: Google が実ユーザーのデータを集計して公開している。PageSpeed Insights で確認可能
- **RUM（Real User Monitoring）**: 自分のアプリに計測コードを埋め込み、実ユーザーのデータを収集する

ラボデータ（開発者の環境で 1 回計測）とフィールドデータ（実ユーザーの集計）は**別物**です。開発マシンでは速くても、低スペック端末 + 遅い回線のユーザーでは全然違う数字が出ます。**改善したかどうかの最終判断はフィールドデータで行う**のが定石です。

## Lighthouse の点数にだまされない

Lighthouse は 0〜100 の総合点を出しますが、この点数は**環境や計測タイミングで簡単にブレます**。同じページを 2 回計測して点が 10 点変わることも珍しくありません。

点数よりも、**3 つの個別指標を見て、どれが基準を超えているかを特定する**ほうが改善に繋がります。「Lighthouse のスコアを上げて」より「**LCP が 4 秒超えているから、メイン画像の最適化を優先して**」と言えるのが、指標を知っている人の指示です。

## まとめ

- Web Vitals は「速い」を 3 つの瞬間に分解した指標: LCP（見えた）・CLS（安定）・INP（反応）
- LCP の敵は巨大画像と遅いサーバー、CLS の敵はサイズ未指定の画像とフォント、INP の敵は重い同期処理
- ラボデータとフィールドデータは別物で、最終判断はフィールドで
- 総合点より個別指標で、どれが基準を超えているかを特定して直す
