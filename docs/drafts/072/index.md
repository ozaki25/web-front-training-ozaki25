# ダークパターン — 「解約ボタンが見つからない」は偶然ではない

## 今日のゴール

- ダークパターンが「意図的に不利な選択へ誘導するデザイン」だと知る
- 代表的な手口を 5 つ見分けられるようになる
- 実装者として「踏みとどまる基準」を持つ

## 使いにくさに設計の意図がある

Web サービスを使っていて、こんな経験はないでしょうか。

- 解約ページが見つからない。見つかっても「本当にいいですか」「特典を失います」と何画面も挟まれる
- 無料トライアルの申し込みに、クレジットカード番号を要求される
- 「お得なオプション」のチェックボックスが最初から入っている

これらは「たまたま使いにくい」のではありません。**ユーザーを意図的に、サービス提供者に有利な行動へ誘導するデザイン**です。これを**ダークパターン**（deceptive pattern、欺瞞的パターン）と呼びます。

仕組みとしては、人間の認知の性質（デフォルトを受け入れやすい、焦ると判断力が落ちる、損失や罪悪感を避けたい）を**逆向きに使った**ものです。性質そのものは中立で、どちらに使うかだけが違います。

## 代表的な 5 つの手口

### 1. ゴキブリホイホイ — Roach Motel

入るのは簡単、出るのは極めて難しい。登録は 1 ページ、解約は 5 ページ + 電話 + 理由の入力。

### 2. こっそり追加 — Sneak into Basket

カートに勝手にオプション商品や保険が追加されている。気づかず購入すると余計な出費になる。

### 3. 羞恥の確認 — Confirmshaming

選択肢の文言で罪悪感を与える。「はい、最新情報を受け取ります」の横に「いいえ、私は損をしてもかまいません」。

### 4. あらかじめチェック済み — Preselection

メルマガ購読や有料オプションのチェックボックスが**最初から入っている**。外す手間を省略させて同意を取る。

### 5. 偽りの緊急性 — False Urgency

「残り 2 室！」「あと 3 分でセール終了」。実際には補充されるし、セールは毎日やっている。焦りで判断力を奪う。

これらは「ずる賢いデザイン」に見えますが、各国で規制が進んでおり、EU のデジタルサービス法や米国の FTC ガイドラインでは**違法と判断される事例**も出ています。日本でも景品表示法や特定商取引法による規制の議論が進んでいます。

## なぜ実装者が知るべきか

「デザインの話でしょ」と思うかもしれませんが、**ダークパターンは最終的にコードで実装されます**。チェックボックスを `checked` にするのも、解約ボタンのリンクを目立たなくするのも、コードを書く人の手を経ます。

AI にUIを作らせるときも注意が必要です。「ユーザーに確認を求めて」という指示に対して、AI が confirmshaming 的な文言を生成することがあります。「いいえ、特典を失います」のような選択肢は、AI の学習データにそういうパターンが大量にあるからです。

実装者として持つべきは、**「これは本当にユーザーのためか？」という問い**です。判断の基準はシンプルです。

- **やめたい人が、やめられるか？** 解約・退会が登録と同じくらい簡単か
- **デフォルトは中立か？** 事前チェックで不要なものを押し付けていないか
- **情報は正直か？** 数字（残り在庫、カウントダウン）は実態を反映しているか

## 反転 — 良いデフォルトという武器

ダークパターンの裏返しが、**良いデフォルト**です。同じ「デフォルトで選ばれている」を、ユーザーの利益のために使えます。

- プライバシー設定のデフォルトを「共有しない」にする
- 決済画面のオプションのデフォルトを「追加しない」にする
- アカウント削除のフローを登録フローと同じステップ数にする

**デフォルトには強い力がある**。だからこそ、その力をどちらに向けるかが問われます。

## まとめ

- ダークパターン = ユーザーを不利な選択へ意図的に誘導するデザイン
- 代表例: 解約の難しさ、こっそり追加、羞恥の確認、事前チェック、偽の緊急性
- 最終的にコードで実装され、AI も学習データから再生産する
- 「やめたい人がやめられるか」「デフォルトは中立か」が踏みとどまる基準
