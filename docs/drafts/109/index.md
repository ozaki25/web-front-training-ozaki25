# ホワイトボックステスト — ステートメントとブランチのカバレッジ

## 今日のゴール

- ホワイトボックスがコードの内部構造を根拠にする系統だと知る
- カバレッジにステートメント（行）とブランチ（分岐）の別があると知る
- カバレッジ 100% でも見逃す欠陥があると知る

## カバレッジは何を数えているか

テストの充実度としてよく出てくるのが**カバレッジ**という数字です。「コードのうち、テストで通った部分の割合」を表します。仕様ではなくコードの中身を根拠にテストを見るこの系統を、ホワイトボックス（構造ベース）と呼びます。

ただし「通った部分」の数え方には種類があります。代表が、行を数えるステートメントカバレッジと、分岐を数えるブランチカバレッジです。この 2 つの違いが分かると、「カバレッジ 100%」という数字を鵜呑みにせずに読めます。

カバレッジのために別のテストを用意するわけではありません。「何を試すか」は仕様（外から見た振る舞い）を根拠に決めるのが基本で、カバレッジはそうして書いたテストに後からかける抜けの監査です。測ってみて通っていない分岐が見つかったら、その分岐を根拠にテストを書き足します。

## ステートメントカバレッジ — 行を通したか

コードの実行可能なステートメント（文）のうち、テストで通った割合です。ふつうは 1 行に 1 文なので、「行をどれだけ通したか」と考えてかまいません。100% なら、すべての行が少なくとも 1 回は実行されたことになります。

```ts
// 12 歳未満は入場無料、それ以外は 500 円
function fee(age: number): number {
  let price = 500;
  if (age < 12) {
    price = 0;
  }
  return price;
}
```

```ts
test("8 歳は無料", () => {
  expect(fee(8)).toBe(0);
});
```

この 1 本で、`price = 500`・`if`・`price = 0`・`return` の行がすべて実行されます。つまりステートメントカバレッジは 100% です。数字の上では「全部テストした」ように見えます。

## ブランチカバレッジ — 分岐の両方を通したか

分岐（ブランチ）は、`if` の真と偽、`switch` の各 `case`、ループの継続と終了のような、条件で枝分かれする経路です。ブランチカバレッジは、その枝を通った割合です。

さきほどの `fee` には `age < 12` という分岐があり、枝は「真（12 歳未満）」と「偽（12 歳以上）」の 2 つです。`fee(8)` は真の枝しか通っていないので、ブランチカバレッジは 50% です。

**ステートメントカバレッジが 100% でも、ブランチカバレッジは 100% とは限りません**。`fee(8)` は全部の行を通しましたが、「12 歳以上なら 500 円」という偽の枝は一度も試していません。もし初期値を `let price = 5000` と書き間違えていても、`fee(8)` は 0 を返して合格してしまい、12 歳以上の料金が壊れていることに気づけません。

偽の枝も通すには、もう 1 本足します。

```ts
test("30 歳は 500 円", () => {
  expect(fee(30)).toBe(500);
});
```

これでブランチカバレッジも 100% になります。さきほどの初期値の書き間違いがあれば、このテストが落ちて見つかります。ブランチカバレッジはステートメントカバレッジを含む関係にあり、ブランチ 100% を満たせばステートメント 100% も自動的に満たします（逆は成り立ちません）。

2 つを並べるとこうです。

| カバレッジの種類 | 数えるもの | `fee` で 100% にするには |
|---|---|---|
| ステートメント | 実行された行 | `fee(8)` の 1 本（全行を通る） |
| ブランチ | 通った分岐（真と偽の両方） | `fee(8)` と `fee(30)` の 2 本 |

なお、ステートメントカバレッジは**命令網羅**（C0）、ブランチカバレッジは**分岐網羅**（C1）とも呼ばれます。現場で「C0／C1」と聞いたら、この 2 つのことです。似た言葉にデシジョン（判定）カバレッジがあります。厳密には定義が別ですが、現場ではほぼブランチカバレッジと同じ意味で使われます。

## 強みと数字の限界

ホワイトボックスの強みは、コードを直接見ることです。仕様が曖昧だったり、古かったり、不完全だったりしても、実装から欠陥を見つけられます。

ただし、カバレッジの数字は便利な一方で、100% でも安心はできません。

- **「通った」と「確かめた」は別**: 行や枝を実行しても、結果を照合していなければバグは見つからない。関数を呼ぶだけで戻り値を `expect` で確かめないテストは、カバレッジだけ上がって中身が無い
- **どの値で試すかまでは教えない**: `age < 12` を `age <= 12` と書き間違えたバグは、両方の枝を通しても 8 歳と 30 歳では見つからない。12 歳ちょうどのような境目の値を試して初めて見つかる
- **無いものは測れない**: 仕様にある機能をコードに書き忘れていると、そもそもコードが無いのでカバレッジには現れない
- **仕様の妥当性は別**: カバレッジはコードを通った割合であって、その仕様が正しいかは何も言わない

カバレッジが教えてくれるのは「まだ通っていない所はどこか」だけで、正しさの保証ではありません。

だから数字は目標ではなく指針です。100% を目指して数字を追いかけるのではなく、見つかった抜けを埋めるために使います。

どこまで念入りに監査するかはリスクで決めます。決済やログインのように壊れたときの被害が大きい機能は分岐まで全部通し、そうでない機能は数字を眺めて大きな抜けがないか確かめる程度で十分です。

## ロジックの単体テストはどちらで作るか

ロジックの単体テストをブラックとホワイトのどちらで作るかは、現場でもよく迷います。基本は、**ブラックボックスで作り、カバレッジで確認する**です。

- まず仕様から、同値分割・境界値でケースを作る（ブラックボックス）
- ロジックの分岐は仕様の場合分けと対応することが多く、これで分岐もだいたい通る
- カバレッジを測って、通っていない分岐が残ったら、そこだけ足す

ホワイトボックスは「別の作り方」ではなく、この抜けチェックと穴埋めの役です。だから単体テストは、まずブラックボックスで作れば十分なことがほとんどです。

## AI への指示とテストの確認

- **指示**: 「ブランチカバレッジで、`if` の真と偽の両方を通して」と、数え方を名指しで頼める
- **確認**: 「カバレッジ 100%」という数字を見たら、ステートメントかブランチか、`expect` が結果を照合しているか、仕様にあって未実装の機能はないか、を分けて確かめる

数字の大きさより、「どの枝を、何を確かめながら通したか」を見ます。

## まとめ

- ホワイトボックスはコードの内部構造を根拠にテストを作る系統
- カバレッジには行を数えるステートメントと、分岐を数えるブランチがある
- ステートメント 100% でもブランチは漏れうる（ブランチはステートメントを含む）
- カバレッジは目標ではなく指針で、100% でも照合の甘さ・未実装・仕様の誤りは見逃す
