# Cookie と認証 — なぜサイトは「私」を覚えていられるのか

## 今日のゴール

- HTTP がステートレス（毎回初対面）であることを知る
- Cookie とセッション / JWT がログイン状態を支える仕組みを知る
- HttpOnly・Secure・SameSite という Cookie の鎧の意味を知る

## リロードしても、ログインしたまま

ログインした後は、リロードしても、ブラウザを閉じて開き直しても、サイトはあなたを覚えています。当たり前すぎて不思議に感じませんが、HTTP の仕様を知ると、これは当たり前ではありません。

HTTP は**ステートレス**、つまり**状態を持たない**プロトコルです。リクエストは 1 回ごとに独立していて、サーバーから見ると**全リクエストが初対面**。「さっきログインした人」という記憶は、HTTP のどこにもありません。

では、誰が覚えているのか。

## Cookie — ブラウザが自動で持ち歩く名札

種明かしの主役が **Cookie** です。仕組みは 2 ステップしかありません。

1. サーバーがレスポンスのヘッダーで「**この名札を持っておいて**」と渡す（`Set-Cookie`）
2. ブラウザはそれを保存し、**同じサイトへの以降の全リクエストに自動で添付**する

```
（ログイン成功時のレスポンス）
Set-Cookie: session_id=abc123; HttpOnly; Secure

（以降、ブラウザが勝手に毎回付ける）
Cookie: session_id=abc123
```

サーバーは毎回届く名札を見て「abc123 さんだ」と分かる。**記憶しているのはブラウザで、サーバーは毎回名札で照合しているだけ**。これがログイン状態の正体です。重要なのは「**自動で**添付される」こと。アプリのコードが fetch のたびに付けて回っているわけではありません。

## 名札の中身 — セッション方式と JWT 方式

名札に何を書くかで、2 つの流派があります。

| | セッション方式 | JWT 方式 |
|---|--------------|---------|
| 名札の中身 | ただの引換券番号（`abc123`） | **情報そのもの**（ユーザー ID、権限、有効期限）+ 改ざん検知の署名 |
| 本人情報の置き場所 | **サーバー側**（番号と突き合わせる） | **名札の中**（サーバーは署名を検証するだけ） |
| 即座に無効化 | しやすい（サーバー側の記録を消すだけ） | しにくい（配った名札は期限まで有効） |
| サーバーの負担 | 照合用の保存場所が要る | 保存場所が不要で、複数サーバーに分散しやすい |

**セッション方式**は、クロークの引換券です。番号だけ渡して、中身（誰か・権限）はサーバーの手元に置く。ログアウトや強制無効化が確実にできるのが強みです。

**JWT**（JSON Web Token）は、情報を名札自体に書き込み、**改ざんできないよう署名**したものです。サーバーは保存無しで検証だけすればよく、規模を広げやすい。一方で「配った名札を今すぐ無効にする」のが苦手という、わかりやすいトレードオフがあります。

AI に認証を作らせると JWT が出てくることが多いですが、「即時無効化（強制ログアウト）の要件はあるか」で選択は変わります。仕組みのトレードオフを知っていれば、この問いを立てられます。

## Cookie の鎧 — 3 つの属性

ログイン状態の名札は、盗まれれば**なりすましがそのまま成立**します。だから Cookie には防御の属性を付けるのが鉄則です。

| 属性 | 守ってくれること |
|------|----------------|
| `HttpOnly` | **JavaScript から読めなくする**。XSS でスクリプトを注入されても、`document.cookie` で名札を盗めない |
| `Secure` | **HTTPS でしか送らない**。盗聴される平文通信に名札が乗らない |
| `SameSite` | **他サイト発のリクエストに添付しない**（既定は Lax）。罠サイトからの意図しない送信を緩和する |

特に `HttpOnly` は大事な分岐点です。ログインのトークンを `localStorage`（JavaScript から自由に読み書きできる保存場所）に置く実装を AI が出してくることがありますが、**localStorage は XSS が一発でも成立したら全部読まれます**。HttpOnly Cookie に置いた名札は、同じ状況でも直接は盗めません。「**認証トークンの置き場所は HttpOnly Cookie が基本**」は、覚えておく価値のある相場です。

## Next.js ではどう見えるか

Server Components や Server Actions は**サーバーで動く**ので、届いた Cookie をそのまま読めます。

```tsx
// Server Component でログイン状態を確認する
import { cookies } from "next/headers";

export default async function MyPage() {
  const cookieStore = await cookies();
  const sessionId = cookieStore.get("session_id")?.value;

  if (!sessionId) {
    return <p>ログインしてください</p>;
  }
  // sessionId でユーザーを照合して画面を作る…
}
```

ブラウザが自動で添付 → サーバーで照合、という Cookie の流れが、App Router では `cookies()` という形でコードに現れます。

## AI のコードを見るポイント

1. **トークンの置き場所**: `localStorage.setItem("token", ...)` を見たら、HttpOnly Cookie にできないか問う
2. **Cookie の属性**: `Set-Cookie` に HttpOnly / Secure / SameSite が揃っているか
3. **方式の選定理由**: JWT を提案されたら「即時無効化の要件は？」と聞き返す

## まとめ

- HTTP はステートレス。覚えているのはブラウザ（Cookie の自動添付）
- 名札の流派は 2 つ: 引換券のセッション方式、署名付き情報の JWT。無効化のしやすさが分岐点
- 鎧は HttpOnly・Secure・SameSite。トークンを localStorage に置かない
- サーバー側（SC / Server Actions）では cookies() で名札を照合する
