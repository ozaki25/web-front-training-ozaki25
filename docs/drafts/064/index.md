# ビルドとデプロイ — npm run build は何を作り、push で何が起きるか

## 今日のゴール

- build が「開発者のコードを配信用に変換する」工程だと知る
- Vercel のようなプラットフォームが push をトリガーに自動ビルド・配信することを知る
- 「本番で動かない」の調査の入口を知る

## 書いたコードがそのまま動いているわけではない

開発中は `npm run dev` でアプリが手元で動きますが、世界中のユーザーに届ける本番のアプリは、手元のコードがそのまま動いているわけではありません。

`npm run build` を実行すると、`.next/` というフォルダに**配信用のファイル群**が生成されます。この変換工程が**ビルド**です。

### ビルドで起きていること

| 工程 | 内容 |
|------|------|
| JSX → JavaScript | ブラウザが実行できる形に変換 |
| TypeScript → JavaScript | 型注釈を除去 |
| コード分割 | ページ単位でファイルを分け、必要な分だけ配信 |
| tree shaking | 使われていないコードを除去 |
| 圧縮 | 変数名の短縮、空白の削除でファイルサイズを最小化 |
| Server Components の事前実行 | 静的なページは HTML を事前に生成 |

dev モードではこれらの多くが省略または簡略化されています。だから dev では動くのに build で初めてエラーが出ることがあるのです。

## デプロイ — ビルド結果を公開する

ビルドが「作る」工程なら、**デプロイ**は「届ける」工程です。ビルドで生成された配信用ファイルを、インターネット上のサーバーに配置して、ユーザーがアクセスできる状態にします。

### Vercel の場合 — push が引き金

Next.js プロジェクトでよく使われる **Vercel** の場合、流れはこうなります。

```mermaid

  B --> C["サーバー上で<br>npm run build"]

  D --> E["URL でアクセス可能に"]
```

1. GitHub にコードを push する
2. Vercel が push を自動で検知し、サーバー上で `npm run build` を実行する
3. ビルドが成功したら、成果物を世界中の CDN エッジに配置する
4. URL にアクセスするとユーザーに届く

「push したら自動でデプロイされる」。これは魔法ではなく、**Vercel が GitHub のイベントを監視して、ビルドとデプロイを自動実行している**仕組みです。この自動化の仕掛けを一般に **CI/CD**（Continuous Integration / Continuous Delivery、継続的インテグレーション・デリバリー）と呼びます。

### プレビューデプロイ

Vercel はさらに、**プルリクエストごとに専用の URL を発行**します（プレビューデプロイ）。本番を壊さずに「この変更を入れたらどう見えるか」をチームで確認でき、レビューの質が上がります。

## 「本番で動かない」の 3 つの犯人

手元では問題なく動くのに、本番（デプロイ先）では動かない。このとき疑うべき犯人は、パターンが決まっています。

| 犯人 | 例 |
|------|---|
| **ビルドエラー** | 型エラー、import の解決失敗。デプロイプラットフォームのビルドログを見る |
| **環境変数の不一致** | 手元には `.env` があるが、デプロイ先に設定していない |
| **dev と build の挙動差** | dev ではスキップされるチェックが build で発動。または動的な値（new Date()）がビルド時に固定される |

AI に「デプロイしたら動かない」と相談する前に、**デプロイプラットフォームのビルドログ**を確認するのがいちばんの近道です。エラーメッセージがそのまま書いてあることがほとんどです。

## まとめ

- build は開発コードを配信用に変換する工程で、JSX・TS の変換・分割・圧縮・事前生成をやる
- デプロイは build 結果を公開する工程で、Vercel は push → 自動ビルド → CDN 配置
- dev で動いても build で落ちることがあるので、build を通す習慣が大事
- 本番で動かないときは、ビルドログ・環境変数・dev と build の差を疑う
