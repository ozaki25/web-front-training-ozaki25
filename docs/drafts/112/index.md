# JavaScript の妙な挙動 — その場で予想して当てるクイズ

いつもは読んで学ぶ回ですが、今日は違います。コードの実行結果を**先に予想してから**答えを開く、クイズ形式の回です。

## 今日のゴール

- JavaScript の意味不明に見える挙動の裏に、型変換の仕組みがあると知る
- 「+」「==」「typeof」が値をどう変換して比べているかを知る
- 浮動小数点の癖が、どの言語でも起こりうる不具合の種だと知る

## 遊び方

各問題のコードを見て、実行結果を予想してください。予想したら「答えと理由」を開いて確かめます。全問、実際に Node で実行して確認済みです。

## 出題 1: `[] + []`

```js
[] + []
```

::: details 答えと理由
`""`（空文字）になります。

`+` は両辺を文字列や数値に変換しようとします。配列を文字列に変換すると、要素をカンマでつないだ文字列になり、空の配列なら空文字です。だから `"" + ""` で空文字になります。
:::

## 出題 2: `{} + []`

```js
{} + []
```

::: details 答えと理由
1 行だけでこのまま実行すると `0` になります。

行の先頭に `{` があると、JavaScript はオブジェクトリテラルではなく**空のブロック文**として読みます。残った `+[]` は単項プラスで、配列を数値に変換する演算です。配列は文字列（空文字）を経て数値に変換されるので `0` になります。

ただし `({} + [])` のように式として書くと、今度は `{}` がオブジェクトリテラルとして読まれ、結果は `"[object Object]"` になります。同じ見た目でも、書かれる場所で意味が変わる例です。
:::

## 出題 3: `typeof NaN`

```js
typeof NaN
```

::: details 答えと理由
`"number"` になります。

NaN は「数値ではない」という意味の値ですが、その正体は Number 型に属する特別な値です。だから `typeof` で確かめると `"number"` が返ります。
:::

## 出題 4: `NaN === NaN`

```js
NaN === NaN
```

::: details 答えと理由
`false` になります。

浮動小数点数の規格（IEEE 754）では、NaN は自分自身とも等しくないと決められています。JavaScript の数値はこの規格に従うので、`NaN === NaN` は常に `false` です。値が NaN かどうかを確かめたいときは `Number.isNaN()` を使います。
:::

## 出題 5: `0.1 + 0.2 === 0.3`

```js
0.1 + 0.2 === 0.3
```

::: details 答えと理由
`false` になります。

`0.1` や `0.2` は 2 進数の浮動小数点では正確に表せません。足すと `0.30000000000000004` という誤差入りの値になり、`0.3` と一致しないのです。金額計算でこの誤差が事故を起こすことがあります。
:::

## 出題 6: `null == 0`

```js
null == 0
```

::: details 答えと理由
`false` になります。

`==` の比較には特別扱いのルールがあり、`null` は `undefined` とだけゆるく等しいと決められています。数値の `0` と比べるときは変換されず、素直に `false` になります。「null は 0 みたいなもの」という思い込みが外れる典型例です。
:::

## AI が書いたコードにも同じ規則が効く

AI が書いたコードも、この言語の上で動きます。変換の規則を知っていれば、出てきたコードの結果が予想と違ったときも、動じずに原因を追えます。

## まとめ

- 「+」は文字列化を、「==」は特別な比較規則を経て値を変換する
- NaN は数値の一種で、自分自身とも等しくない
- 浮動小数点の誤差は言語を問わず起こる、計算の前提
