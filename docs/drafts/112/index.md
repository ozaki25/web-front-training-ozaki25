# JavaScript の妙な挙動 — その場で予想して当てるクイズ

いつもは読んで学ぶ回ですが、今日はコードの実行結果を**先に予想してから**答えを開く、クイズ形式の回です。

どの問題も豆知識ではなく、ふだんのコードを支えている規則の話です。規則を知っていれば、実行結果が予想と違ったときも落ち着いて原因を追えます。

## 今日のゴール

- JavaScript の意外な挙動の裏に、構文解析・型変換・比較の仕組みがあると知る
- 見た目が同じでも、書かれる場所や使うメソッドで結果が変わることがあると知る
- 一見ありえない結果も、変換や比較のルールを追えば説明できると知る

## 見た目にだまされる構文

### 出題 1: `{} + []`

```js
{} + []
```

```js
({} + [])
```

::: details 答えと理由
文としてこのまま実行すると `0`、`({} + [])` のように括弧をつけると `"[object Object]"` になります。

行の先頭に `{` があると、JavaScript はオブジェクトリテラルではなく空のブロック文として読みます。同じ見た目でも、書かれる場所で意味が変わるということです。

ブロック文を除いて残った `+[]` は、配列を数値に変換する単項プラスの式です。配列はいったん空文字列に変換され、それが数値に変換されるので `0` になります。

括弧で囲むと式としてしか読めなくなるため、`{}` はオブジェクトリテラルとして読まれ、結果は文字列 `"[object Object]"` になります。なお、ブラウザのコンソールには入力を式として扱うものもあり、その場合は括弧なしでも同じ結果になります。
:::

### 出題 2: `return` と改行

```js
function foo() {
  return
  {
    bar: 1
  }
}
console.log(foo())
```

::: details 答えと理由
`undefined` になります。オブジェクトが返ってくるように見えますが、そうはなりません。

JavaScript には、文の終わりに自動でセミコロンを補う仕組み（ASI）があります。`return` の直後に改行があると、その位置で `return;` が補われます。

そのため次の行の `{ bar: 1 }` は戻り値ではなく、実行されない別の文として扱われます。`return` と値は同じ行に書くのが安全です。
:::

## 型変換と数値

### 出題 3: `"b" + "a" + +"a" + "a"`

```js
"b" + "a" + +"a" + "a"
```

::: details 答えと理由
`"baNaNa"` になります。

`+"a"` は単項プラスで文字列 `"a"` を数値に変換しようとしますが、数値にできないので `NaN` になります。残りはすべて文字列の連結なので、`"b" + "a" + NaN + "a"` という並びが、そのまま `"baNaNa"` という文字列になります。
:::

### 出題 4: `typeof null`

```js
typeof null
```

::: details 答えと理由
`"object"` になります。

`null` は「値が存在しない」ことを表す値ですが、`typeof` で確かめると `"object"` が返ります。初期の JavaScript の実装のバグで、直すと既存のコードが壊れるため今も仕様として残っています。

値が `null` かどうかを確かめたいときは、`typeof` ではなく `=== null` を使います。
:::

### 出題 5: `NaN === NaN`

```js
NaN === NaN
```

::: details 答えと理由
`false` になります。

浮動小数点数の規格（IEEE 754）では、NaN は自分自身とも等しくないと決められています。JavaScript の数値はこの規格に従うので、`NaN === NaN` は常に `false` です。

値が NaN かどうかを確かめたいときは `Number.isNaN()` を使います。
:::

### 出題 6: `0.1 + 0.2 === 0.3`

```js
0.1 + 0.2 === 0.3
```

::: details 答えと理由
`false` になります。

`0.1` や `0.2` は 2 進数の浮動小数点では正確に表せません。足すと `0.30000000000000004` という誤差入りの値になり、`0.3` と一致しません。

金額計算でこの誤差が事故を起こすことがあります。
:::

### 出題 7: `3 > 2 > 1`

```js
3 > 2 > 1
```

::: details 答えと理由
`false` になります。

比較演算子は左から順に 2 つずつ処理されます。まず `3 > 2` が計算されて `true` になります。

次は `true > 1` の比較ですが、真偽値は数値に変換されるので `true` は `1` になり、`1 > 1` は `false` です。3 つの値を一度に数式として比べているわけではありません。
:::

## 等しさの比較とスコープ

### 出題 8: `{} === {}`

```js
console.log({} === {})
```

::: details 答えと理由
`false` になります。

オブジェクトどうしを `===` で比べるとき、JavaScript は中身ではなく**同じメモリ上の実体かどうか**を見ます。`{}` は書くたびに新しいオブジェクトが作られるので、中身が同じでも別の実体として扱われ `false` になります。

React で依存配列にオブジェクトを入れると想定通りに動かないことがあるのは、この性質が原因です。
:::

### 出題 9: `[NaN].indexOf(NaN)` と `[NaN].includes(NaN)`

```js
[NaN].indexOf(NaN)
[NaN].includes(NaN)
```

::: details 答えと理由
上は `-1`、下は `true` になります。

`indexOf` は内部で `===` と同じ比較を使います。`NaN === NaN` が `false` である以上、`indexOf` は NaN を見つけられません。

一方 `includes` は SameValueZero という別の比較方法を使い、これは NaN どうしを等しいとみなします。同じ「探す」メソッドでも、内部で使う比較の仕組みが違うと結果が変わります。
:::

### 出題 10: `Object.is(NaN, NaN)` と `Object.is(0, -0)`

```js
Object.is(NaN, NaN)
Object.is(0, -0)
```

::: details 答えと理由
上は `true`、下は `false` になります。どちらも `===` とは逆の結果です。

`Object.is` は `===` とほぼ同じ比較ですが、NaN どうしを等しいとみなす点と、`0` と `-0`（負のゼロ）を異なるとみなす点だけが違います。React の `useState` などが値の変化を判定するときにも、この `Object.is` と同じ考え方が使われています。
:::

### 出題 11: `var` とループの中の `setTimeout`

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```

::: details 答えと理由
`3` が 3 回出力されます。`0, 1, 2` ではありません。

`var` は関数（またはグローバル）スコープなので、ループの 3 回の反復すべてが**同じ `i`** を共有します。`setTimeout` のコールバックが実際に実行されるのはループが終わったあとなので、その時点の `i` の値である `3` が使われます。

`let` で書けば、反復ごとに新しい `i` が作られるので `0, 1, 2` と出力されます。
:::

### 出題 12: `[] == ![]`

```js
[] == ![]
```

::: details 答えと理由
`true` になります。配列が自分自身の否定と等しいという、直感に反する結果です。

まず `![]` を計算します。空配列は truthy（真として扱われる値）なので、それを否定した `![]` は `false` になります。

次に `[] == false` を比較します。`==` は両辺を数値に変換して比べようとし、`[]` は空文字列を経て `0` に、`false` も `0` になるので、結局 `0 == 0` で `true` になります。
:::

## まとめ

- `return` の直後の改行や `var` のスコープなど、見た目だけでは判断できない挙動がある
- `+` の文字列化や `==` の数値化など、暗黙の変換規則は演算子ごとに決まっている
- 同じ「等しさ」でも `===`・`Object.is`・`includes` の内部比較は微妙に違う
- NaN は数値の一種で、自分自身とも等しくない
