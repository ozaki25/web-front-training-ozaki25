# 計算量 — データが 10 倍になったら処理は何倍になるか

## 今日のゴール

- 件数が増えたときの処理回数の増え方を表す指標が計算量だと知る
- ループの中のループは件数の 2 乗で重くなると知る
- Map や Set に変換してから 1 回のループで処理する型を知る

## 手元では一瞬なのに本番で固まる

商品一覧の画面を作り、手元のテストデータ 10 件で動かすと快適に動きます。ところがコードを 1 行も変えていないのに、本番相当の 1 万件へ差し替えた途端、ページが数秒固まります。

通信が遅いのかと思って開発者ツールの Network タブを見ても、レスポンスはすぐ返ってきています。遅いのは通信ではなく、受け取ったデータを処理している JavaScript のほうです。

今日は、このコードの中で起きる遅さの話です。データの量を増やしただけで固まる現象の背景には、コードの書き方によって**件数が増えたときの処理時間の増え方**がまったく違う、という事情があります。

## 計算量という考え方

データの件数を n と置いて、n が増えたときに処理の回数がどう増えるかを表す指標を**計算量**と呼びます。

- **実際の秒数には注目しない**: 秒数はマシンの性能や処理の中身で変わってしまう
- **増え方の形を見る**: 件数が 10 倍になったら、回数は何倍になるか

まずは素直な例です。注文の合計金額を求める処理を書きます。

```ts
type Order = { id: string; price: number };

function totalPrice(orders: Order[]): number {
  let total = 0;
  for (const order of orders) {
    total += order.price;
  }
  return total;
}
```

一覧を先頭から 1 回なめるだけなので、10 件なら 10 回、100 件なら 100 回。件数が 10 倍になれば回数も 10 倍です。

> **O(n)**（オーダーエヌ、オー・エヌと読む）= 件数に比例して増える形

O 記法には数学的に厳密な定義もありますが、日常の会話では「増え方の形を表すラベル」と捉えておけば十分です。

一覧の処理はたいてい何かしらのループを含むので、O(n) 自体は健全な形です。問題は次の形です。

## ループの中のループ

記事一覧に、既読かどうかのフラグを付ける処理を考えます。手元にあるのは記事の一覧と、既読になった記事 ID の一覧です。

```ts
type Article = { id: string; title: string };

function markAsRead(articles: Article[], readIds: string[]) {
  return articles.map((article) => ({
    ...article,
    // 記事 1 件ごとに、既読 ID の一覧全体を先頭から探す
    isRead: readIds.includes(article.id),
  }));
}
```

このコードは正しく動きます。テストも通るし、10 件のデータなら一瞬です。しかし実態はこうです。

- `includes` は、配列の先頭から 1 件ずつ照合して探すメソッド。つまり**中身はループ**
- 見た目には for が 1 つもネストしていないのに、実質は**ループの中でもう 1 つのループ**が回っている

記事が 1 万件、既読 ID も 1 万件なら、照合の回数は単純計算で最大 1 億回。件数が 10 倍になると、外のループも中のループも 10 倍になるので、全体は 100 倍に膨らみます。

> **O(n²)**（オーダーエヌ2乗）= 件数の 2 乗で増える形

`includes` だけではありません。

- `find`、`indexOf`、`some`、`filter` も中身は先頭からのループ
- これらをループの中で呼べば同じ形になる
- 一覧処理のコードには、この二重ループが**メソッドの顔をして紛れ込む**

## 件数が増えるほど開く差

O(n) と O(n²) の差は、件数が少ないうちはほとんど見えません。数字を並べると増え方の違いがはっきりします。

| 件数 n | O(n) の回数 | O(n²) の回数 |
|-------|-----------|-------------|
| 10 | 10 | 100 |
| 1,000 | 1,000 | 100 万 |
| 10,000 | 1 万 | 1 億 |

- **10 件のとき**: 差は 10 回と 100 回。どちらも一瞬なので、体感では区別がつかない
- **1 万件のとき**: 片方は 1 万回のまま、もう片方は 1 億回。体感できる待ち時間になる
- **画面も止まる**: 処理の間 JavaScript はメインスレッドを占有するので、待っている間は画面の操作も効かない

> 手元のテストデータで速かったことは、本番の件数で速いことの保証にならない

増え方の形が違うコードは、件数が増えて初めて差が表面化します。

## 先に探しやすい形に変換する

O(n²) を避ける定番の型があります。ループの中で毎回探すのをやめて、**先に探しやすい形に変換しておき、ループの中ではそれを 1 回参照するだけ**にする形です。

「含まれるかどうか」を知りたいなら `Set` を使います。

- `Set` は、重複のない値の集まりを持つ組み込みオブジェクト
- `has` は、件数が増えてもほぼ一定の速さで答える

先ほどの既読フラグの処理は、こう書き直せます。

```ts
type Article = { id: string; title: string };

function markAsRead(articles: Article[], readIds: string[]) {
  // 先に探しやすい形に変換する。ここは 1 回だけ
  const readIdSet = new Set(readIds);

  return articles.map((article) => ({
    ...article,
    // ループの中は Set に聞くだけ。一覧全体を探し直さない
    isRead: readIdSet.has(article.id),
  }));
}
```

Set への変換も 1 回のループで済むので、全体は「1 回のループが 2 本」になり、O(n) に収まります。1 万件どうしでも回数は数万回で、1 億回とは桁が 4 つ違います。

含まれるかどうかではなく、ID に対応する**データそのもの**を取り出したいときは `Map` を使います。たとえば注文の一覧に顧客名を付ける処理です。

```ts
type Order = { id: string; customerId: string };
type Customer = { id: string; name: string };

function attachCustomerName(orders: Order[], customers: Customer[]) {
  // 顧客一覧を「ID をキーにした対応表」に変換する
  const customerMap = new Map(customers.map((c) => [c.id, c]));

  return orders.map((order) => ({
    ...order,
    customerName: customerMap.get(order.customerId)?.name ?? "不明",
  }));
}
```

`customers.find((c) => c.id === order.customerId)` をループの中に書けば O(n²) ですが、先に Map に変換すれば `get` の 1 回参照で済みます。

| 知りたいこと | 変換先 | ループの中では |
|---|---|---|
| 含まれるかどうか | `Set` | `has` で聞く |
| ID に対応するデータそのもの | `Map` | `get` で取り出す |

Set も Map も、内部でなぜ速く探せるのかには今日は踏み込みませんが、「件数によらずほぼ一定の速さで引ける対応表」と覚えておけば使う分には困りません。

> やっていることは、二重ループを「**変換**」と「**1 回のループ**」への分解

一覧と一覧を突き合わせる処理を見たら、この分解ができないかを考えます。

## 本番だけ遅いときの調べ方

「本番だけ遅い」「データが増えてから遅くなった」という症状が出たとき、計算量は疑う場所の定番です。

- **レビューやテストで気づけない理由**: 手元のテストデータは件数が少なく、O(n²) のコードでも一瞬で動いてしまう
- **形の見当をつける方法**: 件数を 10 倍にして処理時間の増え方を見る。比例して増えるのか、それ以上の勢いで増えるのか

コードを確かめる観点は「**ループの中で includes や find を呼んでいないか**」です。

- 実装を頼むとき: 「二重ループにならないように、片方を Map に変換してから 1 回のループで処理して」と最初から指定する
- できたコードを見るとき: ループ内の `includes` や `find` を見つけたら、件数の 2 乗で重くなる構造なので、Set や Map への変換を指示する

この語彙があるだけで、本番で固まる一覧画面を実装の段階で止められます。

## まとめ

- 計算量は秒数ではなく件数が増えたときの処理回数の増え方を表す指標
- ループの中の includes や find は隠れた二重ループで、件数の 2 乗で重くなる
- 定番の対策は先に Set や Map に変換してから 1 回のループで参照する分解
