# コロケーション — 「近くに置く」という整理術

## 今日のゴール

- コロケーション（関連するファイルを近くに置く）の考え方を知る
- 「種類で分ける」と「機能で分ける」の違いを知る
- App Router のファイル規約がコロケーションを前提にしていると知る

## ファイルが増えると、探す時間が増える

アプリが育つと、ファイル数が 100 を超え、200 を超えます。そのとき**どこに何があるか**がすぐ分からないと、機能を 1 つ直すのに何分もファイルを探し回ることになります。

ファイルの整理方法は、大きく 2 つの流派があります。

## 流派 1: 種類で分ける

```
components/
  ProductCard.tsx
  CartButton.tsx
  UserMenu.tsx
hooks/
  useProduct.ts
  useCart.ts
  useUser.ts
types/
  product.ts
  cart.ts
  user.ts
```

コンポーネントは `components/`、hooks は `hooks/`、型は `types/`。**ファイルの種類**でフォルダを分ける方法です。直感的で、小さなプロジェクトでは問題ありません。

しかしファイルが増えると壁にぶつかります。「商品カードのバグを直したい」とき、関連するファイルが **3 つの離れた場所**に散っています。components の ProductCard、hooks の useProduct、types の product。**1 つの機能が、複数のフォルダにまたがっている**のです。

## 流派 2: 機能で分ける（コロケーション）

```
product/
  ProductCard.tsx
  useProduct.ts
  product.ts
cart/
  CartButton.tsx
  useCart.ts
  cart.ts
```

関連するファイルを**同じ場所に並べる**。これが**コロケーション**（co-location = 共に置く）です。

「商品のバグを直す」ときは `product/` フォルダだけ開けばいい。コンポーネントも hooks も型も、**同じ機能に属するものは同じ場所**にあるからです。

| | 種類で分ける | 機能で分ける（コロケーション） |
|---|------------|--------------------------|
| 探しやすさ | 種類は分かるが、機能が散る | **1 機能が 1 か所に集まる** |
| 影響範囲 | 変更が複数フォルダにまたがる | **1 フォルダで完結しやすい** |
| 削除 | 「この機能のファイル全部消して」が怖い | **フォルダごと消せば終わり** |

最後の「削除」は地味に大きいメリットです。不要になった機能をフォルダごと消すだけで、関連ファイルが一掃される安心感があります。

## App Router はコロケーション前提

Next.js の App Router は、このコロケーションの考え方を**ルーティングの仕組みに組み込んで**います。

```
app/
  products/
    page.tsx          ← ページ本体
    loading.tsx       ← 読み込み中
    error.tsx         ← エラー表示
    product-card.tsx  ← このページ専用の部品
    actions.ts        ← このページの Server Actions
    product-card.test.tsx  ← テスト
```

ページ本体（page.tsx）、読み込み表示、エラー表示、専用部品、Server Actions、テスト。**その機能に関わるものがすべて同じフォルダ**に入っています。

App Router では、`page.tsx` や `layout.tsx` のような**規約ファイルだけがルーティングの対象**で、それ以外のファイルは URL に影響しません。だから専用の部品やテストをページと同じフォルダに安心して置けます。

## 共有部品はどこに置くか

「この部品、別のページでも使いたい」が出てきたとき、コロケーションの原則は**使われる範囲のいちばん近い共通の親に置く**です。

- `products/` と `cart/` で共有 → `app/` 直下か `app/components/` に
- アプリ全体で共有 → プロジェクトルートの `components/` に

「最初から共有の場所に」ではなく、**必要になったときに引き上げる**のがコロケーションの流儀です。最初は使う場所のすぐ隣に置き、再利用が発生したときだけ上に移す。これなら「使うのは 1 か所だけなのに、共通フォルダにあって見つけにくい」問題が起きません。

## AI のコードに対して

AI に構成を任せると、種類で分ける構成（`components/` / `hooks/` / `types/`）を提案しがちです。小さなプロジェクトなら問題ありませんが、「この部品とこの hooks は一緒に使うのだから同じ場所に」と指示できるのが、コロケーションを知っている人の違いです。

## まとめ

- 種類で分けると機能が散る。コロケーション（機能で分ける）なら 1 か所で完結
- App Router は規約ファイル以外を無視するので、ページの隣に部品やテストを置ける
- 共有は「いちばん近い共通の親」に置き、最初から上に上げない
- AI の「種類分け」提案に、「一緒に使うなら近くに」と言い返せる
