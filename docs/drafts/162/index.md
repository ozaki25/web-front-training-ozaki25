# 圧縮 — なぜ転送は軽くなるのか

## 今日のゴール

- テキストは送る前に圧縮され、受け取って展開されると知る
- gzip と brotli の違いと使い分けを知る
- 圧縮方式が Accept-Encoding と Content-Encoding で決まると知る

## 送る前に縮めてから送る

HTML・CSS・JS のようなテキストは、同じ単語やパターンが何度も出てきます。だから送る前にぎゅっと圧縮でき、受け取ったブラウザが元に戻します。

転送するバイト数が減るので、表示が速くなります。ビルドで `.gz` や `.br` のファイルが出るのは、この圧縮済みの版です。

## 圧縮方式はどう決まるか

どの方式を使うかは、ブラウザとサーバーの相談で決まります。

- ブラウザは「自分はこの方式を展開できる」と伝える（`Accept-Encoding: gzip, br`）
- サーバーは選んだ方式で圧縮し、「これで圧縮した」と返す（`Content-Encoding: br`）

対応していない古いブラウザには、圧縮なしや gzip で返す、といった切り替えができます。

## gzip と brotli

今よく使われるのは 2 つです。

| 方式 | 特徴 | 向いている場面 |
|---|---|---|
| gzip | 昔からある。展開が速く、ほぼ全ブラウザが対応 | その場で作る動的な応答 |
| brotli | 新しい。より小さく縮む（Content-Encoding は `br`） | 事前に圧縮しておく静的ファイル |

brotli は圧縮率が高い一方、強く縮めるほど時間がかかります。だから、ビルド時に一度だけ強く圧縮しておける静的ファイルと相性が良いです。

## まとめ

- テキストは送る前に圧縮し、受け取って展開するので転送が軽くなる
- 方式は Accept-Encoding と Content-Encoding の相談で決まる
- gzip は動的な応答、brotli は事前圧縮した静的ファイルに向く
