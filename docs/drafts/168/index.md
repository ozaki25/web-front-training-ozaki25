# テストの7つの原則 — 全部通っても安心できない理由

## 今日のゴール

- テストは欠陥があることは示せるが、ないことは示せないと知る
- 全数テストは不可能だから、早く・偏りに・作り直しながら使うと知る
- 「どこまでやるか」「バグゼロ」の判断に原則の裏付けがあると知る

## テストが全部通ったとき

作ったアプリのテストがすべて通って、画面に緑のチェックが並んだとします。自分で書いたテストでも、AI に書かせたテストでも同じです。

このとき「バグがないことが確認できた、リリースして大丈夫」と言いたくなります。実はこれ、テストの結果をいちばん誤読しやすい瞬間です。

緑が証明したのは「**試した範囲では、欠陥が見つからなかった**」ことだけです。この線引きを、ソフトウェアテストの世界は昔から「テストの原則」として整理してきました。

ISTQB（ソフトウェアテストの国際的な資格団体、日本語版は JSTQB）のシラバスに載っている 7 つです。テストする人だけでなく、テスト結果を受け取って判断する人にも効くので、順に追っていきます。

## テストに言えること・言えないこと

### テストは欠陥があることは示せるが、欠陥がないことは示せない

テストが赤くなったら、そこに欠陥（バグとも呼ばれる、期待と違う動きの原因になっている誤り）があるのは確実です。ここに曖昧さはありません。

逆にテストが緑でも、言えるのは「この試し方では見つからなかった」までです。試していない入力、試していない操作の順番に欠陥が潜んでいる可能性は、緑がいくつ並んでも消えません。

だから「テストしたんだから、もう障害は起きないよね」という期待には応えられません。応えられるのは「ここをこう試して、問題は見つからなかった」という範囲つきの報告までです。

### 全数テストは不可能

なら、試す範囲を広げ続ければ「ない」に近づけるのでしょうか。どこまで広げられるか、数えてみます。

半角英数字（62 種類）を 8 文字入れるパスワード欄が 1 つあるとします。入力のパターンは 62 の 8 乗、約 218 兆通りです。

1 秒に 1 万パターン試せる自動テストがあっても、全部終わるまで約 700 年かかります。

入力欄はふつう 1 つではなく、操作の順番やデータの状態も掛け算されます。つまり「全部試す」は、がんばれば届く目標ではなく、最初から選択肢にありません。

テストとは、**全部は試せない前提で、どこを試すかを選ぶ活動**です。ここから先の原則は、その限られたテストの使い方の話です。

## 限られたテストの使い方

### 早期テストで時間とコストを節約

最初の使い方は「いつ試すか」です。同じ欠陥でも、見つかるのが遅いほど高くつきます。

- 仕様を考えている段階で誤解に気づく: 会話と書き直しで済む
- 実装が終わってから気づく: 作った部分を壊して作り直す
- リリース後に気づく: 作り直しに加えて、利用者への影響と障害対応が乗る

AI に指示してアプリを作ったとき、最後に触ってみたら思っていたものと違って、丸ごと作り直したことはないでしょうか。あれが「気づくのが遅いほど高くつく」の実体験です。

だからテストは、最後の工程に置くものではありません。動くものがなくても、「何をどう確かめるか」を考えることは最初からできます。

### 欠陥の偏在

次は「どこを試すか」です。欠陥はコードのすべての場所に同じ確率で潜んでいるわけではなく、少数の場所に集中します。

- 条件が複雑に絡む箇所（割引と会員ランクと在庫の組み合わせ、など）
- 境目（「以上」と「超」の取り違え、0 件・上限ちょうどの扱い）
- 最近変更したばかりの箇所

不具合の記録を並べると、修正がいつも同じ画面・同じファイルに集中している、という現象はよく起きます。偏りが分かれば、限られたテストをどこに厚くかけるかが決まります。

### テストの弱化

よくできたテスト一式を毎回通していれば安心、ともいきません。同じテストを繰り返すほど、新しい欠陥は見つからなくなります。

そのテストで見つかる欠陥は最初のうちに修正され尽くすので、残るのは「そのテストでは見つからない欠陥」だけになるからです。以前のシラバスではこれを、同じ殺虫剤を使い続けると効かない虫だけが残ることになぞらえて「殺虫剤のパラドックス」と呼んでいました。

既存のテストに価値がないわけではありません。変更で壊れていないことの見張り（リグレッションテスト）としては働き続けます。

ただ、CI が何か月も緑のままという状態は「新しい欠陥を探すテストが足されていない」印でもあります。機能を足したら、それを狙うテストも足す必要があります。

## 十分さと成功の判定

### テストはコンテキスト次第

では、どこまでやれば十分なのでしょうか。この問いに、どの製品にも当てはまる正解はありません。

社内の勤怠ツールで画面の表示が少し崩れるのと、EC サイトで決済が二重に実行されるのとでは、起きたときの痛手がまるで違います。前者に決済並みのテストをかけるのは過剰で、後者に勤怠ツール並みで済ませるのは無謀です。

同じ製品の中でも、崩れてもすぐ直せる画面と、お金や個人情報が通る経路では、かけるべき濃さが変わります。「どこまでやるか」は製品と箇所の性質から毎回決めるもの、というのがこの原則です。

### 「欠陥ゼロ」の落とし穴

最後に、テストを十分にやって欠陥の報告がゼロになったとします。それでも成功とは限らない、というのが最後の原則です。

テストが確かめているのは、おもに「仕様のとおりに動くか」です。仕様そのものが利用者の求めるものとずれていたら、仕様どおり完璧に動くアプリが、誰の役にも立たないまま完成します。

欠陥ゼロの報告と、良い製品であることは、別の話です。

冒頭の「全部緑だから大丈夫」に戻ると、この判断には 2 つの飛躍が重なっていたことになります。緑は欠陥の不在を証明しない（原則の 1 つ目）うえに、欠陥がなくても成功とは限らない（この原則）わけです。

## 7 つの一覧

追ってきた流れを一覧に整理します。

| 原則 | 一言で |
|---|---|
| 1. テストは欠陥があることは示せるが、欠陥がないことは示せない | 緑は「試した範囲で見つからなかった」の意味 |
| 2. 全数テストは不可能 | 全部は試せないから、どこを試すか選ぶ |
| 3. 早期テストで時間とコストを節約 | 欠陥は見つかるのが遅いほど高くつく |
| 4. 欠陥の偏在 | 欠陥は複雑な箇所・境目・変更箇所に集中する |
| 5. テストの弱化 | 同じテストの繰り返しでは新しい欠陥が見つからない |
| 6. テストはコンテキスト次第 | どこまでやるかは製品と箇所の性質で決まる |
| 7. 「欠陥ゼロ」の落とし穴 | 欠陥がないことと役に立つことは別 |

名称は JSTQB シラバス v4.0 のものです。5 つ目は以前「殺虫剤のパラドックス」と呼ばれていたので、古い記事ではそちらの名前で出てきます。

## まとめ

- テストの緑は「試した範囲で欠陥が見つからなかった」ことの証明でしかない
- 全部は試せないから、早く・偏りのある場所に・テストを足しながら使う
- どこまでやるかは文脈次第で、欠陥ゼロは成功と同義ではない
