# カスケードと詳細度 — CSS が「効かない」とき何が起きているか

## 今日のゴール

- カスケード（複数のスタイルが重なる仕組み）を知る
- 詳細度が「どのセレクタが勝つか」の数値だと知る
- `!important` が最終手段であり、乱用すると詰む理由を知る

## 書いたのに効かない

CSS で最も頻繁に体験するフラストレーションは「**書いたのに効かない**」です。

```css
.card { color: blue; }
```

これで青くなるはず。なのに画面では黒いまま。開発者ツールで見ると、自分のスタイルに取り消し線が引かれている。**誰かに上書きされている**。

CSS は「1 つのプロパティに、1 つのルール」ではありません。複数のルールが同じ要素の同じプロパティに指示を出すことが日常的にあり、**どれが勝つか**を決める仕組みがあります。それが**カスケード**です。

## カスケード — 勝ち残りのルール

**カスケード**（cascade = 段々の滝）は、複数の指示が重なったとき**どれを採用するか**を決める仕組みです。判定は順番に行われます。

| 優先度 | 判定基準 | 勝つ側 |
|--------|---------|--------|
| 1（最強） | `!important` が付いているか | 付いている方 |
| 2 | **詳細度**（セレクタの強さ） | 高いほう |
| 3 | 書かれた**順番** | 後に書かれたほう |

つまり「効かない」の犯人は、この 3 段のどこかで自分より強いルールが勝っている、ということです。

## 詳細度 — セレクタの「戦闘力」

**詳細度**（specificity）は、セレクタの「強さ」を数値化したものです。3 つのカテゴリの点数で比較します。

| カテゴリ | 例 | 点数 |
|---------|---|------|
| ID | `#header` | (1, 0, 0) |
| クラス / 属性 / 擬似クラス | `.card`、`[type="text"]`、`:hover` | (0, 1, 0) |
| タグ / 擬似要素 | `div`、`p`、`::before` | (0, 0, 1) |

セレクタが複雑になるほど、点数は合算されて高くなります。

```css
/* (0, 1, 0) — クラス 1 つ */
.card { color: blue; }

/* (0, 2, 0) — クラス 2 つ */
.sidebar .card { color: red; }

/* (1, 0, 0) — ID 1 つ。クラスには常に勝つ */
#main-card { color: green; }
```

`.card` が青にしたくても、`.sidebar .card` のほうが詳細度が高いので赤が勝つ。さらに `#main-card` は ID なので、クラスをいくつ重ねても勝てません。

<div class="c76-demo">

  <table class="c76-table">

    <tbody>

開発者ツールでこのテキストを検証すると、青と緑のルールに**取り消し線**が引かれているのが見られます。

### inline style は最強クラス

HTML に直接書く `style` 属性は、すべてのセレクタより強い特別な位置にいます。

```html
<div class="card" style="color: orange;">これはオレンジ</div>
```

ID セレクタにすら勝ちます。Tailwind CSS の各クラスが最終的に inline style と同等の強さで効くのは、ユーティリティクラスが 1 つのプロパティに対して 1 つのクラスで完結する設計だからです。

## `!important` — 核兵器

```css
.card { color: blue !important; }
```

`!important` はカスケードの最優先に割り込みます。何にでも勝つ、最終手段です。

しかしこれを多用すると、**`!important` 同士の戦い**が始まり、今度は `!important` の中で詳細度の勝負になります。さらにそれに勝つために `!important` を重ねる……というエスカレーションで、最終的に CSS が誰にも修正できない状態になります。

`!important` は「外部のライブラリのスタイルをどうしても上書きしたい」のようなやむを得ない場面だけに留めるのが定石です。

## 「効かない」のデバッグ手順

CSS が効かないとき、開発者ツールでこの順番に確認します。

1. **そもそもルールは当たっているか**: Elements パネルで要素を選び、Styles タブに自分のルールが出ているか。出ていなければセレクタの綴り間違い
2. **取り消し線が引かれているか**: 引かれていれば、もっと強いルールに負けている。勝っているルールがすぐ上に表示される
3. **Computed タブ**: 最終的に適用されている値と、それを決めたルールが確認できる

「効かない」で AI に聞く前に、開発者ツールで「**誰に負けているか**」を 10 秒で確認する。これができると、「`.sidebar .card` に負けているからクラスの詳細度を上げて」のような的確な指示が出せます。

## Tailwind が詳細度の問題をほぼ消す理由

Tailwind CSS がここまで普及した理由の 1 つは、**詳細度の戦いが構造的に起きにくい**ことです。すべてが単一クラスのユーティリティで、ID セレクタもネストしたクラスも使わない。つまり詳細度がほぼ均一で、勝敗は「後に書かれたほうが勝つ」という単純なルールに落ち着きます。

しかしライブラリのスタイルや既存の CSS と共存する場面では、依然として詳細度の知識が要ります。「なぜ Tailwind のクラスがライブラリのスタイルに負けるのか」は、今日の話そのものです。

## まとめ

- カスケードは !important → 詳細度 → 順番の 3 段で「どの指示が勝つか」を決める
- 詳細度は ID > クラス > タグの点数で、セレクタが複雑なほど高い
- `!important` は最終手段で、多用はエスカレーションの入口
- 「効かない」は開発者ツールで「誰に負けているか」を確認する
