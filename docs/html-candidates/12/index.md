# Day 12: AI が出した HTML をレビューする観点

## 今日のゴール

- AI が生成した HTML をチェックするための具体的な観点を知る
- 良い HTML と悪い HTML の違いを見分けられるようになる
- 完璧に書けなくても「判断基準を持つ」ことの価値を知る

## なぜ「レビューする力」が必要なのか

AI にコードを書いてもらう時代において、「自分でゼロから完璧に書ける」ことの重要性は下がっています。しかし「出てきたコードが良いのか悪いのか判断できる」ことの重要性はむしろ上がっています。

AI は一見動くコードを高速に出力しますが、アクセシビリティやセマンティクスの面で問題を含むことがよくあります。Day 1 から Day 11 で学んだ知識を使って、AI が出した HTML をレビューする観点を整理しましょう。

## 観点 1: div だらけになっていないか

`<div>` は「意味を持たない汎用コンテナ」です。CSSのスタイリングやレイアウトのために使うタグであり、**意味的には透明**です。

```html
<!-- ❌ div だらけ — ページの構造がわからない -->
<div class="header">
  <div class="logo">MySite</div>
  <div class="nav">
    <div class="nav-item"><a href="/">ホーム</a></div>
    <div class="nav-item"><a href="/about">会社概要</a></div>
  </div>
</div>
<div class="main">
  <div class="title">お知らせ</div>
  <div class="text">新機能をリリースしました。</div>
</div>
<div class="footer">
  <div class="copyright">&copy; 2026 MySite</div>
</div>
```

```html
<!-- ✅ セマンティックタグを使う — ページの構造が明確 -->
<header>
  <a href="/" class="logo">MySite</a>
  <nav aria-label="メインナビゲーション">
    <ul>
      <li><a href="/">ホーム</a></li>
      <li><a href="/about">会社概要</a></li>
    </ul>
  </nav>
</header>
<main>
  <h1>お知らせ</h1>
  <p>新機能をリリースしました。</p>
</main>
<footer>
  <p>&copy; 2026 MySite</p>
</footer>
```

セマンティックタグ（`<header>`、`<nav>`、`<main>`、`<footer>` など）を使うと、スクリーンリーダーが「ナビゲーション」「メインコンテンツ」「フッター」と認識してくれます。`<div>` だけだと、ページのどこにいるのかが伝わりません。

**チェックポイント**: `<div>` しか見当たらないコードが出てきたら、`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<footer>` に置き換えられないか確認する。

## 観点 2: 見出し階層が飛んでいないか

HTML の見出し（`<h1>` 〜 `<h6>`）は**階層構造**を表します。本の章・節・項のように、数字の順番で入れ子の関係を示します。

```html
<!-- ❌ h2 を飛ばして h3 を使っている -->
<h1>サイトタイトル</h1>
<h3>最新ニュース</h3>
<p>新機能をリリースしました。</p>
<h3>お知らせ</h3>
<p>メンテナンスのお知らせ</p>
```

```html
<!-- ✅ 階層が順番通り -->
<h1>サイトタイトル</h1>
<h2>最新ニュース</h2>
<p>新機能をリリースしました。</p>
<h2>お知らせ</h2>
<p>メンテナンスのお知らせ</p>
```

AI は「見た目のサイズ」で見出しレベルを選ぶことがあります。「`<h3>` のほうが文字が小さくてちょうどいい」という理由で `<h2>` を飛ばすのは間違いです。見た目は CSS で変えられます。見出しレベルはあくまで**意味的な階層**で決めます。

スクリーンリーダーは見出しの一覧を表示する機能を持っており、ユーザーはこれを使ってページ内を素早く移動します。階層が飛んでいると、このナビゲーションが混乱します。

**チェックポイント**: h1 → h2 → h3 の順番が守られているか。h1 から突然 h3 に飛んでいないか。

## 観点 3: a と button を正しく使い分けているか

`<a>` と `<button>` は見た目が似ていることがありますが、目的が異なります。

| 要素 | 目的 | キーボード操作 |
|------|------|---------------|
| `<a>` | **別の場所に移動**する（ページ遷移、ページ内リンク） | Enter キーで動作 |
| `<button>` | **アクションを実行**する（送信、開閉、削除など） | Enter キー / Space キーで動作 |

```html
<!-- ❌ ボタンの見た目だけど a タグ -->
<a href="#" class="btn" onclick="deleteItem()">削除</a>

<!-- ✅ アクションなので button -->
<button type="button" onclick="deleteItem()">削除</button>

<!-- ❌ リンクの見た目だけど button -->
<button onclick="location.href='/about'">会社概要</button>

<!-- ✅ ページ遷移なので a タグ -->
<a href="/about">会社概要</a>
```

`<a href="#">` にクリックイベントを付けてボタンのように使うパターンは、AI がよく出力するミスの1つです。スクリーンリーダーは `<a>` を「リンク」、`<button>` を「ボタン」として読み上げるので、ユーザーの期待する動作と食い違います。

**チェックポイント**: クリックしてどこかに移動するなら `<a>`、アクションを実行するなら `<button>` になっているか。

## 観点 4: label と input が紐付いているか

Day 3 で学んだ内容ですが、AI が生成するフォームでは紐付けが欠落していることがあります。

```html
<!-- ❌ label と input が紐付いていない -->
<label>メールアドレス</label>
<input type="email" name="email" />

<!-- ✅ for と id で紐付ける -->
<label for="email">メールアドレス</label>
<input type="email" id="email" name="email" />

<!-- ✅ label で input を囲む方法でもOK -->
<label>
  メールアドレス
  <input type="email" name="email" />
</label>
```

紐付けがないと、ラベルをクリックしても入力欄にフォーカスが移りません。スクリーンリーダーは「テキスト入力」としか読み上げず、何を入力するフィールドかわかりません。

**チェックポイント**: すべての `<input>`、`<select>`、`<textarea>` に `<label>` が紐付いているか。`for` 属性と `id` が一致しているか。

## 観点 5: img に適切な alt があるか

Day 2 で学んだ `alt` 属性は、AI が生成するコードで特に注意が必要です。

```html
<!-- ❌ alt が空でも意味がある画像 -->
<img src="team-photo.jpg" alt="" />

<!-- ❌ alt がファイル名そのまま -->
<img src="hero-banner.jpg" alt="hero-banner" />

<!-- ❌ 「の画像」は冗長（スクリーンリーダーは画像であることを伝える） -->
<img src="logo.png" alt="ロゴの画像" />

<!-- ✅ 画像の内容を具体的に説明 -->
<img src="team-photo.jpg" alt="チームメンバー5人が会議室で笑顔で立っている写真" />

<!-- ✅ 装飾的な画像は alt を空にする -->
<img src="decorative-line.png" alt="" />
```

AI は `alt="image"` や `alt="photo"` のような意味のない alt を出力したり、装飾的な画像にも説明を付けてしまうことがあります。

**判断基準**: その画像が消えたとき、情報が欠落するなら具体的な `alt` を書く。装飾的で情報を持たないなら `alt=""` にする。

## 観点 6: p の中に div がネストされていないか

HTML にはタグの入れ子ルールがあります。代表的な間違いが「`<p>` の中に `<div>` を入れる」です。

```html
<!-- ❌ p の中に div は入れられない -->
<p>
  お知らせ
  <div class="badge">NEW</div>
</p>

<!-- ✅ 別の構造にする -->
<p>
  お知らせ
  <span class="badge">NEW</span>
</p>
```

`<p>`（段落）は**フレージングコンテンツ**（テキストやインライン要素）しか含められません。`<div>` はフレージングコンテンツではないので、ブラウザは `<p>` を強制的に閉じてから `<div>` を配置します。結果として、意図しない DOM 構造になります。

DevTools の Elements パネル（Day 9 で学んだ、DOM ツリーの可視化）で確認すると、HTML ソースとは異なる構造になっていることがわかります。

よくある間違いパターン:

| 親要素 | 入れられないもの |
|--------|-----------------|
| `<p>` | `<div>`、`<section>`、`<ul>`、`<table>` など |
| `<a>` | 別の `<a>`（リンクの入れ子） |
| `<button>` | `<a>`、別の `<button>` |

**チェックポイント**: `<p>` の中にブロック要素が入っていないか。インライン要素（`<span>`、`<strong>`、`<em>` など）を使うべきところか確認する。

## まずは 1 つの観点から始める

6 つの観点を一度にチェックするのは大変です。最初は**1 つだけ**意識してみてください。

おすすめの始め方:

1. **まず「div だらけか」を見る** — 最もわかりやすく、効果も大きい
2. **慣れてきたら「label の紐付け」を見る** — フォームがあるページで確認
3. **さらに慣れたら「a と button の使い分け」を見る**

完璧に書ける必要はありません。「これ、div じゃなくて nav じゃない？」と気づけるだけで、AI に修正を指示できます。**判断基準を持っていること**が大事なのです。

## まとめ

- AI が生成する HTML は一見動くが、セマンティクスやアクセシビリティに問題を含むことが多い
- 6 つのチェック観点: (1) div だらけでないか (2) 見出し階層が飛んでいないか (3) a と button の使い分け (4) label と input の紐付け (5) img の alt (6) p の中に div がないか
- 完璧に書けなくても「良い・悪い」の判断基準を持つことで、AI に的確な修正指示ができる
- まずは 1 つの観点から意識して、少しずつチェックの習慣をつけていく
