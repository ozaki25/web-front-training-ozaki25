# 画像のマークアップ — アクセシビリティと Core Web Vitals

## 今日のゴール

- 画像のマークアップは**アクセシビリティ**と**Core Web Vitals**の両方に直結すると知る
- `alt` 属性は「何を伝えるか考えるもの」だと知る
- Core Web Vitals（CWV）というユーザー体験の指標があることを知る
- `width` / `height` / `loading` / `srcset` / `picture` の役割を知る

## 画像のマークアップは 2 つの観点に直結する

Web ページで画像はよく使われますが、**書き方によって体験が大きく変わる**要素でもあります。特に次の 2 つの観点に大きく影響します。

- **アクセシビリティ**: 画像を見られない人（スクリーンリーダー利用者、画像が読み込めない環境）にも情報を届けられるか
- **Core Web Vitals**: ページの表示速度やレイアウトの安定性。Google の検索順位にも影響する

どちらも `<img>` タグの属性をどう書くかで決まります。ここから順に見ていきましょう。

## 観点1: アクセシビリティ — alt 属性

`alt` 属性は、画像が読み込めないときや、スクリーンリーダー（画面読み上げソフト）利用者に画像の内容を伝える文字列です。

```html
<img src="logo.png" alt="My Blog" />
```

`alt` は "alternative text"（代替テキスト）の略で、**画像の代わりとなるテキスト**です。画像が見えない環境（スクリーンリーダー、画像の読み込み失敗など）で、代わりに何を伝えるべきかを書きます。

判断に迷ったら、W3C が公開している **[alt Decision Tree](https://www.w3.org/WAI/tutorials/images/decision-tree/)** を参考にするのがおすすめです。画像の用途をフローチャートでたどっていくと、書くべき内容が決まる形になっています。

### 用途ごとの書き方

- **意味のある画像** （ロゴ、グラフ、写真など）: 何が伝われば十分かを書く

  ```html
  <!-- 会社のロゴ(リンク) -->
  <a href="/"><img src="logo.png" alt="My Blog" /></a>

  <!-- グラフ -->
  <img src="sales.png" alt="2026年の売上は前年比120%、四半期ごとに右肩上がり" />
  ```

- **装飾目的の画像** （区切り線、飾りのアイコンなど）: `alt=""` で「読み上げ不要」を明示

  ```html
  <img src="decoration.svg" alt="" />
  ```

- **ボタン内のアイコン画像**: ボタンの機能を書く

  ```html
  <button>
    <img src="trash.svg" alt="削除" />
  </button>
  ```

### alt を省略すると何が起こるか

`alt` 属性そのものを省略すると、スクリーンリーダーは**ファイル名を読み上げる**ことがあります。「trash.svg」「image_001.jpg」のような意味のない音声になり、ユーザー体験が悪化します。

装飾目的でも、**必ず `alt` 属性は書く**（空なら `alt=""`）ことを覚えておきましょう。

### AI に alt を生成させるとき

ChatGPT や Claude、Gemini などの AI に画像を見せて `alt` を書いてもらうこともできます。ただし、AI の出力をそのまま使うのは危険です。

- AI は**ページの文脈を知らない**: 同じ富士山の写真でも、旅行記事なら「雪をかぶった富士山」、装飾目的なら `alt=""`、といった判断は人間にしかできません
- AI は**画像を物理的に記述しがち**: グラフを渡すと「棒グラフの画像」と書かれがちですが、本当に必要なのは「売上が前年比120%」という伝えたい内容です
- AI は**機械的に書く**: 装飾画像にも長文の説明が付くことがあります

AI の出力は**たたき台として使い、「この画像で何を伝えたいか」を自分で判断して調整する**のが正しい使い方です。

## 観点2: Core Web Vitals — 表示速度とレイアウトの安定性

**Core Web Vitals**（コア ウェブ バイタルズ、以下 CWV）は、Google が定めた **ユーザー体験を数値で測るための指標**です。Google 検索の順位にも影響するため、Web サイトの品質として重視されています。

CWV には 3 つの指標があります。

| 指標 | 意味 | 何が測られるか |
|------|------|------|
| **LCP**（Largest Contentful Paint） | 最大コンテンツ描画 | ページで一番大きな要素（画像や見出しなど）が表示されるまでの時間 |
| **CLS**（Cumulative Layout Shift） | 累積レイアウトシフト | ページ読み込み中に要素がどれだけ飛ぶか |
| **INP**（Interaction to Next Paint） | 応答性 | ユーザーの操作に対する画面の反応の速さ |

遅い（LCP）、動く（CLS）、反応しない（INP）—どれもユーザーがストレスを感じる要因です。

### 画像が CWV に大きく影響する

CWV の中でも、**LCP と CLS は画像のマークアップに直結**します。

- **LCP**: ページの最大コンテンツは画像であることが多い。画像のサイズが大きい、読み込みが遅いと LCP が悪化
- **CLS**: 画像のサイズを指定していないと、読み込まれたタイミングで下の文字やボタンが押し下げられて、ユーザーが読んでいる内容が飛んでいく

つまり、**画像の属性を正しく書くだけで CWV が改善できる**ところがあります。ここからは関連する属性を見ていきます。

### width と height — レイアウトシフトを防ぐ

画像に `width` と `height` を指定していないと、**読み込み中の画像の場所はゼロ**になります。画像が読み込まれた瞬間に場所が生まれ、下の要素が押し下げられます。

これが **CLS（レイアウトシフト）** です。読んでいた記事の文字が動いたり、押そうとしたボタンが下にずれたりする、あの不快な現象です。

```html
<!-- ❌ サイズ未指定: 読み込み中は幅0・高さ0になる -->
<img src="photo.jpg" alt="風景写真" />

<!-- ✅ サイズ指定: 読み込み前から場所が確保される -->
<img src="photo.jpg" alt="風景写真" width="800" height="600" />
```

`width` と `height` は、画像の実サイズ（または縦横比が保たれる値）を指定します。CSS で実際の表示幅を変えても、縦横比が保たれて自動的に高さが調整されます。

```css
/* 実サイズは 800x600 でも、画面幅に合わせて表示 */
img {
  max-width: 100%;
  height: auto;
}
```

**サイズ属性を書くだけで CLS を防げる**ので、特別な理由がない限り書く習慣を付けましょう。

### loading — 画面外の画像は遅延読み込み

ページを開いた瞬間、ブラウザはすべての画像を読み込もうとします。しかし、スクロールしないと見えない画像まで先に読み込むのは無駄です。

`loading="lazy"` を指定すると、**画像が画面に近づいたときに初めて読み込まれる**（遅延読み込み、Lazy loading）ようになります。

```html
<!-- 画面外の画像は遅延読み込み -->
<img src="photo.jpg" alt="..." loading="lazy" width="800" height="600" />
```

初期表示が軽くなり、LCP の改善にもつながります。

#### lazy を付けてはいけない場所

**ファーストビュー（スクロールせずに見える範囲）の画像には `loading="lazy"` を付けない**のが鉄則です。

ファーストビューの画像こそページの最大コンテンツ（LCP の対象）であることが多く、遅延読み込みすると LCP が悪化します。

```html
<!-- ❌ ファーストビューのヒーロー画像に lazy はダメ -->
<img src="hero.jpg" alt="..." loading="lazy" width="1200" height="600" />

<!-- ✅ ファーストビューは eager（または指定なし）で即座に読み込む -->
<img src="hero.jpg" alt="..." loading="eager" width="1200" height="600" />
```

「画面に入らないと見えない画像」と「すぐ見える画像」で使い分けるのがポイントです。

### srcset と picture — デバイスに合った画像を配信する

スマホの画面にデスクトップ用の大きな画像を配信すると、データ通信量が無駄になり、読み込みも遅くなります。また、同じ画像でも次世代フォーマット（WebP、AVIF）のほうがファイルサイズが小さくなります。

これらを解決するのが `srcset` と `<picture>` です。

**srcset — 同じ画像の複数サイズを用意する**

```html
<img
  src="photo-800.jpg"
  srcset="
    photo-400.jpg 400w,
    photo-800.jpg 800w,
    photo-1600.jpg 1600w
  "
  sizes="(max-width: 600px) 100vw, 800px"
  alt="風景写真"
  width="800"
  height="600"
/>
```

ブラウザは画面サイズや解像度に合わせて、最適なサイズの画像を自動で選びます。モバイルでは 400w、デスクトップでは 800w、Retina なら 1600w、という具合に。

**picture — フォーマットを出し分ける**

```html
<picture>
  <source srcset="photo.avif" type="image/avif" />
  <source srcset="photo.webp" type="image/webp" />
  <img src="photo.jpg" alt="風景写真" width="800" height="600" />
</picture>
```

ブラウザは上から順に対応フォーマットをチェックし、対応している最初の `<source>` を使います。AVIF 対応のブラウザは AVIF、非対応なら WebP、それもダメなら JPEG へフォールバックします。

最新フォーマットを使ってファイルサイズを小さくしつつ、古いブラウザへの互換性も保てます。

## まとめ

- 画像のマークアップは**アクセシビリティ**と**Core Web Vitals**の両方に直結する
- **アクセシビリティ側**:
  - `alt` は「何を伝えるか考えて」書く。装飾画像は `alt=""`
  - 判断に迷ったら [W3C の alt Decision Tree](https://www.w3.org/WAI/tutorials/images/decision-tree/) を参考に
  - AI 生成の alt はたたき台として使う
- **Core Web Vitals 側**:
  - Google がユーザー体験を数値化した 3 指標（LCP / CLS / INP）
  - `width` / `height`: 書くだけで CLS を防げる
  - `loading="lazy"`: 画面外の画像を遅延読み込み。ファーストビューには付けない
  - `srcset` / `<picture>`: デバイスに合ったサイズとフォーマットを配信
