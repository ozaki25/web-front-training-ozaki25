## 候補（1レッスン規模）

### コードがどこで動くか

| テーマ | レッスンの中身 |
|-------|-------------|
| Server / Client の住み分け | 同じ React なのに `useState` が使えるファイルと使えないファイルがある。`'use client'` の意味。サーバーで動くコードにAPIキーを書いていい理由。「window is not defined」の正体 |
| 環境変数と秘密情報 | `.env` にある `NEXT_PUBLIC_` 付きの変数はブラウザに送られる。付いていないものはサーバー専用。1文字の違いで秘密が漏れる仕組み |

### データのやり取り

| テーマ | レッスンの中身 |
|-------|-------------|
| API を叩くとは何か | `fetch('/api/users')` と書いたとき、ブラウザとサーバーの間で何が飛んでいるか。GET / POST / ステータスコード（200/404/500）。Network タブで実物を見る |
| gRPC という選択肢 | REST だけが通信じゃない。Proto ファイルから型が自動生成される世界。なぜこのチームは gRPC を選んだか |
| フォームの裏側 | ユーザーが入力した値をどう検証するか。クライアントだけでは不十分な理由。Server Actions でサーバー側バリデーションが自然にできる |
| エラーが起きたとき何をするか | データ取得が失敗したら画面に何を出すか。エラーコード体系。「予期しないエラーが発生しました」で終わらせない設計 |
| 状態はどこに置くか | `useState` で足りなくなる瞬間。URL に状態を持つ（検索条件やページ番号）発想。グローバル状態の使いどころ |
| なぜ古いデータが表示されるか | ブラウザキャッシュ / CDN / サーバーキャッシュの多層構造。「スーパーリロードで直る」の裏にある仕組み |

### 攻撃されたら何が起きるか

| テーマ | レッスンの中身 |
|-------|-------------|
| ログインの裏側 | ブラウザを閉じても次に開くとログイン済みなのはなぜか。Cookie とトークン。「誰でもなりすませる」怖さ |
| この人はこの画面を見ていいか | 認証（誰か）と認可（何ができるか）の違い。ロールベースアクセス制御。管理者と一般ユーザーで見える画面が変わる仕組み |
| XSS と CSRF | ユーザーの入力をそのまま画面に表示すると何が起きるか。攻撃の仕組みを知ると、対策が「なぜそう書くのか」分かる |
| セキュリティヘッダー | ブラウザに「この動作は禁止」と伝えるHTTPヘッダー。設定1つで防げる攻撃がある |

### テスト

| テーマ | レッスンの中身 |
|-------|-------------|
| なぜテストを書くのか | 「動いてるからOK」の怖さ。コードを直したら別の場所が壊れる。テストがあれば壊れた瞬間に気づける |
| 何をどこまでテストするか | 単体 / 結合 / E2E の違い。ボタンのクリック1つをテストするか、画面全体の流れをテストするか。コストと効果のバランス |
| 見た目が壊れたら自動で気づく | CSS を直したら別の画面が崩れた経験。VRT（ビジュアルリグレッションテスト）で差分を画像比較する |
| 外部 API がなくても開発する | バックエンドがまだできていないのにフロントを作る方法。モックとは何か |

### 技術をどう選ぶか

| テーマ | レッスンの中身 |
|-------|-------------|
| ライブラリの選び方 | npm のダウンロード数だけで選ぶと痛い目に遭う。ライセンス、保守状況、最終コミット日、型対応。チェックリストを持つ |
| なぜラップするのか | 外部ライブラリを直接使うか、薄い層を挟むか。差し替えが起きたときの影響範囲。全部ラップは過剰、全部直は危険 |
| UI ライブラリの世界 | ヘッドレス（見た目なし、a11y だけ提供）vs フル UI（見た目込み）。React Aria / Radix / shadcn/ui はなぜ生まれたか |

### チームで開発する

| テーマ | レッスンの中身 |
|-------|-------------|
| push したら何が起きるか | CI/CD。コードを push → テスト自動実行 → ビルド → デプロイ。手動でサーバーにアップロードしていた時代との違い |
| ブランチの使い方 | main / feature / release。なぜ直接 main に push しないのか。PR とレビューの意味 |
| リンターとフォーマッター | ESLint と Prettier。なぜコードの書き方を機械で強制するのか。チームで揃えないと diff が読めない |
| ADR（設計判断の記録） | なぜこの技術を選んだのか、を後から追える仕組み。新人が「なぜ？」を聞かずに自分で調べられる |
| ログを残す理由 | 本番で「おかしい」と言われたとき、何が起きたかを追うための記録。サーバーログとクライアントログの違い |
| 監視とは何か | 本番が今どうなっているかをリアルタイムで見る。エラー率、レスポンス時間。壊れてから気づくのでは遅い |

### 見た目の設計

| テーマ | レッスンの中身 |
|-------|-------------|
| デザイントークン | 色やサイズをコードに直書きしない理由。変数で管理すると、テーマ変更が1か所で済む |
| このCSS使っていいの？ | Baseline。ブラウザ対応状況をどう判断するか。「Safari で動かない」を防ぐ基準 |
| ローディングの見せ方 | Skeleton / Spinner / 何も出さない。ユーザーを待たせるとき何を見せるかで体感速度が変わる |

### AI と開発

| テーマ | レッスンの中身 |
|-------|-------------|
| AI にルールを伝える | CLAUDE.md やルールファイルの意味。AI に「このプロジェクトではこう書いて」と伝える仕組み |
