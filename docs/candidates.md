# コンテンツ候補

Day 19 まで公開済み。各候補は15分で読み切れるサイズ。

## 未公開の下書き

| テーマ | レッスンの中身 |
|-------|-------------|
| 関数とコールバック | 関数を値として渡す発想。addEventListener や map が動く仕組み。コールバック地獄から Promise への伏線 |

| テーマ | レッスンの中身 |
|-------|-------------|

## データのやり取り

| テーマ | レッスンの中身 |
|-------|-------------|
| HTTP の基本 | `fetch` したとき何が飛んでいるか。GET / POST / ステータスコード（200/404/500） |
| REST API を叩く | エンドポイント、リクエストボディ、レスポンスの JSON。Network タブで実物を見る |
| gRPC という選択肢 | REST じゃない通信方式。Proto ファイルから型が自動生成される。なぜ選ばれるか |
| フォームのバリデーション | クライアントだけでは不十分な理由。「送信ボタンを押せない」だけでは防げない |
| Server Actions とフォーム | サーバー側でバリデーションが自然にできる仕組み。フォームとの統合 |
| エラーが起きたら何を見せるか | 「予期しないエラー」で終わらせない。エラーコード、ユーザーへの伝え方 |
| useState で足りなくなる瞬間 | コンポーネントをまたいで共有したい値。グローバル状態の使いどころ |
| URL に状態を持つ | 検索条件やページ番号を URL に入れる。戻るボタンで元に戻る、共有できる |
| なぜ古いデータが表示されるか | ブラウザ / CDN / サーバーのキャッシュ多層構造。「スーパーリロードで直る」の裏側 |

## 攻撃されたら何が起きるか

| テーマ | レッスンの中身 |
|-------|-------------|
| なぜログインが保たれるか | ブラウザを閉じても次はログイン済み。Cookie とトークンの仕組み |
| 認証と認可の違い | 「誰か」と「何ができるか」は別の問題。ロールベースアクセス制御 |
| XSS — 画面に埋め込まれるスクリプト | ユーザーの入力をそのまま表示すると何が起きるか。攻撃の仕組み |
| CSRF — 知らない間に操作される | ログイン中の人に意図しないリクエストを送らせる攻撃の仕組み |

## テスト

| テーマ | レッスンの中身 |
|-------|-------------|
| なぜテストを書くのか | 「動いてるからOK」の怖さ。直したら別の場所が壊れる。テストで壊れた瞬間に気づける |
| テストの種類と使い分け | 単体 / 結合 / E2E の違い。何をどこまで。コストと効果のバランス |
| VRT — 見た目の壊れを自動検知 | CSS を直したら別の画面が崩れた。画像の差分比較で気づく仕組み |
| モック — バックエンドなしで開発する | 外部 API がまだないのにフロントを作る方法。何を偽装するか |

## 技術をどう選ぶか

| テーマ | レッスンの中身 |
|-------|-------------|
| ライブラリの選び方 | ダウンロード数だけで選ぶと危険。ライセンス、保守状況、型対応。チェックリスト |
| ラップするか直接使うか | 外部ライブラリに薄い層を挟むか。差し替え時の影響範囲。全ラップは過剰、全直は危険 |
| UI ライブラリの世界 | ヘッドレス（a11y だけ）vs フル UI（見た目込み）。React Aria / Radix / shadcn/ui の違い |

## チームで開発する

| テーマ | レッスンの中身 |
|-------|-------------|
| push したら何が起きるか | CI/CD。テスト → ビルド → デプロイが自動で走る。手動アップロードとの違い |
| ブランチの使い方 | main に直接 push しない理由。feature ブランチと PR の意味 |
| リンターとフォーマッター | ESLint と Prettier。書き方を機械で強制する理由。diff が読めなくなる問題 |
| ADR — 設計判断の記録 | 「なぜこの技術を選んだか」を後から追える仕組み。新人が自分で調べられる |
| ログを残す理由 | 本番で「おかしい」と言われたとき、何が起きたかを追う記録 |
| 監視とは何か | 本番が今どうなっているかをリアルタイムで見る。壊れてから気づくのでは遅い |

# 新しい鉱脈（4視点ブレストより）

シニアFE・トラブル対応・AI協働・設計判断の4視点で発想。複数視点で重複したものは確度が高い。

## 言語・計算の足元の罠（AIも地味に間違える）

| テーマ | レッスンの中身 | 確度 |
|-------|-------------|------|
| 0.1 + 0.2 ≠ 0.3 | なぜ計算がズレるか（2進数浮動小数点）。金額計算で1円ズレる事故。整数で持つ定石 | 視点1,2 |
| 日付とタイムゾーン | `new Date()` がどこの時刻か。サーバー(UTC)とブラウザ(JST)のズレ。1日ずれる現象 | 視点1,2 |
| 文字コードと文字化け | UTF-8 とは。`"𩸽".length` がズレる（サロゲートペア）。文字数制限で踏む | 視点1,2 |
| 「ない」の種類 | null / undefined / 空文字 / 0 / 空配列。`if(count)` で 0 が消える。`??` と `\|\|` の違い | 視点1,2 |
| JSON とシリアライズ | メモリ上のオブジェクトと送れる文字列の違い。Date や関数が消える理由。Server→Client で渡せる値の制限 | 視点1,2 |
| 文字列ソートの罠 | `[1,2,10].sort()` が `[1,10,2]` になる理由。日本語の五十音順が効かない | 視点2 |

## 実務で必ずハマる現象（ハマるまで存在を知らない）

| テーマ | レッスンの中身 | 確度 |
|-------|-------------|------|
| 競合状態（race condition） | 検索を速く打つと前の結果が後から上書きされる。投げた順と返る順が違う。AbortController | 視点2 |
| 本番だけ動かない | 開発と本番のビルドの違い。大文字小文字、環境変数の埋め込み、ミニファイ | 視点1,2 |
| 保存したのに変わらない | ブラウザ/ビルド/Service Worker のキャッシュ多層。ハードリロードで直る理由 | 視点1,2 |
| z-index が効かない | `z-index:9999` でも前に出ない。スタッキングコンテキストという見えない仕組み | 視点1 |
| 100vh がスマホで見切れる | アドレスバー込みで計算される問題。dvh/svh/lvh の登場 | 視点2 |
| メモリリーク | タイマーやリスナーが消えずに残る。なぜタブを開きっぱなしで重くなるか。useEffect の return | 視点2 |
| イベントの伝わり方 | クリックが DOM をどう伝播するか（バブリング）。モーダル外クリックで閉じる仕組み | 視点1 |

## AI と働く（自分からは絶対気づかない）

| テーマ | レッスンの中身 | 確度 |
|-------|-------------|------|
| AI の知識には賞味期限がある | 学習データのカットオフ。だから非推奨 API や古い書き方を出す。最新は自分で確認 | 視点3 |
| なぜ存在しない API を自信満々に作るか | ハルシネーション。AI は「正しい」でなく「それっぽく続く」を生成。もっともらしさ ≠ 正しさ | 視点3 |
| AI が「さっき言ったこと」を忘れる | トークンと文脈長。長い会話で古い指示が押し出される。要点の再提示 | 視点3 |
| AI は「あなたに同意したい」 | 迎合バイアス。「良いですか?」と聞くと「良いです」と返る。「3つ問題点を挙げて」と聞く | 視点3 |
| AI は最短で動く道を選ぶ | 「テスト通して」でテストを骨抜きにする。言葉通りに最適化する。指示の穴 | 視点3 |
| 動くコードと読めるコード | AI は動くコードを量産するが保守性は別。理解せずマージは時限爆弾 | 視点3,4 |
| エラーは AI への手土産 | 要約せず全文を貼る。スタックトレースの読み方。何を渡せば的確に直るか | 視点2,3 |

## 設計・判断（会議で置いていかれない）

| テーマ | レッスンの中身 | 確度 |
|-------|-------------|------|
| なぜ密結合は悪いのか | 1つ直すと10箇所壊れる正体。疎結合 = 相手の中身を知らなくても使える。影響範囲が読める | 視点4 |
| 技術的負債はなぜ返せないか | 妥協したコードは利子付きの借金。返済時間は誰も確保しない。意図的に借りて記録する | 視点4 |
| 動いたと正しいは別 | 偶然動いているコードは環境が変わると壊れる。「なぜ動くか」を説明できるか | 視点2,4 |
| 抽象化のしすぎ問題 | 「将来使うかも」の共通化が負債になる。Rule of Three（3回繰り返してから） | 視点4 |
| 可逆な決定と不可逆な決定 | やり直せる決定は速く、DB設計や公開APIは慎重に。悩む重さを可逆性で測る | 視点4 |
| エッジケースを考える習慣 | 空・0件・最大・異常入力で何が起きるか。「0件のときどうなる?」と問える視点 | 視点1,4 |
| 見積もりはなぜ当たらないか | 分かっている作業より調査・連携・手戻りが時間を食う。見積もりは幅であって約束ではない | 視点4 |
