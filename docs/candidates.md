# コンテンツ候補

Day 19 まで公開済み。各候補は15分で読み切れるサイズ。

## 未公開の下書き

|-------|-------------|

## データのやり取り

| テーマ | レッスンの中身 |
|-------|-------------|
| HTTP の基本 | `fetch` したとき何が飛んでいるか。GET / POST / ステータスコード（200/404/500） |
| REST API を叩く | エンドポイント、リクエストボディ、レスポンスの JSON。Network タブで実物を見る |
| gRPC という選択肢 | REST じゃない通信方式。Proto ファイルから型が自動生成される。なぜ選ばれるか |
| フォームのバリデーション | クライアントだけでは不十分な理由。「送信ボタンを押せない」だけでは防げない |
| Server Actions とフォーム | サーバー側でバリデーションが自然にできる仕組み。フォームとの統合 |
| エラーが起きたら何を見せるか | 「予期しないエラー」で終わらせない。エラーコード、ユーザーへの伝え方 |
| useState で足りなくなる瞬間 | コンポーネントをまたいで共有したい値。グローバル状態の使いどころ |
| URL に状態を持つ | 検索条件やページ番号を URL に入れる。戻るボタンで元に戻る、共有できる |
| なぜ古いデータが表示されるか | ブラウザ / CDN / サーバーのキャッシュ多層構造。「スーパーリロードで直る」の裏側 |

## 攻撃されたら何が起きるか

| テーマ | レッスンの中身 |
|-------|-------------|
| なぜログインが保たれるか | ブラウザを閉じても次はログイン済み。Cookie とトークンの仕組み |
| 認証と認可の違い | 「誰か」と「何ができるか」は別の問題。ロールベースアクセス制御 |
| XSS — 画面に埋め込まれるスクリプト | ユーザーの入力をそのまま表示すると何が起きるか。攻撃の仕組み |
| CSRF — 知らない間に操作される | ログイン中の人に意図しないリクエストを送らせる攻撃の仕組み |

## テスト

| テーマ | レッスンの中身 |
|-------|-------------|
| なぜテストを書くのか | 「動いてるからOK」の怖さ。直したら別の場所が壊れる。テストで壊れた瞬間に気づける |
| テストの種類と使い分け | 単体 / 結合 / E2E の違い。何をどこまで。コストと効果のバランス |
| VRT — 見た目の壊れを自動検知 | CSS を直したら別の画面が崩れた。画像の差分比較で気づく仕組み |
| モック — バックエンドなしで開発する | 外部 API がまだないのにフロントを作る方法。何を偽装するか |

## 技術をどう選ぶか

| テーマ | レッスンの中身 |
|-------|-------------|
| ライブラリの選び方 | ダウンロード数だけで選ぶと危険。ライセンス、保守状況、型対応。チェックリスト |
| ラップするか直接使うか | 外部ライブラリに薄い層を挟むか。差し替え時の影響範囲。全ラップは過剰、全直は危険 |
| UI ライブラリの世界 | ヘッドレス（a11y だけ）vs フル UI（見た目込み）。React Aria / Radix / shadcn/ui の違い |

## チームで開発する

| テーマ | レッスンの中身 |
|-------|-------------|
| push したら何が起きるか | CI/CD。テスト → ビルド → デプロイが自動で走る。手動アップロードとの違い |
| ブランチの使い方 | main に直接 push しない理由。feature ブランチと PR の意味 |
| リンターとフォーマッター | ESLint と Prettier。書き方を機械で強制する理由。diff が読めなくなる問題 |
| ADR — 設計判断の記録 | 「なぜこの技術を選んだか」を後から追える仕組み。新人が自分で調べられる |
| ログを残す理由 | 本番で「おかしい」と言われたとき、何が起きたかを追う記録 |
| 監視とは何か | 本番が今どうなっているかをリアルタイムで見る。壊れてから気づくのでは遅い |
